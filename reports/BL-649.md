# BL-649 (companion) — Do already-installed iPhone PWAs get the new launch images? NO.

*The section above (a parallel BL-649 worker) covers the MERGE. This companion answers the owner's actual question, which that section did not: do clippers who ALREADY have the PWA installed pick up the new launch images, or only new installs? Verdict and evidence below. The merge itself was performed by the parallel worker; I found it already on main at `91b84410` and re-verified it independently.*

**The owner's question, answered plainly: NO.** A clipper who already has Clippers HQ on their home screen will **not** automatically get the new launch images. They keep seeing exactly what they see today until they remove the icon and add it again. Nothing about their app gets worse. A brand-new clipper who adds the app from now on gets the fix.

## PART 0 — why, with evidence not guesswork

The launch image is shown by iOS in the window **between the icon tap and the first web paint**, when the web view does not exist yet. So iOS cannot read the live HTML to choose the image; it must hold that info locally. It gets it by reading the `<head>` **once, at "Add to Home Screen" time**, snapshotting the icon, the capable flag, and the full set of `apple-touch-startup-image` tags into the home-screen web-clip, then resolving from that local snapshot on every later launch.

Every current clipper installed when the served HTML declared **zero** startup images (BL-647 proved none existed until BL-648). Their web-clip cached none, and iOS does not re-snapshot an existing web-clip, so they still hit iOS's no-image black screen.

**Honest uncertainty:** per-iOS-version refresh of web-clip metadata is undocumented and I could not test on a physical iPhone this round. Some reports say the apple-touch-ICON is occasionally re-fetched, but the startup IMAGES are not known to refresh. So the answer is **NO**, with that caveat, not a confident yes.

**No force-refresh mechanism works on iOS.** A service worker runs inside the web view, which does not exist during the launch phase, so it cannot touch the launch surface. iOS ignores the manifest for the launch surface and it carries no startup-image field. Cache-busting reaches an install that cached no startup URLs. A version bump is read by nothing. All four are Android-style intuitions that do not transfer; none is shipped.

## PART 1 — so I shipped NOTHING for it. No placebo.

No web-layer mechanism repopulates an installed web-clip's startup images, so shipping one would repeat the five prior black-screen fixes that passed their own tests while the phone stayed black. Worst case confirmed: an existing clipper is **unchanged, never worse** — the new tags are inert HTML an old install ignores and cannot be broken by, with no forced reload and nothing cleared. Declined: an in-app re-add nudge (the owner will not ask clippers to reinstall) and re-adding an app-shell precache (BL-369 made the SW inert on purpose; separate concern, its own round).
