## PART 0 — the auto-update question, with evidence not guesswork

### What iOS actually does with `apple-touch-startup-image`

The launch image is shown by the operating system in the window **between the icon tap and the first web paint** — a moment when the web view does not exist yet. That single architectural fact drives everything:

* Because the image must appear **before any web content loads**, iOS cannot fetch the live HTML to decide which image to show. It must already hold that information locally.
* iOS obtains it by reading the `<head>` **once, at "Add to Home Screen" time**, and snapshotting the icon, the `apple-mobile-web-app-capable` flag, and the full set of `apple-touch-startup-image` link tags into the home-screen web-clip.
* On every later cold launch it resolves the image from that **local web-clip snapshot**, not from the served HTML.

Every current clipper added the app when the served HTML declared **zero** startup images (BL-647 proved there were none until BL-648). Their web-clip therefore cached **none**, and iOS does not re-snapshot an existing web-clip. So they fall through to iOS's no-image behaviour — the black screen — exactly as before.

### The one honest uncertainty

Per-iOS-version refresh of web-clip metadata is **not publicly documented**, and I could not test on a physical iPhone this round. There are scattered developer reports that the apple-touch-**icon** is occasionally re-fetched for an installed clip, but the startup **images** are not known to refresh on any version. So the responsible answer is **NO**, with that caveat, rather than a confident "yes it updates". A confident wrong answer here would be worse than an admitted uncertainty — and five prior black-screen rounds already shipped confident wrong answers.

### Can anything force an installed clip to refresh, without delete-and-reinstall?

| Mechanism | Works on iOS? | Why |
|---|---|---|
| Service-worker update re-fetches the shell | **No** | A service worker runs **inside** the web view, which does not exist during the OS launch phase. It structurally cannot touch the launch surface. It never sees, let alone changes, the startup image |
| Manifest change triggers a re-read | **No** | iOS ignores the manifest for the launch surface (established in BL-647/648), and the manifest carries no `apple-touch-startup-image` field anyway. Those are HTML link tags, not manifest entries |
| Cache-bust the image URLs | **No** | The installed web-clip cached **zero** startup URLs, so there is nothing to bust. Changing URLs in the served HTML reaches an install that never reads the served HTML again |
| Version bump | **No** | There is nothing iOS reads for launch images that a version could invalidate |

All four are Android-style intuitions that do not transfer to iOS home-screen web apps. **None is shipped.**

### VERDICT

**Existing clippers: NO.** Only a fresh Add-to-Home-Screen captures the new tags. **New clippers: YES**, automatically.

## PART 1 — because no mechanism works, I shipped NOTHING for it. No placebo.

There is no web-layer mechanism that repopulates an installed web-clip's startup images. Shipping one anyway would repeat the pattern the owner is tired of: five prior fixes (BL-69, BL-69-V2, BL-85, BL-150.1, BL-150.3) each "passed" their own tests while the phone stayed black, and one verified itself by reading back the JSON file it had just written. **I will not add a sixth.**

**Worst case confirmed as unchanged, never worse.** The new startup-image tags are inert HTML. An install that ignores them is byte-for-byte in the same position it was yesterday: it still shows black, it loses nothing, no session, no form, no BL-615 batch draft is touched, and no reload is forced. There is no path by which BL-648's served-HTML change degrades an existing install.

**Two options considered and deliberately declined:**

* **An in-app "remove and re-add for a nicer launch" nudge.** Declined because the owner has said he will not ask clippers to reinstall. Building a nudge that asks them to do exactly that contradicts his position.
* **Re-adding an app-shell precache** so the post-launch-image web paint is faster for everyone. Declined here because BL-369 deliberately made the service worker **inert** to fix a cross-origin `<img>` regression, re-adding caching is a real risk surface that touches every user, and it is a **different** problem (it would shorten the wait, not deliver the launch image). It is a legitimate future round on its own, not part of this question.

If you still see the black flash at launch, that is **expected** and it **confirms** the verdict — your install predates the images.
