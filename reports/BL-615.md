## PART 2 — constrain the pickers to the campaign's platforms (the correctness bug)
A campaign declares its accepted platforms in the comma-separated `Campaign.platform` field (`prisma/schema.prisma:532`; prod stores `TIKTOK`, `TikTok`, `TikTok, Instagram`, `TikTok, YouTube, Instagram`, case-mixed). New client-safe helper `src/lib/campaign-platforms.ts` (`campaignAllowedPlatforms`, `campaignAcceptsPlatform`, case-insensitive) is the ONE source of truth for the UI and the server.
* **Batch UI:** each row's platform `<select>` offers ONLY the campaign's allowed platforms; the account `<select>` offers ONLY the clipper's connected accounts on those platforms. A TikTok-only campaign shows only TikTok; a TikTok+YouTube campaign shows only those two.
* **Server enforced (a UI-only constraint is not a constraint):** `POST /api/clips/batch` now resolves each row's account platform and REJECTS, before any write, any row whose account is on a platform the campaign does not accept, with a clear message ("This campaign accepts TikTok only, but that account is Instagram…"). This is on top of the core's existing RULE 1b (`clipper-submit-core.ts:122`, URL platform ∈ campaign platforms) and RULE 1c (CPM set).
* **Single-submit had the same gap** (it offered all accounts; it validated only at submit). Fixed: the single-submit account radiogroup (`SubmitClipPremium.tsx`) is now constrained to the campaign's platforms, and its page-level account-platform check was made **case-insensitive** (was a bare `!==` at `page.tsx:219`, which the server core already handled case-insensitively). The server core is unchanged, so single-submit attribution was always safe; this removes the mis-pick and the case wart.
* **Empty case handled honestly:** if the clipper has no connected account on any platform the campaign allows, both flows show "Connect a {TikTok/…} account on your Accounts page to submit here" — never an empty silent dropdown.

Proven (deterministic, live campaign/account ids):
```
campaignAllowedPlatforms("TIKTOK")                     -> ["TikTok"]            (case-insensitive)
campaignAllowedPlatforms("TikTok, YouTube, Instagram") -> ["TikTok","Instagram","YouTube"]
SERVER guard, TikTok-only campaign:  Instagram account -> REJECT   YouTube account -> REJECT   tiktok (lowercase) -> ACCEPT
```

## PART 3 — draft persistence (server-side, chosen deliberately)
localStorage is unreliable in this app's client environment, so the draft is **server-side**, which also survives across devices and cannot be lost to a cleared cache. New table `batch_submit_drafts` (applied with `CREATE TABLE IF NOT EXISTS`, no `prisma migrate`) + new `/api/clips/batch/draft` (GET load / PUT save / DELETE clear), all test-user-404-gated exactly like the batch endpoint. Scoped per **(clipper, campaign)** via a unique key, so a draft for campaign A never appears in campaign B.
* Loads on campaign select; autosaves debounced while editing; clears on explicit **Cancel and clear** and on a successful submit.
* **Holds no authority:** on restore, every row is RE-VALIDATED against the CURRENT campaign platforms, the CURRENT connected accounts, and the CURRENT remaining limit. A row referencing a now-disconnected account, a now-disallowed platform, or a link/platform mismatch is flagged amber ("needs attention", in text + the row's accessible name, never color alone) and Submit is blocked with that reason — it is never silently submitted to the wrong place. Autosave is gated behind a "draft loaded for this campaign" ref so switching campaigns never writes stale rows under the new campaign.

Proven (round-trip against the real table):
```
draft saved + restored (rows survive)   PASS
scoped per campaign (campaign B empty)  PASS
explicit clear removes the draft        PASS
```
