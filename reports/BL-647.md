# BL-647 — The black screen before the splash: cause and fix

## VERDICT (one line): The black screen is iOS showing NOTHING because the app declares ZERO `apple-touch-startup-image` entries, so from the moment the icon is tapped until the web page paints, iOS has no image to display and no fallback; every previous fix edited the web page, which is a layer that does not exist yet while the black screen is on screen.

**2026-07-23 · AUDIT ONLY. READ ONLY.** No code, config or asset change. **NO BUILD WAS RUN.** I inspected an existing `.next` build artifact produced at 22:30 by another round and say so wherever I use it; I do not claim a build or an exit code I did not produce.
**Base:** origin/main `b213ed44` · **Branch:** checkpoint/BL-647 · isolated worktree at a short path, no node_modules junction.

## PART 0 — why the previous fixes failed

Five attempts over roughly a month. **Every single one changed only `src/app/layout.tsx` and/or `public/manifest.json`. Not one added an image asset or an OS-level launch declaration.**

| Attempt | Commit | Files touched | What it did | Why it could not work |
|---|---|---|---|---|
| BL-69 F-LAUNCH-SPLASH | `cb39d00b` | layout.tsx, manifest.json | Built the blue splash with the white triangle | Built the thing that appears **after** the black, not the black |
| BL-69 F-SPLASH-V2 | `cd52f673` | layout.tsx, manifest.json | Cold-open dedupe, 2.5s hold | Tuned the splash's own timing; the black is before it |
| BL-85 bug sweep | `94052efe` | layout.tsx | Explicit first-paint background on html and body | A page background cannot paint before the page exists |
| BL-150.1 | `1d105efe` | layout.tsx, app-layout.tsx | Splash gate, route allowlist | Decides **whether** the splash shows, not what precedes it |
| **BL-150.3** | `9556cb68` | layout.tsx, manifest.json | **"kill the ~1s black before the blue splash"** via `manifest.background_color` + `theme-color` | **Targeted the black directly and still failed, because iOS ignores `background_color` for PWA launch. It is an Android/Chrome feature.** |

**BL-150.3 is the smoking gun.** It aimed at exactly this symptom, set the manifest launch colour to brand blue, and the owner still sees black. That is decisive evidence the black lives outside the manifest's reach on iOS.

**And here is how it "claimed success".** Its verification script `scripts/test-bl-150-3-launch-color.ts:33-40` asserts:

```
"manifest.background_color === '#2596be'"  →  manifest.background_color === "#2596be"
```

It reads back the JSON file the same commit had just written. **That assertion cannot fail and proves nothing about an iPhone.** Every prior round passed its own tests while the phone was unchanged. This is why PART 6 gives a test performed on the actual device.

**Not repeated here:** any further edit to the splash element, its timing, its gate, or the page background. Those four layers are already correct and are not where the black lives.

## PART 1 — which launch phase the black belongs to

**It is phase (a): the OS launch, before the web view has painted anything.** Evidence, in order of decisiveness:

**1. There are zero startup images. Verified two ways.**
```
grep -rn "apple-touch-startup-image" src/ public/   →  no matches
grep -c "apple-touch-startup-image" .next/server/app/dashboard.html   →  0
```
Not in the source, and not in the HTML actually shipped to the device. `public/` contains no splash or launch asset of any kind: the only images are icons, favicons, email art and a login background.

**2. What the app does declare, and why none of it helps on iOS.**

| Declared | Where | Effect on the iPhone black screen |
|---|---|---|
| `background_color: "#2596be"` | `manifest.json:6` | **None.** iOS does not use manifest `background_color` for the launch surface. Chrome on Android does, which is why this looked reasonable |
| `theme_color` / `<meta name="theme-color">` | manifest + `layout.tsx` | Status bar tint only, after paint |
| `apple-mobile-web-app-capable: yes` | `layout.tsx` | Enables standalone mode, which is precisely what **activates** the startup-image mechanism the app never populated |
| `apple-mobile-web-app-status-bar-style: black-translucent` | `layout.tsx` | Makes the status bar area transparent over the page, contributing to the all-black impression |
| `apple-touch-icon` | `layout.tsx` | Home-screen icon only |

**3. It is not phase (c) or (d).** The blue splash is **server-rendered HTML**, not a React component: `<div id="launch-splash">` is present in the shipped HTML (9 occurrences in `dashboard.html`) with inline CSS. It needs no JavaScript. And `/dashboard` is **statically prerendered** (confirmed in the build's prerender manifest), so there is no slow dynamic server render. The page is about as fast to first paint as this stack allows. **The black is not the app being slow to render. It is iOS having nothing to show while it waits.**

**4. It explains desktop being instant.** A desktop browser has no OS launch phase at all. The tab already exists and is already painted, so there is no window in which a launch image would be needed. This asymmetry is a fingerprint of phase (a), not of app performance.

**What iOS does when no startup image matches:** it shows a blank surface for the whole cold-start window. Exact rendering has varied across iOS versions (some show white, some black, some briefly the last snapshot), which is consistent with the owner seeing black and others reporting 5 to 10 seconds. **Stated honestly as UNMEASURED:** I could not test on a physical iPhone in this round, so the precise per-version behaviour is inferred from the declaration set, not observed. What is certain and directly verified is that no matching image exists for any device.

## PART 2 — what must load before anything can paint

From the existing build artifact (not a build I ran):

| Step | Resource | Size | Blocking? |
|---|---|---|---|
| 1 | `/dashboard` document, statically prerendered | **38,240 bytes** | yes, the document |
| 2 | `chunks/0-i0d7yqc7p1b.css` | 3,712 bytes | **render-blocking** |
| 3 | `chunks/0aaqrmlhh_0qe.css` | **227,795 bytes** | **render-blocking** |
| 4 | 2 woff2 font preloads | via `next/font` | not blocking paint (`font-display: swap`) |
| 5 | 15 script tags, ~708 KB raw root bundle | deferred by Next | **not** blocking first paint |

**The critical path to the blue splash is: document, then both stylesheets, then paint.** That is a minimum of two network round trips on a cold connection. The splash markup arrives in step 1 but **cannot paint until step 3 completes**, because the browser builds the CSSOM from every `<head>` stylesheet before first paint. A 222 KB stylesheet on the critical path is a real, measurable contributor.

**Something does paint before JavaScript**, which is a genuine strength here and worth stating so it is not "fixed" again: the splash is pure HTML and CSS. The JavaScript bundle is not what the user is waiting for during the black.

## PART 3 — why desktop is instant and mobile is not

Four independent reasons, all pointing the same way:

• **No OS launch phase on desktop.** The decisive one. No launch image is ever needed because the tab is already painted.
• **No service worker cache, on any platform.** `public/sw.js` was deliberately reduced to an **inert shell** by BL-369 (`SW_VERSION = 'bl-369-inert-1'`). Its fetch handler never calls `respondWith`, so it caches nothing and serves nothing. **The app shell is NOT precached.** Every cold launch goes to the network for the document and both stylesheets.
• **Warm versus cold HTTP cache.** The owner's desktop visits are typically a warm tab or a recent visit; a phone opened after days is fully cold.
• **Network and CPU.** Cellular latency dominates two serial round trips, and mobile CSS parsing of a 222 KB sheet is slower than desktop.

**A PWA that does not precache its shell will always feel slow to launch, and this one precaches nothing.** That is by deliberate BL-369 decision to fix a cross-origin image regression, so re-adding caching must be done narrowly (see PART 5, fix 3).

## PART 4 — why a returning user still waits

**The deploy cadence fully explains it, and the number is striking. Measured on main:**

```
merges to main, last 30 days:            279
days with at least one merge (of 30):     30
```

**That is roughly 9.3 deploys per day, every single day for thirty days.** Every deploy rehashes every bundle filename, so all previously cached JavaScript and CSS become unreachable and are re-downloaded in full.

**A returning user is therefore essentially always cold, no matter how recently they visited.** There is no expiry to tune, because the cache is invalidated by a new deploy long before it would ever expire. Combined with the inert service worker (which would otherwise let a stale-but-instant shell paint immediately), the practical result is that **every launch is a first launch.** This is the answer to "it happens when I have not opened it in a while": it also happens when he has, he just notices it more after a gap.

## PART 5 — the fix, ranked by seconds saved per unit of risk

### 1. Add a complete `apple-touch-startup-image` set. THE FIX.
**Saving:** removes the black **entirely**, not partially. iOS paints your blue image the instant the icon is tapped and holds it until the page paints, so the user never sees black regardless of how slow the network is.
**How:** generate one PNG per supported iPhone geometry (brand blue `#2596be` with the white triangle, matching the existing splash so the handoff is invisible), and declare each with a media query in `layout.tsx`. The media query uses CSS points and pixel ratio, and **must match the device exactly or iOS ignores it**, which is the trap that makes this fiddly:

```
<link rel="apple-touch-startup-image"
      media="(device-width: 393px) and (device-height: 852px)
             and (-webkit-device-pixel-ratio: 3) and (orientation: portrait)"
      href="/splash/1179x2556.png">
```

Minimum set to cover current and recent iPhones (points at ratio, giving the pixel size of the PNG):
430×932@3x → 1290×2796 · 402×874@3x → 1206×2622 · 393×852@3x → 1179×2556 · 428×926@3x → 1284×2778 · 390×844@3x → 1170×2532 · 375×812@3x → 1125×2436 · 360×780@3x → 1080×2340 · 414×896@3x → 1242×2688 · 414×896@2x → 828×1792 · 375×667@2x → 750×1334.
**The owner should confirm his exact iPhone model first**, so the build round can guarantee his device is covered on day one rather than guessing.
**Risk:** very low. Additive `<link>` tags plus new static files. Nothing existing changes behaviour.
**What could break:** essentially nothing. A wrong media query means that device falls back to today's behaviour, no worse. Adds a small amount to the repo.
**How the owner verifies:** PART 6. This is the only fix whose result is visible in the first second on his own phone.

### 2. Reduce the render-blocking CSS.
**Saving:** shortens the real wait before the splash paints. **UNMEASURED in seconds**; measuring it needs a throttled trace on a physical device, which this round could not run.
**How:** the 222 KB stylesheet is on the critical path of every launch. Splitting route-specific CSS off the entry route, or inlining only what the splash needs, removes one blocking download.
**Risk:** medium. Touches the styling of every page.
**What could break:** a missed rule causes an unstyled flash, exactly the class of regression the owner has been burned by.
**Verify:** the splash appears sooner on a cold launch, and every page still looks identical.

### 3. Precache the app shell in the service worker.
**Saving:** turns the second and later cold launches into an instant paint from cache, and would blunt the 9-deploys-per-day problem in PART 4.
**How:** precache the document and the two stylesheets, same-origin navigation requests only.
**Risk:** **high, and historically proven.** BL-369 deliberately made this SW inert because request interception broke cross-origin YouTube and Supabase cover images on soft reloads, and a stale SW kept the bug alive in browsers that had installed it.
**What could break:** the exact image regression BL-369 fixed, plus a stale shell serving an old app after a deploy.
**Verify:** cold launch paints instantly; then specifically re-check that YouTube covers still load on a soft reload.
**Do this only after fix 1, and only as its own round.**

### 4. Change `apple-mobile-web-app-status-bar-style` away from `black-translucent`.
**Saving:** cosmetic only; it removes a black status-bar strip, not the black screen.
**Risk:** low. **Do not ship this alone**, it would look like another failed attempt.

### The plain answer
**Fix 1 alone removes the black screen entirely**, because it fills the exact window in which the black occurs with your own artwork. It does **not** make the app load faster; the underlying wait for document and CSS is unchanged. The user simply sees brand blue with the logo for that time instead of nothing, which is what every native app does. **Fixes 2 and 3 shorten the actual wait; only fix 1 removes the black.**

## PART 6 — the pass/fail test the owner runs himself

Previous rounds passed file-reading assertions. This is performed on the phone and takes under a minute.

**Force a genuine cold launch. All four steps matter, in order:**
1. **Delete the PWA** from the home screen (press and hold, Remove App, Delete App). Re-adding it is what makes iOS re-read the startup images; an existing install keeps the old set.
2. In Safari, **Settings → Safari → Clear History and Website Data**.
3. Re-add the app: open clipershq.com in Safari, Share, **Add to Home Screen**.
4. **Fully close it**: swipe up from the bottom and swipe the app card away. Also swipe away Safari. Then wait about 30 seconds so iOS releases the process.

**Now tap the home-screen icon and watch the first two seconds.**

| | PASS | FAIL |
|---|---|---|
| Second 0, the instant you tap | **Blue screen with the white triangle** | Black or blank |
| Seconds 1 to 4 | Still blue, no flicker, no black gap at any point | Black at any moment |
| Then | Blue splash fades into the app | A visible black flash between splash and app |

**The single question: was there ever any black, even for a fraction of a second?** If yes, it failed, regardless of what any test script reports.

**Repeat the whole sequence twice**, because the first launch after install is the slowest and the most honest test. Also worth testing on cellular with Wi-Fi off, since that is the slow path where the black is longest today.

**A note on what a pass means:** the app will still take the same few seconds to become usable. What changes is that the wait is blue and branded rather than black and broken-looking. If the owner also wants the wait itself to be shorter, that is fixes 2 and 3, and they should be judged separately.
