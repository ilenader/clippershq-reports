## The headline

The rule is implemented exactly as stated, enforced on the **server** at **all three** payout creation paths (there were three, not one), with the threshold as **one constant in one place**, compared against the **net amount actually sent on-chain**.

Two things worth the owner's attention before anything else:

1. **Nobody is broken by this.** There is **no saved payout method** anywhere in the system. Not one clipper holds a method that is about to disappear, because nothing is ever saved. PART 3's dead-end scenario cannot occur.
2. **The accessibility review caught a money-loss trap in my own plan.** Storing the label `"ERC-20"` would have silently **disabled wallet address validation**, letting a Solana address be accepted as an Ethereum payout. This is not hypothetical: production already holds 2 payouts stamped `"ERC20"` whose addresses were never format-checked. Details below.

## PART 1: enforced on the server

### Three creation paths, not one

The brief assumed one. There are three, and all three now carry the rule:

1. **`src/app/api/payouts/route.ts`** — the main clipper path.
2. **`src/app/api/payouts/referral-request/route.ts`** — referral commission cashout. **This is a second real way money leaves the platform** and it had the same hole.
3. **`src/actions/payouts.ts` `requestPayout()`** — a `"use server"` action. **It is dead**: nothing imports it, so Next never registers a callable action. But it is a payout creation site that **accepts no asset or chain at all**, so I guarded it rather than leave a latent bypass for whoever wires it up later. It **fails closed by construction**: it cannot supply a method, so `validatePayoutMethod` always refuses. Guarding cost three lines.

### The threshold: one constant, one place

`src/lib/payout-methods.ts` is new and is the only definition of the rule:

```ts
export const SOLANA_MIN_NET_USD = 12;
```

The API imports it. The UI imports it. It is re-declared nowhere. The owner changes `12` there and every surface follows, with no treasure hunt.

### Gross or net? NET. The amount actually sent.

**Decided: the $12 is compared against `calculatePayoutBreakdown(...).finalAmount`, the amount that actually leaves on-chain, never the gross the clipper typed.**

The reasoning, which is written into the constant's docblock so the code answers the argument rather than a report:

> The owner's rule exists because of what it costs **them** to move coins on a chain. The constraint applies to the transfer that leaves, not to a number in a form.

`payout-calc.ts:84`: `finalAmount = round2(requestedAmount - feeAmount - expressFeeAmount)`. That is the transfer.

**The brief's own example confirms the reading exactly.** It predicted the argument would be about "a $13 request that nets $11.83". That is precisely 13.00 minus 9%:

```
$13.00 gross, 9% fee -> feeAmount 1.17 -> finalAmount 11.83 -> UNDER 12 -> ERC-20 ONLY
```

Comparing against gross would let an $11.83 transfer go out on Solana, which is the exact thing the owner's constraint forbids. So: net, always. Proven live below.

Note the fee is **9%, or 4% for referred users** (`route.ts:308`), and express adds a further 4%. All three are inside `finalAmount`, so the rule follows the real number automatically without knowing about any of them.

## The money-loss trap the accessibility review caught

I asked the accessibility lead for the markup contract. It came back with a **money** finding I would otherwise have shipped.

`src/lib/wallet-validation.ts:75-81`:

```ts
// Unknown / unspecified chain — don't block. Admin will sanity-check
// in Discord before paying. Still enforce max length so a junk paste
// can't bloat the row.
if (a.length > 200) { return { valid: false, error: "..." }; }
return { valid: true };
```

`EVM_CHAINS` (`:8-20`) contains `ETHEREUM` and `ETH`. **It does not contain `ERC20`.**

So if `walletChain` is stored as the pretty label `"ERC-20"`, `validateWalletForChain` **does not recognise it, does not block it, and skips the address format check entirely.** A Solana base58 address would sail through as an Ethereum payout.

**This is already live.** Production holds **2 payouts stamped `walletChain="ERC20"` ($111.24)** whose addresses were never format-checked, because `"ERC20"` was never in the EVM set either.

**The fix: label and wire value are decoupled.**

* The clipper **reads** `"ERC-20"`.
* The stored value **stays** `"Ethereum"`.

Both routes additionally **re-run the address check against the canonical chain** after normalizing, so the format is genuinely enforced on this path for the first time.

The existing code even warned about this and was ignored by my first instinct. `PayoutRequestFlow.tsx:79-82`, before I touched it:

> *"These map to the exact tokens validateWalletForChain recognizes (TRON / SOLANA / ETHEREUM), so picking a chip enables the server's chain-aware address check rather than bypassing it."*

## Canonical writes: the free-text era ends

Both routes now store `methodCheck.asset` / `methodCheck.chain`, never the raw client string. From here forward there is exactly **one asset value and two chain values**.

Normalization is deliberately **generous on input, strict on output**: `usdt`, `Usdt`, `Erc20`, `Ethereum (ERC-20)` all map through, because a clipper must not be punished for our spelling history. **USDC is not an alias.** It is a different asset and it is removed.

### Two UI traps found and closed

**1. Express takes a further 4% AFTER the chain is chosen.** Chain is picked on the details step; speed on the next one. So:

```
$13.20 standard -> sends $12.01 -> Solana fine
$13.20 express  -> sends $11.48 -> server REJECTS
```

The clipper would have hit a failure **after committing**, which PART 2 explicitly forbids. The Express card is now locked with a plain reason whenever it would drop a Solana payout under the threshold.

**2. A stale Solana selection.** Picked at a valid amount, then the amount is lowered. Caught by validation, with focus moved to the control.

**Deliberately NOT auto-reselecting ERC-20.** If the chain silently flipped while a Solana address sat in the wallet field, the clipper would confirm a payout **to a mismatched chain** and the money would be gone. Block and explain. The destination never changes without them.
