## THE TWO ANSWERS UP FRONT

1. **Detection misses 70/11 because the one proportionality signal it has runs in the wrong direction, and never at that size.** The current ratio check fires on too FEW likes per view (the view bot). The owner's case is the opposite: too MANY likes for the views (bought likes, an engagement pod, or friends). And the check is gated behind a 500 view floor, so **1556 of 3227 approved clips (48%) are never evaluated at all.** The 70 view clip is in that blind half. R-5 already proved the aggregate `fraudScore` is inert (score 0 and score 40+ both reject at 19%), so the fix is a NEW, separately surfaced RANK, not another input to a dead number.
2. **The screenshot reader is a TikTok Studio only parser with no platform detection.** It applies one flat TikTok label map to every image, so Instagram and YouTube analytics screens (different labels, different layout) are mis read or unread.

## PART 1: WHY DETECTION MISSES OBVIOUS CASES

### What detection uses now (from code)

`computeFraudLevel` (`src/lib/fraud.ts:89`) is the only money path detector. It sums points from 10 signals over the view/like/comment/share time series into a 0..100 score (full signal map in BL-511). The proportionality signals are #2 (views:likes ratio, `:116`), #3 (comments:views, `:128`) and #6 (likes greater than views, `:147`). Two facts in that code decide the owner's case:

* **Signal #2 only runs when views are above 500** (`MIN_VIEWS_FOR_ENGAGEMENT`, `fraud.ts:64`). At 70 views it never executes.
* **Every band of signal #2 is a LOW engagement test**: ratio above 200 / 500 / 2000 views per like (`:118-123`). There is NO high engagement band. 11 likes on 70 views is 6.4 views per like, the healthiest end of the ratio, so even above the floor it scores zero.

So the owner is not looking at a threshold set too high. He is looking at a signal that does not exist. Nobody computes "too many likes for the views," and the whole check is dark below 500 views.

Reading these three tables:

* **Normal is niche specific and the spread is huge.** IG Facts p90 is 0.76%; TikTok Relatable p50 is 8.26%. A 5% like rate is p90 anomalous for IG Facts and dead median for TikTok Relatable. One global threshold cannot be right for both. This is the owner's niche point, and the data backs it.
* **Normal is tier specific.** Under 100 views p90 is 22.2%; at 2k to 10k views p90 is 5.5%. Tiny clips are dominated by like noise.

### The proportionality check, designed

Score a clip's like rate against the band of its OWN peer group (platform, then niche, then view tier) built from our own APPROVED history:

* Compute p10 / p50 / p90 per peer group (done above; refresh periodically as a read only job).
* Rank by how far outside its peer p10..p90 the clip sits, in EITHER direction. Below p10 is the view bot shape (bought views, few likes). Above p90 is the like bot / pod shape (bought or farmed likes), which is the owner's case.
* Corroborate, never decide on the ratio alone. Pair with the incremental new likes per new view idea already in signal #10, and with account context (BL-511 Proposal C), so a single noisy ratio never ranks high by itself.

### Where the owner's clip lands, honestly

70 views / 11 likes = 15.7% like rate. Against ALL clips it is above p90 on every platform (TikTok p90 11.81%), which is exactly why his eye reads it as bought. **But against its OWN tier (under 100 views) p90 is 22.2%,** so roughly 1 in 10 honest tiny clips look like this from like noise alone. So:

* **The signal is real and worth surfacing.** 243 approved clips sit above the TikTok p90 like rate today and NONE are flagged for it, because no over engagement signal exists. 258 approved clips are the exact 70/11 shape (under 500 views, 5+ likes) and all are invisible.
* **The confidence is LOW at tiny view counts and RISES with views and corroboration.** At 70 views the ratio alone cannot separate bought from friends. So it RANKS for a human, it must not reject.

### False positive risk, stated plainly

* A genuinely viral clip has a low like rate while views outrun likes, and would rank as a view bot. A genuinely tiny clip a few friends liked has a high like rate, and would rank as a like bot. Both are innocent. **Ranking, not rejection, is the only safe use (BL-518: a fraud signal must never, on its own, change status, earnings, tracking, or anything a clipper sees).**
* Niche is free text and messy ("Clipping " with a trailing space, "Meme" vs "meme comedy"), about 20% unset. Bands need normalization and a minimum sample per group; below that, fall back to platform by tier. Follower tier is NOT available (`followerCount` filled on 0.7% of accounts, BL-511), so peer grouping is platform by niche by view tier only.

## PART 2: THE SCREENSHOT READER, PER PLATFORM

### What it is and how it works today

This is the BL-345 creator scan evidence tool, NOT the bio code account check (that is BL-564, a separate path). Upload route `/api/admin/creator-scan/[accountId]/screenshot/route.ts` (OWNER / `CREATOR_SCAN_REVIEW` only, never clipper visible). It stores the image, then OPTIONALLY OCRs it via OCR.space (env gated `OCR_SPACE_API_KEY`; with no key it degrades to store only plus manual entry, never blocking upload) and pre fills the reviewer's Layer 2 form. Parsing is `parseStudioMetrics` in `src/lib/creator-scan/screenshot-ocr.ts`.

### Why it mis reads

The file's own header states the scope: "Reads the printed text numbers from an uploaded TikTok Studio analytics screenshot." The `EXTRACTORS` list (`screenshot-ocr.ts:30-48`) is a single flat set of TikTok Studio labels applied to every image. **There is no platform detection.** Consequences:

* **YouTube's core metrics are simply absent from the map:** Impressions, Impressions click through rate, Average view duration, Watch time (hours), Subscribers. "Average view duration" does not match the `avgWatchTimeSec` regex (`/avg\.?\s*watch\s*time/`), so YouTube's central retention number is unread.
* **Instagram labels differ:** "Accounts reached", "Accounts engaged", "Plays" / "Views", "Interactions", "Profile visits". Most are unmapped; the one that happens to match ("Profile visits" hits `/profile\s*(?:views|visits)/`) was written with TikTok semantics.
* **Cross column bleed.** A value is read from up to 40 chars after the label (`extractValue`, `:61`), and matching is first hit per field wins across the whole text (`:91-102`). On an unfamiliar multi column layout the number from the next column can bind to the wrong field. The code even notes the "profile views vs profile" ordering hazard, which is a within TikTok patch, not a per platform fix.

### Designed fix: detect platform, then map labels

Two pure, unit testable stages:

1. **DETECT the platform from the OCR text before parsing.** Score the text against three small signature sets: TikTok Studio ("For you", "Total play time", "Retention rate", "Traffic source"), Instagram ("Accounts reached", "Accounts engaged", "Profile activity", "Plays"), YouTube ("Impressions", "click through rate", "Average view duration", "Watch time (hours)"). Highest signature count wins; a tie or zero means "unknown", fall back to today's generic pass and ask the reviewer to pick the platform. Detection is cheap and calls no API.
2. **MAP with a platform specific label table.** One `EXTRACTORS` set per platform, each mapping that platform's exact labels to the same `StudioField` keys the form already uses. Instagram "Accounts reached" to `reach`; YouTube "Average view duration" to `avgWatchTimeSec`, "Impressions" to a new field, and so on. Keep the existing sanity clamps. Chart SHAPES (the retention curve) stay for the human to eyeball, as today.

### What the owner should upload, per platform

State this in the panel so the reader receives the fields it needs:

* **TikTok** (per video, TikTok Studio, Analytics, the post): one screenshot of the post analytics showing Views, Total play time / Avg watch time, Retention, Traffic sources, plus Likes / Comments / Shares / Saves. Usually one tall screenshot; if traffic sources sit on a second screen, two.
* **Instagram** (per reel, Insights): one screenshot of the reel Insights (Views / Plays, Accounts reached, Interactions, Likes / Comments / Shares / Saves). Add a second of Profile, Professional dashboard (Profile visits, Accounts reached) only if profile level numbers are wanted.
* **YouTube** (per short, YouTube Studio, Analytics): one screenshot of the video Overview (Views, Watch time, Impressions, CTR, Average view duration, Subscribers), plus one of the Audience or Engagement tab if retention is wanted.

### Handling multiple screenshots for one verification

The storage layer already supports many per account: each POST stores one image linked to the account (optionally a `clipId` / `videoUrl`), and GET lists up to 200 per account. So the reader should MERGE across screenshots: run detect plus map on each image, then union the fields, keeping the highest confidence value per field and tagging each value with its source image. That lets a TikTok "traffic sources" screen and a "post overview" screen combine into one filled form, and lets a reviewer add a missing screen without losing what was already read.

## PART 3: THE VERDICT, RANKED BUILD ORDER

Two independent tracks. Neither is built here (audit only). Both RANK for a human; neither auto acts.

**Detection (proportionality by niche, RANK only):**
1. Add a two sided, peer relative like rate rank, computed from the live distribution above (platform by niche by view tier, p10..p90). Surface it as its OWN column in the existing `/admin/fraud-review` queue. Measurable now: the bands exist in today's data; build it as a read only shadow script first.
2. Remove the 500 view blind spot for the RANK path only (not for any auto action): let tiny clips be ranked, with confidence explicitly scaled DOWN by view count so like noise cannot promote them.
3. Corroborate with account context and the incremental like ratio (BL-511 Proposals B and C) before a clip ranks high. Never fold this into `fraudScore`, which R-5 proved inert.
Do NOT: auto reject, auto freeze, or show any of it to clippers (BL-518; `clip-sanitize.ts` already strips the fraud fields from clipper responses).

**Screenshot reader (detect then map):**
1. Add platform detection over the OCR text (pure, no API), defaulting to the current generic pass on "unknown".
2. Split `EXTRACTORS` into three per platform label maps; add the missing YouTube and Instagram fields.
3. Merge multiple screenshots per account by field union; add the per platform upload guidance to the panel copy.

**What is measurable now vs needs data collection first:**
* **Measurable now:** the like rate bands (computed above), the blind spot size (1556 clips), the over engagement cohort (243 clips), and the screenshot mis map (code proven). The RANK and the reader split can be built against these today.
* **Needs collection first:** a hand labelled bought vs honest set to prove the PRECISION of the rank. R-5 showed the old score had none, so claim none until labelled (BL-511 Part 4 already specifies the shadow score plus confusion matrix plus false freeze protocol). And per niche bands are thin below 25 samples, so rare niches need more history before their band is trustworthy; fall back to platform by tier until then.

**Do not over claim.** This designs a RANKING that puts the 70/11 shape in front of a human. It does not, and on current data cannot, prove any single clip is bought. Precision is a validation output, not a promise.
