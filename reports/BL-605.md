## PART 1 — Bug 1: mixed carousel stored 0 (hikerapi.ts:483)

`classifyV2Media`'s mixed-carousel branch summed the video children's play counts
and returned `views: sum`. When a video child exposes NO readable play count
(IG hides a feed-video's view count — the live BL-604 case DbAuwN2F0MO), the loop
never adds anything, `usedKey` stays null, and the function returned `views: 0`.

Fix: `views: usedKey === null ? null : sum`. If at least one video child yielded a
count, return the real sum (even a legitimate readable 0). If none did, the honest
value is unknown -> NULL, never a fabricated 0.

Proven LIVE on BL-604's three IG posts, id-matched (shortcode == requested), through
the real `fetchHikerInstagramByUrl` after the fix:

| shortcode | classification | views | storesZero |
| --- | --- | --- | --- |
| DbA5DNyDkhA | carousel_image_only | null | false |
| DbA1tCrjun7 | carousel_image_only | null | false |
| DbAuwN2F0MO | carousel_mixed | null (was 0) | false |

The two image-only carousels STAY null (unchanged); the mixed carousel now returns
null instead of 0. None returns 0.

Honest storage note: the tracking overlay caller `tryHikerForInstagram` already
rejects ALL carousel classifications at hikerapi.ts:712 (before its views check),
so today the mixed-0 fell through to Apify rather than being stored via the overlay.
Returning 0 was still wrong (BL-543) and a latent landmine — any future caller
reading `classifyV2Media` directly, or a relaxed gate, would have stored the 0, and
BL-604's diagnostics showed a misleading 0. The fix returns the correct value at the
source; the gate is no longer the only defense.

## PART 2 — Bug 2: first ClipStat not null-safe (clips/route.ts:1137)

The submit transaction wrote `views: fetchedStats?.views ?? 0`, so a null resolve
(a failed fetch, or a view-less carousel) became a stored 0 on the first snapshot.

`ClipStat.views` is `Int @default(0)` — NON-NULLABLE (schema.prisma:1061) — so a
literal null row is impossible without a schema change to a hot table. The
BL-543-correct realization is therefore to SKIP the first snapshot when the resolve
is null (identical to the established tracking.ts null pattern: no baseline written,
the recurring poll writes the first REAL stat next tick), rather than persist a
fabricated 0:

```
const resolvedFirstViews = fetchedStats?.views;
if (resolvedFirstViews != null) {
  await tx.clipStat.create({ data: { clipId, views: resolvedFirstViews, likes, comments, shares } });
}
```

The ONLY behavior change is the null case (0 -> no stat). Every resolving path is
unchanged. Proven (exact replica of the gate):

```
provider returned null stats   -> SKIP (no first stat)
carousel/failed views=null     -> SKIP (no first stat)
real fresh post views=0        -> WRITE views=0        (a genuine readable 0 is a real count)
normal video views=125430      -> WRITE views=125430   (normal path byte-identical)
```

A real 0 (a brand-new post with 0 views) is `0 != null` -> still written, so a
legitimate zero is never lost; only an UNRESOLVED null is skipped. The submit-path IG
resolver is Apify (`fetchClipFreshnessWithRetry` uses `skipHikerOverlay:true`,
apify.ts:2232), so this fix protects every provider-null on submit, not only
carousels.
