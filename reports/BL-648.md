## PART 1 — the iOS launch images (the black screen)

### What was wrong
BL-647 proved the app declared **zero** `apple-touch-startup-image` entries. iOS shows a PWA launch image **only** when one matches the device's exact width, height, pixel ratio and orientation. With none present it paints **black** for the entire window between the icon tap and the first web paint. That is the 4 to 10 seconds.

**Why five previous rounds failed:** BL-69, BL-69-V2, BL-85, BL-150.1 and BL-150.3 all edited `src/app/layout.tsx` colours and `public/manifest.json`. **iOS ignores `manifest.background_color` for the launch surface** (it is an Android/Chrome feature), so none of them could ever change anything on an iPhone. BL-150.3's own test asserted `manifest.background_color === "#2596be"` by reading back the JSON the same commit had just written, which cannot fail. **This round changed neither of those two files' colour values.**

### What was built
`scripts/generate-ios-splash.mjs`, using **only the Node standard library** (`zlib` + a hand-written PNG encoder). Adding an image library to render two flat colours would have been a heavier change than the fix itself, and this keeps the output byte-deterministic.

**Every value is DERIVED from the existing splash, never guessed:**

| Value | Source |
|---|---|
| `#2596be` background | `layout.tsx` `#launch-splash { background: #2596be }` |
| `50,10 90,85 10,85` white triangle | `layout.tsx` `<polygon points="50,10 90,85 10,85">` (identical to `public/logo-white.svg`) |
| 96 CSS px logo size | `layout.tsx` `#launch-splash svg { width: 96px }`, so device pixels = 96 × DPR |
| `-6%` vertical nudge | `layout.tsx` `#launch-splash svg { transform: translateY(-6%) }` |

Because all four come from the in-page splash, the OS launch image and the web splash are **visually identical and the handoff is invisible**.

## The accessibility review changed the design, and one finding was serious
The harness blocked the UI edits pending review. That review found **five** problems in my first draft, all now fixed:

1. **`aria-busy="true"` and `aria-live="polite"` on the same element cancelled each other out.** `aria-busy` tells assistive tech to defer presenting changes, so the region was marked busy while being asked to speak. Both removed.
2. The `sr-only` text **duplicated the already-visible** "Loading daily series…" copy, so browse-mode users heard it twice. Removed in favour of the visible text.
3. A bare 220px placeholder would have been **~34px shorter** than the loaded tree (heading + margin + chart), shifting the page when the chunk landed. The heading wrapper is now reproduced exactly.
4. A live region inserted **together with** its text announces unreliably. There is now **one persistent `role="status"` region outside the swapped subtree**, and the early return was replaced with a single return so the region is never destroyed and recreated.
5. **The serious one.** `next/dynamic` is `React.lazy` underneath, and a rejected import **throws to the nearest error boundary**. There is no `error.tsx` in the `(app)` segment, so a failed chunk on a bad connection would have **blanked the whole earnings page, including the balance and withdrawal figures**. That directly violates this round's own safety rule. The import now carries a `.catch` returning a static text fallback, so a bad connection costs you a graph, never a balance.

Two pre-existing issues were **reported, not fixed** (CLAUDE.md rule), and filed as **BL-649** (the viewport meta blocks pinch zoom, WCAG 1.4.4) and **BL-650** (the recharts SVG has no text alternative in any of its six usages, WCAG 1.1.1).

## PART 4 — the owner's pass/fail test, under a minute

Previous rounds claimed success without ever checking a phone. This is checked on the phone.

**Force a genuine cold launch. All four steps, in order:**
1. **Delete the PWA** from the home screen (press and hold, Remove App, Delete App). This matters most: re-adding is what makes iOS read the new startup images. An existing install keeps the old empty set.
2. Safari → Settings → Safari → **Clear History and Website Data**.
3. Open **clipershq.com** in Safari → Share → **Add to Home Screen**.
4. **Fully close it:** swipe up and swipe the app card away. Swipe Safari away too. Wait about 30 seconds.

**Now tap the icon and watch the first two seconds.**

| Moment | PASS | FAIL |
|---|---|---|
| The instant you tap | **Blue screen with the white triangle** | Black or blank |
| Next few seconds | Still blue, no black at any point | Any black flash |
| Then | Blue fades into the app | A black gap between splash and app |

**The single question: was there ANY black, even for a fraction of a second?** If yes, it failed, whatever any script says.

**Do it twice**, and once on cellular with Wi-Fi off, which is the slow path where the black used to be longest.

**What a pass does NOT mean:** the app still takes the same few seconds to become usable. This round made that wait **blue and branded instead of black**. It did not make it shorter. Making it shorter is a separate piece of work (service worker precaching and the 222 KB render-blocking stylesheet), deliberately not attempted here.
