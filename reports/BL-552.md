2. **TikTok is not missing a tier-2 fallback. It has two, and they run.** Tier depth on TikTok escalations: **172 logged 3 tiers**, **39 logged 4**, only 6 logged 1. The cascade ran `tiktok-plain-fetch` → `tiktok-apidojo-profile` → `tiktok-browserless-content` and **still did not find the code**.

## PART 4 — the 200-second budget and the self-imposed concurrency

**Both confirmed on main, unchanged.**

* `apify.ts:1284` — `const TICK_HIKER_BUDGET_MS = 200_000;`
* `apify.ts:1292-1293` — the clock starts at `cronTickStartMs`, the **cron tick start**, not when the IG fan-out begins. Every millisecond spent on TikTok, YouTube, DB work and everything else earlier in the tick is charged against HikerAPI's budget before Hiker is ever called. Any IG clip reached after 200s is handed to Apify **without HikerAPI being tried**.
* `apify.ts:1229` — `const HIKER_GLOBAL_CONCURRENCY = 10;` while HikerAPI's own `/sys/balance` reports `rate: 15`. **We self-limit to two thirds of the allowed rate with no stated reason**, which lengthens the fan-out and makes abandonment more likely.

**Recommendation: both, in this order.** Measuring from `fanStart` is the correctness fix and it is the one that matters: the budget is meant to bound *Hiker's* work, and charging it for the tick's earlier phases is simply the wrong clock. Raising concurrency toward 15 is a throughput improvement that shortens the window in which abandonment can happen, but it treats the symptom. **Do the clock first; it is a one-line change of which timestamp is read, and it cannot change any view count** (it only decides whether Hiker is asked before Apify). **I changed neither.**

**Still not measurable:** whether a production tick actually reaches the IG fan-out with budget left. `budget_abandoned` is not persisted (BL-551 is fixing that). BL-550's own sweep of all 1,268 clips at concurrency 10 took 137.5s, inside the budget, but that was a standalone sweep with the whole 200s to itself, which is exactly the condition production does not have.

### THE LANDMINE — the stratified shadow proof required before any view-serving endpoint changes

BL-544 proved **84.1% of TikTok clips at or above 10k views are DISPLAY-ROUNDED**: same video, same minute, one endpoint said **16,466** exact, another **16,500** rounded, and the stored value was **16,300**. LamaTok **already serves 68.5% of TikTok polls**, so this is live, not hypothetical. **Which endpoint answers is a pricing decision**: every clipper's earnings are `(views / 1000) × cpm`.

**The proof that must pass first:**

1. **Stratify at the 10k boundary.** At least **100 clips below 10k** and **100 at or above 10k**, per platform. The rounding behaviour differs across that boundary, so a blended pass rate would hide it.
2. **Fix the pass criteria in advance, per stratum**, before looking at any data. Below 10k: exact match. At or above 10k: decide explicitly whether rounded-versus-exact is acceptable, and if it is, decide **which one is canonical**, because switching silently re-prices every clip above the boundary.
3. **Zero null-versus-real cases.** Any clip where the new source returns null or 0 while the incumbent returns a real number is a **hard fail**: that is the case that freezes a clipper's earnings.
4. **Same timestamp, both sources**, and run **7+ days** so lag shows up and not just value.
5. **Match ids, never count rows.** BL-550's trap: an Apify TikTok probe returned an item for a video **not in the request list**, so a naive row count would have invented a fall-through that did not exist. Every comparison must join on the id it asked for and discard anything else. My own probes above matched ids for exactly this reason.

## PART 6 — the verdict

**Can Apify reach zero? On the non-money paths, yes, today.** Bio, avatar, and the dead legacy batch are provably replaceable, and the submit path is *already* Apify-free on both platforms.

**On the view-count paths, the evidence says yes and the discipline says not yet.** BL-544 and BL-545 concluded Apify must stay; BL-550 then proved that fear rested on a phantom population (the "669 clips a week" HikerAPI supposedly could not resolve do not exist; Apify resolved **0 of 37**, and agreed **10 of 10** where both answered). **I settle it on measurement, not deference to either:** nothing found here shows Apify resolving anything the cheap providers cannot, so the *technical* case for retiring the view path is strong. What blocks it is not capability, it is that **84.1% display-rounding above 10k makes the switch a pricing decision**, and that proof has not been run. That is a governance gap, not a technical one.

### The stale env var: confirmed, and inert

`.env.local` still carries **`APIFY_INSTAGRAM_ACTOR=apify/instagram-reel-scraper`** — the actor BL-543 removed for a measured 0.0% win rate — and `APIFY_TIKTOK_ACTOR=clockworks/tiktok-scraper`. The probe printed it live: `APIFY_INSTAGRAM_ACTOR : "apify/instagram-reel-scraper"`.

**It is harmless today**, because the only reader of `getInstagramActor()` is `fetchInstagramStatsBatchLegacy`, which is dead code (call site 7). But it is a loaded gun: anyone who revives that function inherits a 0.0%-win-rate actor. **I could not check the production value.** Railway's env is not readable from here, and I did not attempt to read it. **The owner should check Railway for `APIFY_INSTAGRAM_ACTOR` and delete it**; it is a one-line change with no code impact.
