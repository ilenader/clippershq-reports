# BL-642 — Can /api/campaigns/spend be made fast without breaking anything?

## VERDICT (one line): YES, and the right fix is a short role-keyed response cache, because this endpoint is provably DISPLAY ONLY (no budget gate, cap check or eligibility rule reads it); but the honest recommendation is to WAIT, because my direct measurement on the real `clips` table says the wall is far further out than BL-639's cross-table proxy suggested, and the two fixes BL-639 recommended are the two I measured as either useless or actively harmful.

**2026-07-23 · AUDIT ONLY. READ ONLY on code, config, data, schema, index and money.** Nothing built, changed, created or pushed to product. EXPLAIN ANALYZE was run read-only; **no index, extension or table was created** (hypopg is available but NOT installed, deliberately). The probe script lived in the session scratchpad, never in the repo.
**Base:** origin/main `9f5de583` · **Branch:** checkpoint/BL-642 · isolated worktree, no node_modules junction. A markdown-only diff cannot change tsc or build, so none was run and none is claimed.

**Live shape (2026-07-23):** 4,132 clips (2,743 in the aggregate), 1,316 agency_earnings rows, 30 campaigns, 1,165 clippers, marketplace earnings tables **empty**.

## PART 1 — what exactly is this number, and who sees it?

### It is not two aggregates. For a CLIPPER it is FOUR.
BL-639 counted two. Correcting that, `src/app/api/campaigns/spend/route.ts` runs:

| # | Line | Query | Bound? | Measured median |
|---|---|---|---|---|
| A | `:63` | `clip.groupBy` by campaignId, `_sum: earnings`, where APPROVED + not deleted + **not videoUnavailable** | no campaign filter, no `take` | **2.36 ms** |
| B | `:94` | `agencyEarning.groupBy` by campaignId, `_sum: amount` | **no `where` at all** | **0.62 ms** |
| C | `:146` | `marketplaceCreatorEarning.groupBy`, joins `clip` | CLIPPER only | 0.06 ms (table empty) |
| D | `:151` | `marketplacePlatformEarning.groupBy`, joins `clip` | CLIPPER only | 0.07 ms (table empty) |

C and D are free today only because both tables hold zero rows. They **join `clips`**, so once the marketplace carries volume they scale worse per row than A and B, not better. Any fix must cover all four.

### The asymmetry BL-638 found is real and still live
The clip side excludes `videoUnavailable` clips; the agency side has **no filter whatsoever**. So the figure is neither clipper-only nor a true total. It is a third definition: **clipper earnings on live clips only, plus the owner's entire agency accrual**. Confirmed to the cent on live data, matching BL-638 exactly:

### Do clipper and owner see the same number? TODAY YES, ON EVERY CAMPAIGN. But the rule that separates them is armed.
For a CLIPPER the route applies `resolveClipperFacingSpent` (`:168`), which raises the figure to the full budget once the clipper pool is exhausted (`campaign-clipper-view.ts:181`). **I evaluated that predicate against live data: it fires on ZERO campaigns right now.**

The closest is STRAENGE: clipper side **$1,997.56** against a pool cap of **$2,000.00**, short by **$2.44**. Worth stating plainly, because it is not in any prior report: **BL-535's headline behaviour has silently stopped firing.** BL-535 shipped because STRAENGE's pool was full and the bar advertised unreachable headroom; the 2026-07-18 `videoUnavailable` stamping that BL-638 documented removed $4.15 of STRAENGE clip earnings from the live sum and pushed it back under its own cap. The rule is state-derived and correct, so this is the rule working, not breaking, but the practical effect is that the phantom-headroom case BL-535 was built for is currently latent rather than active.

**Consequence for any fix: the cache MUST be keyed by role.** CLIPPER and non-CLIPPER responses are produced by different code paths and are identical only by today's coincidence. A single shared cache entry would serve one role's shaping to the other the moment any campaign's pool fills.

### The safety fact that decides this whole round
**No decision path reads this endpoint.** Every budget gate, cap check and eligibility rule calls `getCampaignBudgetStatus` from `balance.ts` directly against the database: the submit gate (`clipper-submit-core.ts:101`), the L1 hard lock (`clip-earnings-writer.ts:197`), auto-pause (`tracking.ts:4190`, `:4283`), clip review, owner submit, marketplace listing, proportional cut, cpm-restamp, gamification, archive, force-recalc and payout adjust. **A stale value from this endpoint can therefore mislead an eye, but it can never move a dollar or open a cap.**

## PART 2 — every option, with its real risk

A wrong money number is far worse than a slow one. Options are ranked on that basis, not on speed.

### (d) Index or query rewrite. MEASURED, and it is NOT the free win it appears to be.
`clips_campaign_status_idx` on `(campaignId, status, videoUnavailable, isDeleted)` **already exists and already matches aggregate A's WHERE clause exactly.** Postgres still chooses a Seq Scan, correctly, because the index does not carry `earnings`, so it would face 2,743 random heap fetches against a 308-page table. The only rewrite that changes the plan is a **covering** index carrying `earnings`.

**The hidden cost, and it is a serious one.** Live table statistics: `clips` has **3,282 inserts against 247,676 updates, of which 228,245 (92.2%) are HOT updates.** HOT is possible *only while no indexed column changes*. The tracking cron rewrites `clip.earnings` continuously across 4,036 active tracking jobs. **Putting `earnings` into an index converts roughly 228,000 HOT updates into non-HOT ones**, which means write amplification on `writeClipEarnings` (the single most safety-critical write path on the platform), faster index and heap bloat, and a visibility map that is dirtied constantly, which in turn degrades the very Index Only Scan the index was added for. **Failure mode:** no wrong number, but a slower money write path and a covering index that quietly stops covering. **How you would notice:** `n_tup_hot_upd` collapsing as a fraction of `n_tup_upd`, and index size growing between vacuums. This option is not free and should not be described as low risk.

### (c) Narrowing the aggregate. MEASURED, and it does NOT work.
BL-639's recommendation was `where: { campaignId: { in: <displayed ids> } }`. I measured it: **2.59 ms scoped versus 2.36 ms unscoped, and it is still a Seq Scan.** It is *slower*. The reason is structural: the campaign list displays essentially every live campaign, so 12 ids already cover 2,725 of the 2,743 qualifying rows. Filtering removes 0.7% of the rows and adds a per-row `ANY()` check. **Narrowing cannot help an aggregate whose grouping key is the same thing you would filter on.** Changing *which* rows are summed (for example dropping retired clips, or the archived-campaign decision documented at `:34-62`) would change displayed numbers and is out of scope for a performance round.

### (a) Short-lived role-keyed cache. RECOMMENDED.
**How it works:** memoise the built `spendMap` per role for a few seconds to a minute. **Speedup:** collapses N concurrent viewers to one query per TTL. **What goes stale:** the displayed progress bar, by at most the TTL. **What cannot go wrong:** nothing in the money core, because no decision path reads this endpoint (proven in PART 1). **Failure mode if it breaks:** a bar reads a few seconds behind; refresh corrects it. **How a reader notices:** the number does not move immediately after a submit, which is already true because earnings do not exist at submit time (see PART 3).

**Why it beats the alternatives on the real cost driver.** The driver is not single-query latency, it is *concurrency multiplied by table size*. At 10,000 clippers a hundred simultaneous campaign-list views cost a hundred full scans; caching turns that into one, regardless of viewer count. The index attacks only the row term and taxes the write path; narrowing attacks nothing. **The cache is the only option that removes the multiplier, and it is the only one that touches no money code at all.**

### (b) Materialised or incrementally maintained running total. REJECTED.
**Risk:** it drifts the moment any write path forgets to update it, and a drifted money figure is precisely the bug class that has cost this platform weeks. There are many writers to `clip.earnings` (tracking, force-recalc, payout adjust, proportional cut, restamp) plus `agency_earnings`. **Detection would require a reconciliation job that runs the very aggregate we were trying to avoid**, so the design pays the full cost anyway and adds a drift surface. Not worth it for a display figure.

### (e) Do nothing yet. HONEST, and partly correct.
See PART 4 for the threshold and watch query.

### Risk-adjusted ranking
**1. (a) cache**, the only option with zero money-path contact and the only one that removes the concurrency multiplier. **2. (e) wait**, correct today. **3. (d) covering index**, real read win but a real write tax on the money path, and it should never ship without measuring `n_tup_hot_upd` before and after. **4. (c) narrowing**, measured useless. **5. (b) materialised total**, drift risk on a money figure for no proportionate gain.

## PART 3 — every edge case

• **Clipper submits a clip and immediately reloads.** The bar does not move, **with or without a cache**. A submitted clip is PENDING with zero earnings and is excluded by `status: "APPROVED"` at `:66`. Earnings only appear after review plus a tracking tick. A cache adds nothing confusing here, because the honest current behaviour is already "no immediate movement".
• **Two clippers submit near a cap.** Neither the cached nor the live figure gates anything. The real cap is enforced by `getCampaignBudgetStatus` inside the submit path (`clipper-submit-core.ts:101`) and clamped per write by the L1 lock (`clip-earnings-writer.ts:197`). A stale bar can make a clipper *believe* room remains; the server will still refuse. This is a UX question, not a money question, and it is unchanged by caching because the bar was already behind by a tracking tick.
• **Top up, pause, resume, PAST, COMPLETED (BL-641 is moving somesome).** The endpoint reads no status field, so a transition changes the figure only through the underlying sums, which a TTL delays by seconds. PAST campaigns with a `manualSpent` override bypass the endpoint entirely (`CampaignsRedesign.tsx:109`). **One thing a build round must verify:** a status transition that changes `budget` or the locked share flips the CLIPPER exhaustion predicate, so the role-keyed cache must be invalidated or short enough that the two roles cannot visibly disagree across the transition.
• **Retired videoUnavailable clips.** Excluded from the clip side, included in the agency side, and that asymmetry is unchanged by every option here. On somesome that is **$3,433.29 excluded from the numerator while the owner's full $3,427.03 accrual is added**. A cache preserves this exactly; it must not be presented as fixing it. Fixing it is BL-638's question, not this round's.
• **Ghost-fee campaigns (BL-630/631).** The endpoint sums realised earnings and never reads `platformFeePctDecimal`, so it reflects **actual money moved**, not marketed or real budget. The denominator on screen is the marketed budget. All 30 campaigns carry a NULL fee today, so nothing is currently affected, but the first fee campaign will show spend against the marketed total while the earnable pool is smaller. Caching does not change this and does not fix it.
• **Timezone and day boundaries.** **Nothing in this endpoint buckets by day, week or timezone.** There is no date filter, no `startOfDay`, no local offset. A cached value therefore cannot cross a day boundary wrongly. This is the one edge case that is provably a non-issue, which matters given the +02:00 incident.
• **Empty campaign:** absent from both groupBy results, renders `|| 0`, bar at 0%. **One clip:** one group, exact. **Exactly at cap:** the clipper predicate uses `>=` (`campaign-clipper-view.ts:162`), so exactly-at-cap counts as exhausted and reads 100%; `Math.max` guarantees an over-budget campaign keeps its real number rather than being clamped down.

## PART 4 — verdict and spec

### Recommendation: WAIT, then cache. Do not index, do not narrow, do not materialise.

**The honest measured position.** A full aggregate on the real `clips` table is **2.36 ms at 4,132 rows**, and a Seq Scan is linear in pages. Extrapolating from the measured baseline (308 pages, 4,132 rows) rather than from BL-639's cross-table proxies: **~20 ms at 36k clips, ~90 ms at 162k, ~215 ms at 377k.** BL-639's 733 ms and 2,440 ms figures came from `clip_stats` and `email_events`, which are wider rows crossing Postgres's parallel-plan threshold, so **they materially overstate the curve for `clips`.** BL-639's own caveat said as much; this round measured the actual table and the wall is further out than "biting near 10k clippers / 36k clips" implied. Also relevant: **none of the five callers auto-refreshes**, so this fires once per page view, not on a 60 s loop like `/clips`.

**So the trigger is throughput, not latency.** Build the cache when either of these is true:

```sql
-- WATCH QUERY. Build when clips_in_aggregate crosses ~50,000,
-- or when the p95 of this aggregate crosses 25 ms.
SELECT count(*) AS clips_in_aggregate
FROM clips
WHERE status='APPROVED' AND "isDeleted"=false AND "videoUnavailable"=false;
-- today: 2,743
```

### Build spec, for the later round
**What changes:** one file, `src/app/api/campaigns/spend/route.ts`. Wrap the four aggregates in an in-process memo keyed **`role === "CLIPPER" ? "clipper" : "full"`**, TTL 30 s to 60 s, serving the built `spendMap`. Nothing else. No schema, no index, no money file, no `balance.ts`, no write path.

**What must be proven before merge:**
1. **Every surface's number is byte-identical for today's data.** Capture the full `spendMap` for both roles before and after; all 30 campaigns must match to the cent.
2. **Role separation.** A CLIPPER request must never be served a non-CLIPPER entry or the reverse. Test by forcing the exhaustion predicate true on a fixture and asserting the two roles diverge as designed.
3. **No decision path touched.** `getCampaignBudgetStatus` callers unchanged; the 6 money files byte-identical by blob OID.
4. **Cold start correctness.** First request after boot returns live values, not an empty map.

**The exact test that catches it going wrong:** seed two campaigns, one with genuine headroom and one with an exhausted clipper pool, request as CLIPPER and as OWNER within the same TTL window, and assert four values: the owner sees real total spend on both; the clipper sees real total spend on the first and the full budget on the second. **A single shared cache entry fails this test**, which is precisely the failure mode a naive implementation would ship.

**Rollback:** revert the single commit. There is no data, schema or index change to undo, which is a large part of why this option was chosen.

### What MUST NOT change
Every surface's displayed number stays identical for today's data. The clip side keeps excluding `videoUnavailable` and the agency side keeps its unfiltered sum, asymmetry and all, because changing which rows are summed is a **money-meaning** change and belongs to BL-638's thread, not to a performance round. **No decision path may ever read a cached value**, and the named paths that must keep reading `balance.ts` live are: the submit gate, the L1 hard lock in `clip-earnings-writer.ts`, auto-pause in `tracking.ts`, clip review, owner submit, marketplace listing, proportional cut, cpm-restamp, gamification, archive, force-recalc and payout adjust.

### What could not be measured, and why
• **The covering index's actual speedup.** Proving it requires creating the index or installing hypopg, both of which are changes. It is available but was deliberately not installed. The HOT-update tax on it is measured and real regardless.
• **The exact `clips` curve at 10x and 100x**, which needs a seeded copy of the table. The linear page model above is derived from the measured 308-page baseline and is a better estimator than a different table's curve, but it is still a model.
• **Real concurrency today.** No request-rate telemetry was read, so the throughput argument rests on clipper count and page-view shape rather than on observed queries per second.
