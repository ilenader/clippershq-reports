# BL-595 — Email Deliverability Emergency Audit (READ ONLY)

Date: 2026-07-20. Branch: `checkpoint/BL-595`. Scope: prove whether the growth
dashboard's "32 SENT, 0 DELIVERED, 0 opened, 0 bounced, 0 spam" means email is
broken platform wide, or is a measurement artifact. Nothing was changed: no email
sent, no env, no DNS, no code. All findings are evidence backed.

## VERDICT (one line)

Email is **delivered but unmeasured**. Mail is leaving normally (61 real
Resend accepted sends on record, most within the last four days). The zeros are a
dashboard/webhook measurement gap, **not** an outage. Payout and verification
email are unaffected.

## PART 1 — is mail broken, or is the dashboard lying?

**It is the dashboard (case a): mail flows, the counters are unfed.** Proof:

• **The send path only records a `sent` row after Resend returns HTTP 2xx.**
  `sendGrowthEmail` writes `EmailEvent("sent", dryRun:false)` **only after**
  `sendEmail()` returns `true` (`src/lib/growth-email/send-growth-email.ts` steps
  3 and 4), and `sendEmail()` returns `true` **only** on `res.ok` from
  `POST https://api.resend.com/emails` (`src/lib/email.ts:229`). A rejected send
  (401/403 bad key or domain, timeout, 4xx) returns `false`, yields status
  `"error"`, and writes **no** `sent` row.

• **There are 61 live `sent` rows.** `email_events` where `dryRun=false`:

  | type | dryRun | count |
  | --- | --- | --- |
  | sent | false | 61 |
  | sent | true | 377,441 |

  First live send `2026-07-16 12:36:36`, last `2026-07-20 08:26:54` (today).
  Recent triggers: `STALLED_7D`, `FIRST_CLIP_CELEBRATE`. Because a `sent` row
  cannot exist without a Resend 2xx, **Resend accepted 61 real messages.** If mail
  were genuinely failing (nothing leaving), the count would be 0, not 61. The
  "nothing leaves" case is disproven.

• **"delivered/opened/clicked" are populated ONLY by a Resend webhook.** The
  dashboard computes them purely from `email_events` rows of those types
  (`src/app/api/admin/growth/route.ts:123-129`), and those rows are written
  **exclusively** by the Resend webhook handler
  (`src/app/api/webhooks/resend/route.ts` -> `recordEmailEngagement`). `bounced`/
  `complained` ("spam") come from `email_suppressions`, also webhook only
  (route.ts:72-79). No webhook event means those columns are structurally frozen
  at 0, regardless of how much mail flows.

• **`email_events` contains ZERO delivered/opened/clicked rows, ever** (only
  `sent`, per the table above). So the `email.delivered` / `email.opened` /
  `email.clicked` events are not reaching (or not being recorded by) the platform.

• **The send path never captures the Resend message id.** `sendEmail()` returns a
  boolean and never reads the response body's `id` (`src/lib/email.ts:229-236`);
  the `sent` rows carry only `{bucket, idempotencyKey, source, variant}` in
  metadata (verified on the 6 most recent rows), **no provider id.** So a send can
  be correlated to a delivery **only** through the webhook, and the webhook is the
  thing that is not landing delivery events. Answer to "are the 32 STALLED_7D
  sends recorded with provider ids": **no** — by design the send path stores none.

Conclusion for Part 1: the delivered counter is fed by a webhook that is not
delivering `email.delivered` events, so the counter is stuck at 0 while mail
actually flows. **Delivered but unmeasured.**

## PART 2 — the webhook IS wired (bounce proves it); it just lacks the delivered event

Refined with evidence, the webhook is not dead. `email_suppressions` holds 4 rows,
all written by the webhook's bounce path (`route.ts:267`):

  | reason | detail | createdAt |
  | --- | --- | --- |
  | BOUNCE | Transient | 2026-06-30 23:37:17 |
  | BOUNCE | Transient | 2026-06-25 08:01:33 |
  | BOUNCE | Transient | 2026-06-24 06:56:44 |
  | BOUNCE | Transient | 2026-06-24 01:37:28 |

In production, an unset `RESEND_WEBHOOK_SECRET` makes `verifySignature` return
`ok:false` -> 401 -> nothing processed (`route.ts:55-59, 210-222`). Four
signature verified bounce rows therefore **prove the endpoint exists, the signing
secret is set in prod, and Svix verification passes.** The webhook works. (Note:
`RESEND_WEBHOOK_SECRET` reads MISSING in the local `.env.local`, which is
expected — prod env lives in Railway, and the 4 processed bounces confirm it is
set there.)

What the webhook is NOT doing: writing a single `delivered`/`opened`/`clicked`
row. The clean split (bounces captured, delivered absent) points to the exact
cause:

1. **The endpoint is subscribed to `email.bounced`/`email.complained` but NOT to
   `email.delivered`.** That single missing subscription is why "delivered" is 0
   while mail (and even bounce capture) works.
2. **Open and click tracking are off** (Resend default). Without them Resend emits
   no `email.opened`/`email.clicked`, so those columns can never populate either.
3. **The 0 in the "bounced/spam" column is a window artifact, not zero bounces.**
   The dashboard counts suppressions with `createdAt >= since` (route.ts:74). The
   only 4 bounces are from Jun 24–30, older than the current stats window, so they
   fall outside it. Real bounces exist; the recent window simply has none.

**Domain authentication is healthy** (public DNS, `8.8.8.8`, read only lookup —
the only probe performed; no mail sent, nothing written):

• `send.clipershq.com` TXT: `v=spf1 include:amazonses.com ~all` — Resend's SES
  based SPF, present and valid.
• `resend._domainkey.clipershq.com` TXT: DKIM record present.
• `_dmarc.clipershq.com` TXT: `v=DMARC1; p=none;` — DMARC present (monitor mode,
  non blocking).

So SPF, DKIM and DMARC are all in place on the sending subdomain. The API key
(`EMAIL_API_KEY`, `re_...`) is valid — 61 accepted sends prove it. Rate limits
(`MAX_LIVE_EMAILS_PER_SWEEP=50`, `EMAIL_CONCURRENCY<=2`) throttle volume but never
drop it to zero. None of these are the cause.

One config to confirm (owner only): the local `EMAIL_FROM` is the Resend sandbox
`Clippers HQ <onboarding@resend.dev>`. The sandbox can only mail the account
owner, yet 61 sends to real users were accepted — so prod almost certainly uses
the verified `@send.clipershq.com` / `@clipershq.com` domain. Worth a 10 second
check that Railway's `EMAIL_FROM` is the verified domain, not the sandbox.

### Blast radius (same path carries payout and verification mail)

`sendEmail()` in `src/lib/email.ts` is the **shared chokepoint** for BOTH growth
mail AND transactional mail: `sendPayoutApproved`, `sendPayoutAdjusted`,
`sendClipApproved`, `sendCallScheduled` (verification call), plus the magic link
verification routes all call it. **Because mail is actually leaving (2xx),
transactional payout and verification email are being delivered normally.** The
webhook gap is measurement only. The one real (slow, non urgent) side effect: the
`email.delivered`/open data is missing for engagement gating, and bounce
suppression only fires for addresses that hit the bounce webhook — which does
work, so dead addresses are being suppressed. **No clipper is failing to receive a
payout or verification email because of this.** This is not a payout/verification
outage.

## PART 3 — cause and ranked fix (all owner only, no code required)

**One line:** delivered but unmeasured — the Resend webhook is subscribed to
bounce/complaint but not to `email.delivered`, and open/click tracking is off, so
the dashboard's delivered/opened/bounced columns cannot populate while mail flows
normally (61 Resend accepted sends; 4 verified bounces).

Ranked exact fixes:

1. **Resend dashboard -> Webhooks -> the existing `clipershq` endpoint -> add the
   `email.delivered` event** (and confirm `email.opened` / `email.clicked` are
   subscribed). This alone unfreezes the "Delivered" counter. No code, no deploy.
2. **Resend dashboard -> Domain / Sending settings -> enable Open tracking + Click
   tracking** (off by default). Populates the opened/clicked columns.
3. **Leave `RESEND_WEBHOOK_SECRET` in Railway as is** — it is set and working (4
   verified bounces prove it). Only re verify it if the Resend endpoint or its
   signing secret is ever rotated.
4. **Confirm Railway `EMAIL_FROM`** is the verified `@send.clipershq.com` /
   `@clipershq.com` domain, not the `onboarding@resend.dev` sandbox seen locally.
   DNS shows the domain is fully set up (SPF/DKIM/DMARC), so this is a
   confirmation, not a change.
5. **Dashboard window note (optional):** the "bounced/spam = 0" is because the 4
   real bounces predate the stats window; widening the window (or accepting it)
   makes historic bounces visible. Cosmetic.
6. **Optional code, later (not required to fix the outage — there is no outage):**
   capture the Resend response `id` into the `sent` row metadata so sends map
   directly to their delivery/open events even without the tag round trip.
