# BL-611 — Can Apify be hard-cut to zero without breaking anything?

**2026-07-21 · AUDIT ONLY. READ ONLY on code, config, data and money. Nothing changed, pushed to main, merged, flipped, or built.**
**Base:** origin/main @ `4f1279e` (post BL-610) · **Branch:** checkpoint/BL-611 · **Worktree:** C:/b611
**No Apify actor was run. The hard-cut was modelled on paper and against the ALTERNATIVE providers' already-proven behaviour (BL-550, BL-610), never by disabling anything.**

## THE ANSWER (one line)

**NO, a hard Apify-cut is NOT safe today. Exactly ONE path fails broken: Instagram account verification (`verify/route.ts:238`), the earning gate. Every other path fails safe by construction (NULL never 0). Ship the BL-589 verify-to-HikerAPI cut FIRST, then the hard-cut is safe.**

## PART 2 — the five paths the owner named

### 1. SUBMIT a clip — FAIL SAFE on every platform
BL-552 proved submit is Apify-free (TikTok to LamaTok, IG to HikerAPI). Confirmed on current main with one nuance: the submit **freshness** retry (`fetchClipFreshnessWithRetry`, `apify.ts:2402`) deliberately passes `skipHikerOverlay:true` (`:2457`, BL-137) so IG reads a parseable `createdAt` from Apify. If Apify is cut, that call returns `null`, and the caller (`clipper-submit-core.ts`, the freshness handler) treats a null result as **transient: it accepts the clip and lets the cron verify**, and the first ClipStat is skipped (see 2.3). TikTok freshness uses LamaTok (not the Hiker overlay, not Apify) and is unaffected. **A clipper can still submit every platform with Apify at zero.** Soft consequence: with Apify off, IG loses its only submit-time `createdAt` source, so the 30-minute posting-window gate is not enforced for IG at submit (the clip is accepted and verified later). That is a policy softening, **not a break** — no clip blocked, no 0 stored.

### 2. CONNECT an account / VERIFICATION — **FAIL BROKEN for Instagram. This is the one break. Stated plainly.**
Traced end to end: `POST verify/route.ts` calls `checkInstagramBio` (`:217`), whose **ATTEMPT 1 is Apify** (`:229-296`) and only falls to Browserless (`:301`) on Apify failure. BL-589 flagged this and it is still true on `4f1279e` — the BL-589 verify-to-HikerAPI cut was **never shipped** (the `BL-590` that merged was the gone-counter / retire-dead-clips work, a different change). So a hard-cut TODAY sends every IG verification to the **Browserless-only** path. BL-552 measured why that is unsafe: Instagram verification escalates **zero** times precisely because it reads a structured `biography` field from an API, while the HTML-scraping tiers (Browserless) "run and fail" on client-rendered bios. **Therefore, with Apify cut, a clipper connecting or re-verifying a NEW Instagram account may be unable to verify, and verification is the gate to earning.** Blast radius: NEW IG account verifications only (already-verified accounts keep earning; they do not re-verify). TikTok/YouTube verify start with free plain-fetch and fall to Browserless; their Apify leg (site 12) is a middle tier that signals transient and is skipped without blocking. **The IG verification bio call is the single FAIL-BROKEN path.**

### 3. FIRST video tracking (first ClipStat on submit) — FAIL SAFE (NULL, not 0)
The first ClipStat is written from the freshness result. With Apify cut, IG freshness returns null → `fetchedStats` stays null → `clipper-submit-core.ts` **skips the first ClipStat** when the resolve is null (BL-605, the null-safe skip merged in BL-609: `if (resolvedFirstViews != null)`). It stores **NULL (no row), never 0**. Tracking writes the first real stat on the next tick. FAIL SAFE.

### 4. VIEW tracking (the poll) — FAIL SAFE (keep last-known via NULL)
The IG batch fan-out (`fetchInstagramStatsBatch`, `apify.ts:1373`) calls HikerAPI first at `HIKER_GLOBAL_CONCURRENCY=15` (`:1371`) with `TICK_HIKER_BUDGET_MS=TRACKING_LOOP_BUDGET_MS` (210s, `:1444`), all BL-610. Budget overflow is **left absent from `primaryMap`** (`:1504-1515`, `[SKIP-APIFY] reason=budget-overflow-wait-next-tick`) → null → keep last-known → re-probed next tick. Definitive-gone (`:1527`) and quarantined carousels skip Apify. Only `unknown`/opaque 200s reach Apify (the rescue floor). With Apify cut, those rescue-floor clips return null → `tracking.ts:1442` keeps last-known. **Nothing stores 0.** FAIL SAFE.

### 5. SLIDESHOWS / CAROUSELS — FAIL SAFE
BL-610 quarantines any carousel HikerAPI cannot count (image-only, or mixed with hidden child `play_count`): `verdict:"quarantine"`, `useResult:false`, caller SKIPS Apify (`apify.ts:2309-2312` single, mirrored in the batch), keeps last-known (NULL never 0), and is **not** added to the gone counter (alive, not gone). A mixed carousel with a readable summed count is a genuine HikerAPI HIT and never touches Apify. So with Apify cut, a quarantined clip keeps its last real number and does not freeze-to-zero. FAIL SAFE.

## PART 3 — the 2 to 3 percent genuine-rescue floor

**What it is.** BL-550 probed all 1,268 live IG clips: HikerAPI resolved 100% of the resolvable population and **Apify resolved 0 of 37** clips HikerAPI could not (all genuinely gone). BL-610's live sweep (6 days ago, 93 clips) measured the residual Apify rescue floor at **2 of 93 (~2%)** — the rare `unknown`/opaque 200 that HikerAPI declined but Apify can still read. On TikTok, BL-550 found LamaTok resolves 73.5%, 24.8% are genuinely gone (Apify 0/12), and only **1.75%** are a fixable transient `http_500` (which a LamaTok retry, not Apify, addresses).

**Scale today (disclosed read-only DB count this round).** Active tracked clips: **~923 Instagram, ~1,022 TikTok, ~1,317 YouTube**. Applying BL-610's ~2% floor to the IG population is on the order of **~20 IG clips per tick** that would have used the Apify rescue floor — and BL-550 proved the truly-only-Apify-can-resolve subset of those is **~0**.

**Is it acceptable? YES.** With Apify cut, every rescue-floor clip returns null and **keeps its last-known view count** via `tracking.ts:1442` — it does **not** freeze-to-zero and does **not** flip `videoUnavailable`. The only cost is that such a clip **stops updating** until it resolves on a cheap provider on a later tick or is retired by the dead-clip cron. Because earnings are `(views/1000) × cpm` off the **stored** value, a clip that keeps its last real number keeps paying its last real amount; it simply does not grow while stalled. **No clip in this floor freezes earnings-to-zero.** That is the whole risk of PART 3, and it is within the NULL-never-0 invariant.

## PART 4 — the verdict and the safe-cut mechanism

### Verdict: NO, not safe today. One thing must be true first.
Every path fails safe **except** Instagram account verification (`verify/route.ts:238`), which on a hard-cut degrades to an unreliable Browserless-only bio read and can lock a clipper out of verifying, and therefore earning. That is the one FAIL-BROKEN path.

### The one fix that must ship first (BL-589's proven design)
**Move the IG verification bio read off Apify to HikerAPI**, exactly as BL-589 proved zero-loss by live probe:
1. Add `fetchHikerInstagramProfileByUsername(username)` to `src/lib/scraper-providers/hikerapi.ts`: `GET /v2/user/by/username`, header `x-access-key`, map `biography` / `follower_count` / `is_verified` / `hd_profile_pic_url_info.url ?? profile_pic_url`. Log via the existing `recordHikerUsage`.
2. In `checkInstagramBio` insert HikerAPI as **ATTEMPT 0 before** the Apify block (`verify/route.ts:229`); keep Apify as fallback for one release, measure, then remove.
3. Cut the cosmetic sites (`verify/route.ts:884`, `account-profile.ts:132`) to the same helper; their failure is already harmless.

BL-589 proved HikerAPI returns every field these sites consume (`biography` with emoji/CJK intact, `follower_count`, `is_verified`, HD avatar). Once ATTEMPT 0 is HikerAPI, the Apify attempt becoming null is harmless, and the hard-cut is safe.

### Then: the safe hard-cut MECHANISM (a single global kill switch — designed, NOT built)
Add one env flag, e.g. **`APIFY_HARD_OFF=1`**, read by one helper `apifyHardOff()`. At the **top of every Apify-actor-invoking function** (the 13 sites in PART 1) add a guard that returns the site's existing **null-shaped** result WITHOUT calling Apify:
• View funcs (sites 1 to 8, 13): `return { stats: null, provider: "apify-off" }`; batch funcs: leave the URL absent from the result map (identical to an un-started chunk).
• `verify-cascade.ts` tier 2 (site 12): `return { found: false, transient: true }`.
• `account-profile.ts` (site 11) and `verify/route.ts:884` (site 10): return the existing `{ok:false}` / skip branch.
• `checkInstagramBio` ATTEMPT 1 (site 9): skip the Apify block — **but only after the HikerAPI ATTEMPT 0 above exists**, otherwise this alone routes verify to Browserless.

**Why it guarantees NULL-never-0 and never-block by construction.** The guard returns the exact same null shape a genuinely-dead URL already produces, so it exercises only code paths BL-543 / BL-605 / BL-610 already proved keep last-known, store no 0, and block no user. It touches **only the Apify call sites** — no money-file logic, no `writeClipEarnings`, no change to `tracking.ts`'s null handling (the safety is already there). It is fully reversible: unset the flag and Apify resumes as the fallback floor. **Note the near-equivalence:** every site already guards on `if (!APIFY_API_KEY) return <safe null>` (verified at `verify:231`, `verify:880`, `account-profile:126`, `verify-cascade:193`, and the actor funcs), so **simply emptying `APIFY_API_KEY` is already a near-complete kill switch that fails safe everywhere except site 9** — which is the same reason site 9 must get its HikerAPI source first. An explicit `APIFY_HARD_OFF` flag is preferable only because it is discoverable and self-documenting; the safety property is identical.

**Order of operations for the owner:** (1) ship the BL-589 IG-verify-to-HikerAPI cut; (2) confirm in production that HikerAPI resolves verify bios; (3) flip `APIFY_HARD_OFF` (or empty `APIFY_API_KEY`). After step 1, every one of the five owner-named paths fails safe, and Apify can go to zero without breaking submit, connect/verify, first-video tracking, view tracking, or slideshows.
