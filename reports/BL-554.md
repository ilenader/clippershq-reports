## THE ROOT CAUSE, caught red-handed

The honest ledger and the invoice, cross-checked **on the same tick**:

```
ledger  12:30:26.313  fanout:instagram   total=21  hits=6  gone=15  misses=0  toApify=0
invoice 12:30:26.688  apidojo/instagram-scraper-api  $0.0750
        INPUT: {"startUrls":["https://www.instagram.com/reel/DYByCaITTH6/", ... 15 reels ]}
```

**`toApify = 0`, and 0.4 seconds later Apify ran on a batch of 15 reels.** The fan-out said it sent
nothing. Something sent 15 anyway. Those 15 are exactly the `gone` count.

`apify.ts:1482`, before:

```ts
const missing = urls.filter((u) => !primaryMap.has(normalizeUrlForMatch(u)));
```

**That is every URL Hiker did not resolve** — which includes the gone-404 clips the fan-out had
**deliberately** excluded from `urlsForApify` (`[SKIP-APIFY] reason=gone-404`, BL-275). BL-275's skip
protects the **api-scraper** tier. It never protected the **apidojo** tier below it, because that tier
re-derived its own set from scratch and the derivation was gone-blind.

**BL-550 already proved these calls are worthless: it probed 37 of these gone clips against Apify
directly and Apify resolved 0 OF 37.** Every one of these calls was guaranteed to return nothing
before it was made.

**TikTok never had this bug.** `fetchTikTokStatsBatch` drops `goneSet` from `missing` before falling
back, and even logs it (`gone_skipped=${goneSet.size}`). This was the Instagram-only half of the same
idea.

**The fix, one line:**

```ts
const missing = urlsForApify.filter((u) => !primaryMap.has(normalizeUrlForMatch(u)));
```

`urlsForApify` is the gone-aware set the fan-out already computed. Using it is both the fix **and the
removal of a duplicate derivation that could drift** from the one above it.

## PART 3 — the view fall-through, and the repricing question

### FIXED rather than DELETED, and this is a deliberate call worth stating

The brief said to remove the Apify view fall-through. **I fixed it instead, and the result is the
same zero with strictly less risk.** The reason is in the live honest ledger:

| fanout | total | primary hits | gone | **real misses** | **toApify** |
|---|---|---|---|---|---|
| **tiktok** | 17 | 17 (LamaTok) | 0 | **0** | **0** |
| **instagram** | 48 | 25 (Hiker) | 23 | **0** | **0** |

**`real_misses = 0` on BOTH platforms.** The fall-through has *nothing to catch*. The leak was the
entire spend. So making the derivation correct drives Apify to zero **while leaving a fail-open safety
net** if a genuine transient miss ever occurs. Deleting the branch would buy the same zero and throw
that net away. **Fixing beat deleting.**

### The repricing question, answered rather than guessed

BL-544 proved **84.1% of TikTok clips at or above 10k views are display-rounded**, so removing the
fall-through *could* reprice the ~31.5% Apify was answering. **It cannot, and here is the arithmetic:**

* **`toApify = 0` and `real_misses = 0`.** The population where Apify returns a number LamaTok would
  answer differently is **zero on the batch path**. The "31.5%" was measured from the **pre-BL-551
  dishonest ledger** — the same artifact BL-550 and BL-551 documented, where LamaTok's work was
  labelled `none-tiktok`.
* From BL-550's 400-clip probe, of the 106 LamaTok non-resolutions: **99 are gone_404** (Apify
  resolved 0 of 12 of them, so they are unrepriceable by anyone), leaving **7 real defects (1.75%)**.
  **6 of those 7 are the http_500 the new retry now covers**, leaving **~1 in 400 = 0.25%**.

**The repriced population is 0.25%. That is tiny. Proceeded.** Live proof, id-matched not row-counted
(BL-550's trap: an Apify probe once returned a video that was never requested):

```
stored=1500000  lamatok=1500000  delta=0     id_match=true   <- same-day check, IDENTICAL
stored= 595900  lamatok= 596000  delta=+100  id_match=true   <- stored 07-16, probed 07-17: view GROWTH
stored= 570000  lamatok= 570200  delta=+200  id_match=true   <- same
HTTP 404  stored=2500000  -> NULL, fails open
HTTP 404  stored=1100000  -> NULL, fails open
```

The clip checked most recently (07-17 07:51) reads **delta 0**. The +100 and +200 are ~21 hours of
real view growth on clips last checked 07-16, not repricing.

### The retry: the only genuinely fixable cluster anyone has found

LamaTok now **retries a transient 5xx once** (600ms backoff). BL-550 measured **6 http_500 in 400
TikTok clips (1.5%)** with **no retry at all** — a single server blip sent the clip straight to a
billed Apify actor.

**The retry is free**, and the file already said so at its own non-200 branch: *"LamaTok bills
successful responses; server errors are free."*

* A **404 is NOT retried**: it is a definitive answer, and asking twice cannot undelete a post.
* A **429 is NOT retried**: rate limiting needs backoff above the level of one clip, and hammering it
  makes things worse.
* **Fails open, BL-543's rule:** if both attempts fail the result is `stats: null`, **never 0**.
  Tracking keeps the last known value on null, so no clipper's views are zeroed and no earnings freeze.
