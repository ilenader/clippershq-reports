# BL-577 — HikerAPI Cooldown Audit (READ ONLY)

**Question:** BL-571 hypothesised that when HikerAPI enters a cooldown, `tryHikerForInstagram`
short-circuits and returns `verdict="transient"` without asking the provider, so during a cooldown
nothing gets a `gone` verdict, everything looks like a miss, and everything escalates to a billed
Apify actor. If true, our own throttle would be converting directly into Apify spend.

**Verdict up front:** The *mechanism* is real in code. The *money* is not. A cooldown has **never
fired in production history** (zero `429`, zero `402`, zero outage-guard trips, zero cooldown-skip
rows ever recorded). The cooldown-driven fraction of the surviving Apify spend is **0%**. BL-571's
hypothesis is REFUTED by the data. The real residual IG Apify spend is the single-fetch individual
path escalating on `null_views_on_200`, not cooldowns.

## Probes run (all disclosed, no Apify actor run)
1. Supabase reads via `scripts/run-select.js` (our own DB, read-only, **$0**). Timestamps cast `::text`.
2. One live `GET https://api.hikerapi.com/sys/balance` (prepaid HikerAPI quota, one status request,
   effectively **$0**; 399,333 requests remain).
3. **No Apify actor run. No Apify billing-API pull** (not needed: zero cooldown rows means no invoice
   line can be cooldown-attributed regardless of the absolute bill).

Code read from `main` @ `1d69a62` in an isolated worktree; `apify.ts` and `hikerapi.ts` were READ,

## PART 1 — THE COOLDOWN, EXACTLY (file:line)

Cooldown state is a single module-level in-memory variable, **global** across all IG calls in a
process, and **process-local** (a fresh Railway deploy starts with no cooldown):
`let cooldownState` at `src/lib/scraper-providers/hikerapi.ts:787`; setter `setHikerCooldown` at
`:789` (longest-cooldown-wins at `:793`); reader `getActiveHikerCooldown` at `:798`, which
self-clears on read once `remaining <= 0` (`:801`). It is NOT per-endpoint and NOT durable
(comment `:776`).

**The short-circuit under test** is `hikerapi.ts:613-621`, inside `tryHikerForInstagram`, BEFORE
any network call:
```
const cooldown = getActiveHikerCooldown();      // :613
if (cooldown.active) {                           // :614
  ...
  void recordHikerUsage("fallback", t0, { cooldown: cooldown.label });   // :619
  return { useResult:false, stats:null, reason:`cooldown:${label}`, ..., verdict:"transient" }; // :620
}
```
So during a cooldown the provider is never asked, and the caller receives `verdict="transient"`.
BL-571 is correct that this is a `transient`, not a `gone`.

**What triggers a cooldown, and for how long:**
* `402` balance exhausted, `setHikerCooldown("balance", BALANCE_COOLDOWN_MS)` at `:651`;
  `BALANCE_COOLDOWN_MS = 10 min` (`:781`). Provider-side (billing).
* `429` rate-limited, `setHikerCooldown("rate", RATE_COOLDOWN_MS)` at `:659`;
  `RATE_COOLDOWN_MS = 60 s` (`:782`). Provider-side (their per-second rate limit).
* Outage guard, `checkAndMaybeFireOutage()` at `:992`, sets `setHikerCooldown("outage",
  OUTAGE_COOLDOWN_MS)` (`:996`); `OUTAGE_COOLDOWN_MS = 10 min` (`:783`). Trips only when the rolling
  1 h window (`ROLLING_WINDOW_MS`, `:817`) has `>= 30` calls (`OUTAGE_MIN_CALLS`, `:818`) AND
  `>= 50%` of them are `outcome="error"` (`OUTAGE_FAIL_RATIO = 0.5`, `:819`). This one is **OUR**
  throttle, reacting to `429`/`5xx`/`network`/`null_views_on_200` errors.

**What resets it:** expiry on read (`:801`), a fresh deploy (in-memory), or the test-only
`_resetHikerCooldown` (`:809`).

**Our throttle or their rate limit?** Both live here, but they are different triggers. Crucially the
cooldown-skip itself is recorded as `outcome="fallback"` (`:619`), NOT `"error"`, so a cooldown does
**not** feed the outage guard: cooldowns cannot self-amplify.

**The concurrency oddity, resolved.** `HIKER_GLOBAL_CONCURRENCY = 10` lives at
`src/lib/apify.ts:1297` (the audit brief cited `:1229`; the constant has moved on `main`, same value).
It is a **semaphore over simultaneous in-flight requests** (`sem.acquire()` at `apify.ts:1381`), NOT
a requests-per-second cap. HikerAPI's `/sys/balance` `rate:15` is a per-second limit. These are
different units, so "we self-limit to two thirds of the allowed rate" is not what the code does. At
the observed per-call latency (a few seconds) and 10 in-flight, throughput is roughly 5 requests/s,
comfortably under 15/s. The concurrency cap therefore makes `429`s **less** likely, not more. Raising
it toward 15 in-flight would push throughput up but still stay under the per-second ceiling; it is
not a cooldown driver either way.

## PART 3 — WHY IS IT IN COOLDOWN AT ALL?

It is not. Live `/sys/balance` at audit time returned **HTTP 200**,
`{"requests":399333,"rate":15,"currency":"USD","amount":275.55}`. Balance $275.55 (barely moved from
BL-567's $276.30), 399,333 requests remaining (BL-550 measured the quota at ~400,000 and barely
touched; a full 1,268-clip sweep spent ~2,153 requests, about half a percent). The provider is
healthy, not degraded, not rate-limiting us, not erroring beyond the trivial 52-in-5-weeks. There is
nothing to cool down from.

## PART 4 — SHOULD A COOLDOWN ESCALATE TO A PAID ACTOR? (argued both ways, then quantified)

**For escalating:** a clipper's views must keep updating; BL-543's rule is that an unresolvable clip
returns NULL (keep last-known), never 0, so nothing freezes. If HikerAPI genuinely could not answer,
asking Apify protects real earnings.

**Against escalating:** BL-550 probed 37 clips HikerAPI could not resolve and Apify resolved **0 of
37**, while agreeing 10 of 10 where both answered. On Instagram, Apify rescues almost nothing that
HikerAPI misses. A cooldown is temporary and the next tick is ~10 minutes away, so escalating during
one pays a billed actor to wait for the cooldown to end.

**Quantified for cooldowns specifically:** the number of cooldown-escalated clips is **0** (no
cooldown has ever fired), so the number Apify actually rescued during a cooldown is **0 of 0**, and
the number that would have resolved on the next tick is also moot. The escalation-during-cooldown
question is real in principle and would resolve *against* escalating, but it decides **$0** of
today's bill. Do not build a fix whose entire target set is empty.

## PART 5 — THE VERDICT AND THE NEXT STEP

**Is the cooldown the mechanism behind the surviving ~$88/month? No.** Fraction: 0%. BL-571's
hypothesis is mechanically plausible and provably inert: the short-circuit exists, but the condition
that arms it has never occurred. Do not "stop escalating during a cooldown" (the branch never runs)
and do not "raise concurrency to 15 to stop cooldowns" (there are no cooldowns to stop, and the
concurrency cap is an in-flight bound, not a per-second rate). Both would be fixes to a non-problem.

**What the money actually is.** The live residual IG Apify spend is the **single-fetch individual
path**, not the batch and not cooldowns. Billed IG actor runs, 7 days: `apify/instagram-api-scraper`
278 runs, only 57 usable (~20%); `apify/instagram-scraper` 18 runs, 0 usable. Each is a single-URL
call (`clipCount=1`). These are driven by `tryHikerForInstagram` returning `verdict="transient"` on
`null_views_on_200` (231 in 7 d: HikerAPI answers 200 but views are <=0/null, so the never-zero gate
at `hikerapi.ts:729` sends it to Apify) plus carousel/classification misses (10). The single-fetch
overlay at `apify.ts:2139-2164` only skips Apify on `verdict="gone"` (a 404); a `transient` correctly
falls through to `fetchInstagramStats`, and Apify then also fails on these clips (`success=false`),
which is exactly the 05:46 worthless burst.

**Caveat on the $88 figure, stated honestly.** Our ledger's `actor:*` estimate is ~$0.80 of IG actor
cost over 7 days (~$3.4/month), far below the ~$88/month BL-571 read off Apify's billing API. The
ledger is a known undercount and I did not reconcile it to the invoice, because that is not needed to
answer this question: zero cooldown rows means zero of whatever the true bill is can be
cooldown-attributed. I am not claiming to have located the whole $88; I am killing the claim that
cooldowns are any of it.

**Exact ordered next step (build-prompt ready):**
1. Do NOT touch the cooldown path or `HIKER_GLOBAL_CONCURRENCY`. This audit proved both are inert.
2. Target the single-fetch worthless-call loop instead. Extend **BL-576's persisted gone/strike
   counter** so it increments not only on a definitive `gone` (404) but also on the outcome where
   HikerAPI returns `verdict="transient"` via `null_views_on_200` AND the subsequent single Apify
   call returns `success=false` (billed and worthless). After K consecutive such strikes, retire the
   clip so tracking skips the single Apify call entirely, the same way a `gone` clip is skipped.
3. Interaction with BL-576: BL-576's counter is the correct durable fix, but its value is **not** the
   mid-cooldown case (which never occurs). Its value is retiring clips that cost a worthless single
   Apify call on every tick (the 221 billed-and-worthless IG single calls per 7 days). If BL-576 ships
   counting only 404-gone, it will miss this population; widening the strike trigger to include the
   `transient + Apify-failed` outcome is what actually removes the surviving IG single-fetch spend.

**If asked "so where is the leak": ** the single-fetch individual path retrying clips HikerAPI can
answer but not usefully (views<=0) and Apify then also fails, roughly 40 IG single Apify calls/day at
~20% usable. That is a strike-and-retire problem (BL-576), not a cooldown problem (BL-577).
