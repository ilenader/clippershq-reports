## PART 1 — the effective apify.ts:616 skip (persisted, cooldown-immune)

Shipped in `apify.ts`. A helper `isClipPersistedGone(clipUrl, threshold)` reads BL-576's persisted
`consecutiveGone` from the DB and, in the IG single-fetch branch, returns NULL BEFORE calling the
billed `fetchInstagramStats` when the clip is known gone at least `threshold` ticks running.

Why this is not BL-571's no-op: BL-571 proved a LIVE gone-recheck here is worthless because
`tryHikerForInstagram` short-circuits to `verdict="transient"` during a cooldown, so nothing earns a
"gone" verdict and everything escalates. A PERSISTED read fires NO network call, so no cooldown can
suppress it. The clip's own prior 404s (already recorded by the counter) drive the skip.

Safety properties, all held:
* FAIL OPEN: any lookup error returns false → the caller proceeds to Apify exactly as before. An
  unresolvable clip yields NULL, never 0 (BL-543), so last-known views are kept.
* Exact-URL match (`clip.clipUrl`, which tracking passes verbatim) so no normalization is risked.
* $0 ledger truth: the skip returns `provider="persisted-gone-skip"`, added to
  `ZERO_COST_PROVIDERS` in `apify-ledger.ts`, so it can never invent the $0.10 unknown-provider cost
  (the BL-551 inflation trap).
* Threshold K=5 (BL-576's recommended retirement bar; the same bar gates this cheaper skip).

**Does it fire tonight? No, and I say so plainly.** The counter went live ~1 day ago (BL-578); the
