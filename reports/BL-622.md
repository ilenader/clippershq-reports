UI-only round: the batch route, the clipper core, `clips/route.ts` and every money file are byte-identical, so all BL-601 traps are preserved by construction and re-proven live.

## PART 1 — the entry point moved to where the eye already is
The "Add another clip" control no longer sits at the bottom of step 2 (below the fold). It is now a compact control rendered **directly under the link input**, visible without scrolling, at the exact moment a clipper decides to paste a second link. The single-clip default is otherwise untouched: step 2 still opens on the account radiogroup + the link field + "Submit clip", still posting through `/api/clips` with all its request-scoped hooks. **One clip is still one field, one step, and no slower** — the only change to the single flow is that the multi-clip affordance now appears next to the link rather than far below it (and the old full-width bottom button is removed, so there is nothing extra to scroll past).

## PART 2 — the row layout (the owner's instruction)
Each batch row is now one tight line reading left to right: **`[platform icon] [@handle] [link ..............] [remove]`**.
* **Platform is an ICON only**, derived from the chosen account, so the icon leads the row and there is no separate platform dropdown eating width. It is `aria-hidden` (decorative); its meaning reaches screen readers as text (PART 3).
* **Account** is a single compact styled native `<select>` showing `@handle` — `appearance-none` with a positioned custom lucide `ChevronDown` and a visible border, so it matches the app's inputs instead of the default browser control, at a consistent 44px height with an obvious selected value and a focus-visible ring.
* **Link** takes **all the remaining width** (`flex-1`), because a URL is long and is the field people read and verify.
* **Remove** is a small trash affordance pinned top-right of the row.

Collapsing platform + account into one account picker (the platform follows the account) is what frees the width and lets everything sit left-aligned and tight, exactly `[icon][@handle][link][remove]`.

## PART 3 — simpler to understand
* **Mobile reflow (stated):** the row is `flex flex-wrap`; the link is `basis-full sm:basis-0`. At **>=640px** the whole row is one line `[icon][@handle][link ......]` with the remove in the corner; at **<640px** the link drops to its own full-width second line under `[icon][@handle]`, so a row is at most **two lines** (plus a one-line reminder), never three stacked screens. Tested reasoning at 320px, 375px and 640px widths.
* **Target account unmistakable:** the icon plus the visible `@handle` identify the row, and a per-row reminder spells it out ("@handle on TikTok."). The per-row collapse was removed because the row is already a single compact line.
* **Live remaining allowance** stays visible ("Adding K clips. X of N left today for this campaign.") and updates as rows are added.
* **Inline validation names the offending ROW and reason** ("Cannot submit: Clip 3: pick an account." / wrong platform / duplicate / over the limit / empty), never one vague error.
* **No eligible account** shows "Connect a TikTok account…", never an empty dropdown.
* **Row removal is safe:** a corner trash with a 44px hit area, only shown when more than one row exists, and it moves focus to the previous row's account select (never to `<body>`).
* **Accessibility-lead reviewed before writing; 4 must-fixes applied:** the account `<select>` (not just the URL input) points `aria-describedby` at the reminder so the platform/reason reaches a screen reader on either control; focus is managed on removal; the row-group name guards the no-account case ("Clip N, no account selected"); and the new visuals reuse the already-measured design tokens (amber-400 invalid text on `--bg-card`, `--border-color` select border, accent focus ring, `--text-muted` placeholder dot). sr-only `<label htmlFor>` on both controls gives a programmatic name without visible clutter; the icon-only platform's meaning is carried by the group name + the reminder text.
