## PART 2 — full submit simulation (traced, nothing written)

• URL PARSING / platform: `detectPlatform` + `socialPlatformForUrl` are host-based,
  so `/photo/` classifies TikTok and `/p/` classifies Instagram correctly; a
  `/photo/` is never mis-read as a `/video/`. The `?img_index=` param is a trailing
  query the `/p/(shortcode)` and `/photo/(\d+)` regexes stop before, so it does not
  break parsing (none of the six carried it; the three shortcodes parsed clean).
• TikTok slideshows are BLOCKED AT SUBMIT: `areCarouselsDisabled()` defaults ON
  (`CAROUSELS_DISABLED` unset, `carousel-config.ts:26`) and `isTiktokPhotoUrl`
  matches `/photo/`, so `clips/route.ts` rejects with TIKTOK_PHOTO_BLOCK_MESSAGE. A
  clipper CANNOT submit any of the three TikTok slideshows today. So the
  catastrophic-0 and wrong-retire paths below are all MOOT for TikTok while the
  flag stays ON.
• IG carousels are NOT blocked (a `/p/` link is not URL-classifiable), so a clipper
  CAN submit the three Instagram posts.
• PROVIDER ROUTING: an IG submit routes to HikerAPI `by/code` (the reader that
  works, 200 above), not a 500 path. A TikTok submit would route `by/url` (500) ->
  Apify (the gap) — but it is blocked before that.
• FIRST STAT WRITE: `clips/route.ts:1137` writes `views: fetchedStats?.views ?? 0`.
  For an image-only carousel (`views=null`) or the mixed-carousel-0 case, the FIRST
  ClipStat is written as **0, not null** — the BL-543 landmine at the submit
  boundary. For an image-only post 0 earnings is harmless (no views exist); for a
  mixed carousel whose video views are merely unreadable it UNDER-COUNTS a real clip
  to 0.
• TRACKING JOB: a submitted IG carousel polls HikerAPI every tick and resolves 200
  (null or a real sum), so it does not escalate forever. A TikTok slideshow (if the
  flag were flipped) would `by/url` 500 every tick and fall to Apify each time — but
  it never marks gone (below).

## PART 3 — every slideshow/carousel failure mode hunted

• THE DANGEROUS ONE — a live slideshow wrongly auto-retired (freezing a clipper):
  DOES NOT HAPPEN. Verified in code. (1) A `by/url` 500 yields verdict "carousel"
  (`lamatok.ts:261`, checked before the 404 gate), never "gone"; the gone-counter
  increments ONLY on a definitive provider-404 in `goneClipIds` and "NEVER
  increments on a transient 5xx" (`gone-counter.ts:26-31`), so a live slideshow that
  500s every tick neither increments nor resets — it can never reach K=3. (2)
  `retire-dead-clips.ts:74` filters candidates to `clipUrl contains "instagram"` —
  TikTok is EXCLUDED entirely — and every retirement requires a MANDATORY fresh
  HikerAPI HTTP-404 re-probe THIS run (`:101-111`); a live post re-probes 200 and is
  left untouched. So neither a TikTok nor an IG live slideshow can be condemned.
• DELETED vs live-500 gone-handling: correct. A deleted post returns 404 +
  MediaNotFound -> "gone"; a live slideshow returns 500 AppException -> "carousel"
  (never gone). The `MediaNotFound` field is the clean separator (`lamatok.ts:257-265`).
  Confirmed live: all three slideshows are 500/AppException, none 404.
• IMAGE-ONLY carousel: returns null (correct), NOT 0, in the classifier
  (`hikerapi.ts:459-471`). BUT the submit-time first-stat coalesces null to 0
  (`clips/route.ts:1137`) — see fix 3.
• MIXED video/image carousel: sums the video children's play_count; when NO video
  child exposes a count it returns 0 (should be null) — the live DbAuwN2F0MO case.
  This is the one genuine mis-store found (`hikerapi.ts:483`).
• DISPLAY-ROUNDING: by/id is display-rounded (proven 25,700 vs 25,835 etc.). It is
  stable tick-to-tick for a given true count (it only steps when the true count
  crosses a rounding boundary), so it would NOT jitter earnings — but if by/id is
  ever promoted it stores the rounded value and earnings ride ~0.3 to 0.9% under the
  true count. This is a PRICING decision to settle before promotion (BL-544/600), not
  a bug to silently flip.
• SHORT-LINK (vm./vt.) slideshows: `extractTiktokPhotoId` returns null (no id until
  the redirect resolves), so the by/id path is deferred for them (`lamatok.ts:94,114`).
  Irrelevant while TikTok slideshows are blocked; a blocker to resolve before enabling.
• MUSIC-ONLY slideshows: all three TikTok posts resolved via by/id regardless of
  audio, so view resolution does not depend on music. These three do not ISOLATE a
  pure music-only variant, so BL-600's music-only gap remains UNVERIFIED; to verify,
  probe a confirmed music-only `/photo/` id via by/id.
