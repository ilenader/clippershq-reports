## PART 1 — persist what each attempt saw (shipped)
BL-552 and BL-564 both had to live-probe because the bio is not stored at attempt time. Fixed with the
"one column" BL-564 recommended: additive, nullable columns on `clip_accounts` (applied via
`ADD COLUMN IF NOT EXISTS`, no `prisma migrate`):
`lastVerifyBio` (text), `lastVerifySource` (which tier answered), `lastVerifyFoundInBio` (bool),
`lastVerifyBioBytes` (int), `lastVerifyAt`.
Written best-effort in the verify POST from data the checkers already fetch; it NEVER gates verification
(wrapped, swallows errors). For Instagram it stores the clean structured Apify `biography`; for TikTok and
YouTube (HTML scrape, no clean bio) it stores `bioBytes` so a ~1462-byte bot wall (we could not see the
bio, our defect) is distinguishable from real content (they likely did not add the code). Now one SELECT
answers "code was there, we missed it" vs "no code in the bio":
```sql
SELECT platform, "lastVerifySource", "lastVerifyFoundInBio", "lastVerifyBioBytes",
       (length(coalesce("lastVerifyBio",''))>0) AS saw_bio, "lastVerifyAt"::text
FROM clip_accounts WHERE status='PENDING' AND "deletedByUser"=false AND "lastVerifyAt" IS NOT NULL;
```
**Storage + retention + redaction:** a real person's bio is stored only as the LATEST snapshot (overwritten
each attempt, capped at 2000 chars, never a history), never rendered raw in any owner UI, and is redacted
in this report. Recommended follow-up: a 90-day null-out of `lastVerifyBio` on APPROVED accounts (a small
cron), not built here.

## PART 2 — validate the profile link AT ENTRY (shipped; unblocks the permanently-stuck)
Root cause found: pasting a full URL into the username field makes the auto-build wrap it into a doubled
link, and `validateAccountLink`'s `@[\w.]+` regex matched `@https`, so `tiktok.com/@https://www.tiktok.com/@name`
passed. New pure, client-safe `src/lib/verify-handle.ts` (`normalizeHandleInput`) extracts the real handle
from any pasted URL or doubled link (taking the LAST `@token`), and REJECTS a handle with a space with a
plain, non-accusatory message plus a per-platform example. Wired server-side in `POST /api/accounts` (the
authoritative gate) and client-side in `AddAccountFlow.tsx` at submit (reusing the existing accessible
`role="alert"` + `aria-describedby` + focus-first-invalid pattern; reviewed and cleared by the
accessibility-lead, no must-fix).

**The malformed accounts today (redacted, run live against `clip_accounts`):** 8 stored malformed links
(BL-564 saw 12; churn since). By type and remedy:
- **4 doubled URLs** (3 TikTok, 1 Instagram) → **auto-correctable**: the normalizer extracts the real handle,
  so on re-entry (or a one-time backfill) they verify. Proven: each normalized to a valid handle.
- **4 spaces in the handle** (2 YouTube, 2 TikTok, e.g. "@Static Harbor") → **must re-enter**: a space
  handle cannot be guessed, so the normalizer rejects it at entry with a clear per-platform message.
The stored 8 are historical; the fix stops NEW ones for everyone and makes the doubled ones fixable on
re-add. No handle or address appears in this report.

## PART 3 — the real defect, and why the scraper flow is designed not shipped
BL-564 proved abhijot4993's bio literally contains their code AA5Z and the 4-tier HTML cascade missed it 14
times, while LamaTok `author.signature` reads it exactly. The reason is BL-552's: Instagram reads a
STRUCTURED `biography` field (escalates zero times), while TikTok and YouTube SCRAPE client-rendered HTML
and cannot see the bio. The structured TikTok source (LamaTok `author.signature`) exists, is already paid,
but only on a media (video) response; LamaTok's profile endpoint is `NotImplementedYet` (BL-564 probed
501/404). So using it means verifying against a VIDEO URL, not a profile URL, which is a flow change.

**Designed within the ONE-CALL-PER-PROFILE ceiling (not shipped this round):**
1. When TikTok profile scrape misses, ask the clipper to paste ONE of their video links (natural: clippers
   post videos). Read `author.signature` from `/v1/media/by/url` and match the code.
2. **Cache the verdict per profile** (key: normalized handle) for 24h, so BL-564's 21-retry clipper makes
   at most ONE LamaTok call, not 21. A retry inside the window reads the cache; only a genuinely new
   profile spends a call. This is what respects the ceiling given the key already serves ~2,000 calls/day
   and BL-540 hit HTTP 429 on TikHub in under 10 requests.
This is deferred exactly per the brief ("ship PARTS 1 and 2 alone if the ceiling cannot be met") and per
BL-564's ranking: measure first (PART 1), unblock the malformed (PART 2), then decide the flow change with
the split measured rather than inferred.

**YouTube confirmed:** `verify-cascade.ts:337` reads `process.env.YOUTUBE_API_KEY`; when PRESENT (owner
confirmed it is set in Railway) it calls Google's free Data API and reads `snippet.description`; when absent
it returns transient without calling. The code correctly handles the key being present, so YouTube tier-2 is
live in prod. No code change needed; the only open YouTube question (whether clippers put the code in the
"About" box `snippet.description` vs the short tagline) is now answerable from PART 1's persisted data.
