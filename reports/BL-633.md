# CLIPS_PER_TICK: which service reads it, and what production really uses (BL-633)

Audit only, read only. Measured 2026-07-23 from live prod data and the code. No load test. Where a number could not be measured it is marked NOT MEASURABLE FROM HERE.

## One line answer
**The tracking tick runs on a SEPARATE Railway cron service (start command `tsx scripts/run-tracking-cron.ts`), which reads CLIPS_PER_TICK from its OWN environment. That service does not have the variable, so production is really running the hardcoded fallback of 30. The `CLIPS_PER_TICK=60` the owner set on the WEB service is ignored by tracking. To change the tick, set it on the cron service, not web.**

## PART 1 — where CLIPS_PER_TICK is read, and by which process
**Every read (file:line):**
• `src/lib/tracking.ts:160` defines `clipsPerTick(envValue = process.env.CLIPS_PER_TICK)`. This is the ONLY functional read of the env var.
• `src/lib/tracking.ts:172` is the **hardcoded fallback: `if (!Number.isFinite(n) || n < 5) return 30;`** (clamped to a 5 to 500 band, ceiling 500). So when the variable is absent, **the effective cap is 30**.
• `src/lib/tracking.ts:3428` calls `clipsPerTick()` (the value the cron tick actually uses). `:3719` applies that cap to the due-clip collection. `:3482` and `:3491` re-read the env only to print the `[TRACKING-CONFIG]` diagnostic line.

All of these live inside `runDueTrackingJobs`. So the question "which process reads CLIPS_PER_TICK" is really "which process calls `runDueTrackingJobs`". There are three callers:

1. **`scripts/run-tracking-cron.ts:132` — the Railway CRON SERVICE. THIS IS THE ACTIVE TRACKING PATH.** The file header (lines 4 to 6) states it plainly: "Railway's Cron Schedule setting runs a service on a schedule by invoking its startCommand. This script is the startCommand for the cron service: `tsx scripts/run-tracking-cron.ts`." It loads its own environment first (`import "dotenv/config"`, line 40), then calls `runDueTrackingJobs({ source: "cron", isBatchTick: true })` **in process** (line 132). It does NOT make an HTTP call to the web service. Confirmed as a distinct start command in `package.json:13` (`"cron:tracking": "tsx scripts/run-tracking-cron.ts"`), separate from the web service's `npm start` (`railway.json:7`). Therefore this service reads CLIPS_PER_TICK from the CRON SERVICE'S environment.

2. **`src/app/api/cron/tracking/route.ts:315` — the WEB HTTP route. NOT the active trigger.** `src/lib/railway-cron-scheduler.ts:63` registers tracking as `explicitOnly: true`, and its header (lines 18 to 21) says "'*' enables every job EXCEPT tracking ... it is ALREADY firing via an external trigger today, so it is NEVER [auto enabled]". So the web in process scheduler does not fire the tracking route; the "external trigger" is the cron service in caller 1. If this route were the live path it would read the WEB env (60), and production would show up to 60 clips per tick. It does not (see PART 2), which independently proves this route is not what is running the ticks.

3. **`src/app/api/admin/track-all/route.ts` — the WEB manual "check clips now" button (source: "manual").** This runs on the web service and DOES read the web env (60). It only fires on an owner action, not on a schedule.

**Architecture verdict:** the tracking tick is a **separate short lived worker spawned per tick by Railway's native cron on the cron service**, not an HTTP pinged route on the web service. The cron service's environment is the one that governs CLIPS_PER_TICK. `src/lib/marketplace-timers.ts:3` corroborates ("same lock, same Railway service").

## PART 2 — what production is actually doing (measured, not inferred)
Distinct clips with a stat written per 10 minute tick window, from `clip_stats.checkedAt` versus DB `now()`:
• **Last 24h, busy ticks: exactly 30 clips, never more.** (Sample: 07-22 16:40, 16:20, 15:30, 18:40, 15:20, 13:20, 22:20 all = 30.)
• **14 day cron only (isManual false) daily maximum per tick: 30 to 37** (the 31 to 37 values are 10 minute bucketing noise where one ~74s tick straddles a boundary), **daily average ~21 to 25 clips per tick**, ~144 ticks per day on healthy days.
• Production is therefore really running **30**, i.e. the hardcoded fallback, NOT 60.

Real throughput and headroom:
• **Ticks per hour: 6** (the cron fires every 10 minutes; ~144 ticks per day when healthy; fewer on gap days such as 07-18, the compute cap outage).
• **Polls per day: ~2,900 to 3,200** delivered (average ~22 clips per tick x ~144 ticks).
• Against **4,028 tracked clips**, the hard ceiling is 30 x 6 x 24 = **4,320 polls per day**, so production sits at roughly **70 to 75 percent of the cap, about 1.4x headroom**. This empirically confirms BL-628's estimate: production genuinely runs at 30, so BL-628's ceiling analysis stands unchanged.

## PART 3 — was 60 ever live, and did anything change when web got 60?
**No. 60 has never been effective on the tracking path.** The cron only per tick count has been steadily capped at ~30 across the entire 14 day window with **no step change** to anything near 60 at any point (daily max 30 to 37 throughout). A 30 day cron max of 43 appears once as an isolated outlier (bucketing across a boundary or a post gap catch up), and manual runs on the WEB service reached ~44 in 30 days (only 3 manual ticks) which is consistent with the web env of 60 being effective for the rare MANUAL "check now" but never for the scheduled cron. Nothing stepped to 60 when web got 60, because the scheduled tracking tick does not read the web environment.

**So the 30 versus 60 debate was about a variable the tracking cron service never reads.** Setting 60 on web changed the manual button's ceiling and nothing about the automatic tracking throughput.

## PART 4 — the verdict and the exact instruction
• **Which service must hold CLIPS_PER_TICK for the tracking tick to see it:** the **tracking cron service**, the Railway service whose start / cron command is **`tsx scripts/run-tracking-cron.ts`** (the `npm run cron:tracking` script). NOT the web service.
• **What production is really using right now:** **30**, the hardcoded fallback at **`src/lib/tracking.ts:172`**. The 60 on the web service is inert for tracking.
• **Exact action the owner should take:**
  1. To raise the automatic tracking tick, set the variable **on the tracking cron service**: name `CLIPS_PER_TICK`, value `60` (or, per BL-629, step 30 to 60 to 90 to 120 and confirm Railway compute headroom first, since a compute cap took the site down on 2026-07-18). 
  2. The `CLIPS_PER_TICK=60` currently on the WEB service does nothing for tracking. It can be left (it only affects the manual "check now" button and the inactive HTTP route) or removed to avoid future confusion. Do not treat the web value as production's tracking cap.
  3. Do nothing else. No code change; the fallback and the read are correct, they were simply reading a different service's environment than the one the owner edited.
• **The file:line that truly governs throughput today:** `src/lib/tracking.ts:172` (`return 30`).

**Query to confirm any future change took effect** (run after editing the cron service and letting a few ticks pass; busy ticks should rise above 30 toward the new value):
```sql
WITH t AS (
  SELECT DISTINCT "clipId",
    date_trunc('minute', "checkedAt")
      - (EXTRACT(minute FROM "checkedAt")::int % 10) * interval '1 minute' AS tick
  FROM clip_stats
  WHERE "checkedAt" > now() - interval '3 hours'
    AND ("isManual" IS NULL OR "isManual" = false)
)
SELECT to_char(tick,'MM-DD HH24:MI') AS tick, COUNT(*) AS clips
FROM t GROUP BY tick ORDER BY clips DESC LIMIT 8;
```
Also, on the CRON service's Railway logs, the `[TRACKING-CONFIG] clipsPerTick=<n> (source=env|default)` line (`src/lib/tracking.ts:3491`) prints the effective value and whether it came from the env or the default, so `source=env` with the new number is the definitive confirmation that the cron service now reads it.

## What could not be measured from here
The actual Railway environment of each service (web versus cron) is not readable from this repo; the conclusion that the cron service lacks CLIPS_PER_TICK rests on the owner's report plus the decisive empirical fact that scheduled ticks cap at exactly 30 (the fallback) while manual web runs reach into the 40s. The exact attribution of the single 43 clip cron outlier could not be pinned down. No load test was run.
