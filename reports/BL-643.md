# BL-643 — can the 269 KB /earnings page be made light without breaking anything?

**2026-07-23 · AUDIT ONLY. READ ONLY. Nothing was built.** No code, config or data change. This designs the fix and proves it safe; a later round implements it.
**Base:** origin/main `9f5de583` · **Branch:** checkpoint/BL-643 · **Worktree:** `C:/b643` (short path, real `npm ci`, no `node_modules` junction)

## FIRST LINE, because of the two prior breakages

**YES, the recommended fix changes loading behaviour, but only in one narrow window, and it is SMALLER than what is already on this page today.** The chart area **already** renders a fixed-height **pulsing skeleton** in production right now (`EarningsChart.tsx:54-72`, `animate-pulse`, `data-earnings-chart-skeleton`) every time the page loads, while the daily-earnings data is fetched. The fix leaves that existing skeleton **completely untouched and static**, and adds only a **plain empty 220px box with no animation, no text and no shimmer** for the window while the chart code downloads. In the common case the clipper sees **literally no difference**, because the existing skeleton already covers that window. **I am not proposing a skeleton, a Suspense fallback, or any restructuring of the page.** If the owner wants zero change of any kind, the honest answer is in PART 5: decline, and 269 KB stays.

**Nothing about a clipper's earnings numbers, balances or withdrawal availability changes, at all, under any failure mode.** That is proven in PART 3, item 6.

## PART 1 — the real weight

Measured by resolving each route's `page_client-reference-manifest.js` to its emitted chunks and gzipping the actual files. **Gzip is what the phone pays.**

| Page | Total raw | **Total gzip** | recharts chunk raw | **recharts chunk gzip** | **Gzip without the chart** |
|---|---|---|---|---|---|
| **`/earnings`** | 1,067 KB | **269 KB** | 380.9 KB | **110.8 KB** | **158 KB** |
| `/referrals` | 1,072 KB | 270 KB | 380.9 KB | 110.8 KB | 160 KB |
| `/clips` | 752 KB | 176 KB | none | 0 | 176 KB |
| `/progress` | 691 KB | 159 KB | none | 0 | 159 KB |

**The chart is 110.8 KB gzip, which is 41.2% of the entire `/earnings` download.** Deferring it takes the page from **269 KB to 158 KB gzip**, a saving of **111 KB**. This exactly reproduces BL-639 on an independent clean build.

**Sanity check that recharts is the whole anomaly:** with the chart removed, `/earnings` lands at **158 KB**, essentially identical to `/progress` at **159 KB**. There is no second heavy thing hiding in the page. The rest is the shared framework chunk (283.8 KB raw / 74 KB gzip, also on `/clips`), the shared 226 KB raw CSS bundle, and six small chunks between 8 and 51 KB. **Nothing else on `/earnings` is worth deferring.**

**The same 380.9 KB chunk is byte-identical on `/referrals`,** which is also clipper-facing (`ReferralsRedesign.tsx:16`), and on four non-clipper pages: `admin/analytics`, `admin/command-center`, `admin/referrals` and `client/page.tsx`.

## PART 2 — what the chart is, and where it sits

**What it shows:** a single-series area chart of **daily earnings** over a selectable timeframe, with a gradient fill, X and Y axes, a grid and a hover tooltip (`area-gradient-chart.tsx:60-126`). Data is bucketed "by the day each dollar was OBSERVED, not the day the clip was submitted" (`EarningsChart.tsx:91`).

**Where it sits: ninth block down, far below the fold.** Every clipper sees `EarningsPremium` (BL-477 rolled it out to all clippers, `earnings/page.tsx:172-178`), and its render order is:

| Order | Block | Line |
|---|---|---|
| 1 | Header + campaign filter dropdown | `:76-91` |
| 2 | **Hero: available for payout** (the 40px/6xl money number) | `:95-107` |
| 3 | **Request payout** button | `:108-119` |
| 4 | **Stat trio** (Earned / Approved / Paid out) | `:121-135` |
| 5 | Marketplace breakdown (hidden when zero) | `:136-153` |
| 6 | Pending / payout-queue chips | `:155-169` |
| 7 | **Earnings by campaign** list, with photos, unbounded | `:171-212` |
| 8 | "Earnings over time" heading + Timeframe + Filters | `:214-221` |
| 9 | **The chart** | **`:223`** |

On a 375px phone the clipper must scroll past their balance, the payout button, a three-stat grid, chips and the **entire per-campaign list** before the chart is on screen. **A clipper who opens Earnings to check what they are owed never sees the chart at all.**

**Therefore deferring it is invisible to the common case and near-zero risk.** This is the single most important fact in this report: the 41% of the page that costs the most is the part a phone user reaches last, if ever.

**Height is fixed at 220px in every state.** `EarningsPremium.tsx:223` passes **no** `height` prop, so `EarningsChart`'s default `height = 220` applies (`EarningsChart.tsx:44`), and all three states pin to it: loading skeleton (`:63`), chart body (`area-gradient-chart.tsx:68`) and the "No data yet" state (`:120`).

## PART 3 — every way this could break

`next/dynamic` is nearly untried here: the **whole `src/` tree contains exactly one usage**, `VoiceRoom` at `app-layout.tsx:36-40`, added for precisely this reason ("pulls in @jitsi/react-sdk (large). Lazy-load so the Jitsi iframe only fetches when someone actually opens a call"). So the pattern is established for this exact purpose, but only once. Every failure mode, walked honestly:

### 1. Does it change WHEN the chart's data is fetched or displayed? NO.

`EarningsChart.tsx` and `area-gradient-chart.tsx` contain **zero** `fetch(` calls (verified by count: 0 and 0). They are purely presentational and receive `gainedDate` as a prop. The daily-earnings request lives in the **page**, in its own `useEffect` (`earnings/page.tsx:114-138`, `fetch('/api/analytics/views-by-day?...')`), keyed on `[timeframeDays, selectedCampaigns]`. That effect runs on mount whether or not the chart component has loaded. **Deferring the chart cannot move, delay or duplicate any request.**

### 2. Server rendering and hydration mismatch? NO RISK, and this is the strongest safety argument.

`earnings/page.tsx:42` sets `loading = useState(true)` and `:33` sets `earnings = useState(null)`. The very first thing the component does is `:164` `if (loading && !earnings) return <spinner/>`. **On the server, both conditions hold, so the server renders ONLY the spinner.** `EarningsPremium`, and therefore the chart, **never appear in the server HTML at all** and never participate in hydration. Adding `ssr: false` changes nothing about what the server sends, because the server already sends nothing but a spinner.

One version detail that matters: Next 15+ forbids `ssr: false` in **Server** Components. `EarningsChart.tsx:1` is `"use client"`, so the option is legal here. Next in this repo is **16.2.6**.

### 3. Layout shift? ZERO, if the placeholder reserves the height.

Because all three existing states are already pinned to 220px (PART 2), reserving 220px in the dynamic placeholder makes the shift **exactly zero**. If the placeholder were omitted, the slot would be 0px and would jump 220px when the chunk lands, which is a real shift **below the fold**. That is why the recommendation includes a fixed-height placeholder rather than nothing.

### 4. A slow or failed chunk on bad signal? DEGRADES CORRECTLY.

With the recommended shape (defer the recharts **leaf**, not the whole `EarningsChart`), everything outside the leaf stays static and renders immediately:

| Still renders if the chunk never arrives | Source |
|---|---|
| The whole page above the chart, including **every money number** | `EarningsPremium.tsx:95-212` |
| "Earnings over time" heading, TimeframeSelect, EarningsFilters | `:214-221` |
| The chart `Card`, its "Daily earnings" wrapper, the loading skeleton, the empty-state message, the error banner | `EarningsChart.tsx:54-112` |

The clipper sees a working page with all their figures, and a blank 220px box where the graph would be. **No error, no crash, no blocked interaction.**

### 5. Does another page share the chunk? YES, and it limits the win, not the safety.

The identical 380.9 KB chunk is on **`/referrals`** (clipper-facing) and on four non-clipper pages. Deferring inside `EarningsChart.tsx` fixes **`/earnings` only**. `/referrals` imports `AreaGradientChart` directly (`ReferralsRedesign.tsx:16`) and would need its own separate one-line change to get the same 110.8 KB back. **Recommendation: do `/earnings` first, alone, prove it, then decide about `/referrals`.** Touching both at once doubles the surface of a pattern this codebase has used once.

### 6. Does ANY money figure depend on the chart? NO. Proven.

| Figure | Source | Depends on chart? |
|---|---|---|
| Hero **available for payout** | `summary.available` (`EarningsPremium.tsx:103`) | **No** |
| Available (repeat, payout row) | `summary.available` (`:115`) | **No** |
| Earned / Approved / Paid out trio | `summary.totalEarned` / `.approvedEarnings` / `.paidOut` (`:124-126`) | **No** |
| Pending / in payout queue chips | `summary.pendingEarnings` / `.lockedInPayouts` (`:160,165`) | **No** |
| Earnings by campaign | `byCampaign`, built from `clips` | **No** |
| **Request payout** action | a plain `<Link href="/payouts">` (`:109`) | **No** |

`gainedDate` (the chart's data) appears in exactly **three** places in the entire flow: `earnings/page.tsx:51` (state), `:189`/`:345` (passed down), and `EarningsPremium.tsx:228` (passed to `EarningsChart`). **It feeds nothing but the chart.** `summary` comes from `computeBalance` in `balance.ts` via `/api/earnings`, and withdrawal availability is enforced server-side at `/api/payouts`, entirely independent of anything rendered here.

**Stated plainly: a clipper's earnings numbers, balance and withdrawal availability cannot be affected by this change, in success or in any failure mode. Money display remains exactly correct and always visible.**

## PART 5 — the verdict and the build spec

### Recommendation: ONE approach

**Defer the recharts leaf inside `EarningsChart.tsx`.** Measured saving **111 KB gzip, 41.2% of the page**, taking `/earnings` from **269 KB to 158 KB**. Do `/earnings` alone first; leave `/referrals` for a later decision.

**Why the leaf and not the whole `EarningsChart`:** deferring `EarningsChart` itself would pull its loading skeleton, empty-state copy and error banner into the deferred chunk, so none of them could render until the chunk arrived, and it would need edits in **two** files (`earnings/page.tsx:13` and `EarningsPremium.tsx:15`). Deferring only the leaf is **one line in one file**, keeps every existing loading and error state static, and covers both call sites automatically.

### The exact change (spec for a later round, NOT applied here)

File: `src/components/earnings/EarningsChart.tsx`. Replace the static import on **line 4**:

```tsx
import { AreaGradientChart } from "@/components/ui/area-gradient-chart";
```

with:

```tsx
import dynamic from "next/dynamic";

// BL-643 — recharts is 380.9 KB raw / 110.8 KB gzip, 41% of the /earnings
// download, and the chart sits ninth on the page, far below the fold. Defer it
// so a clipper on mobile data gets their earnings numbers without paying for
// the charting engine first. ssr:false is safe and free here: earnings/page.tsx
// returns a spinner on the server (loading=true, earnings=null), so this tree
// never server-renders and never hydrates. The placeholder reserves the same
// 220px every other state uses, so nothing shifts.
const AreaGradientChart = dynamic(
  () => import("@/components/ui/area-gradient-chart").then((m) => m.AreaGradientChart),
  { ssr: false, loading: () => <div style={{ height: 220 }} aria-hidden="true" /> },
);
```

**The placeholder is a plain empty box. No pulse, no shimmer, no text, no fake content. It is not a skeleton.** It is strictly less visible than the pulsing skeleton already shipping at `EarningsChart.tsx:54-72`.

**One coupling to note honestly:** the placeholder hardcodes 220 while the real height is a prop. Today that is always correct, because `EarningsPremium.tsx:223` passes no height and the default is 220. If a future caller passes a different height, the placeholder would be wrong and a shift would return. Worth a comment in the code.

### Rollback

**Restore the single static import line.** One line, one file, no data, no schema, no config. `git revert <sha>` also suffices.

### What would prove it safe before shipping

1. `npm run build` with **BUILD_EXIT=0** read from a log, plus the hooks gate at 0 errors and at most 11 warnings.
2. Re-run the bundle measurement and confirm `/earnings` gzip drops **269 → ~158 KB** and the recharts-bearing chunk is **absent** from the `/earnings` manifest.
3. Throttle to slow 3G and open `/earnings`: the hero **available for payout** figure renders with the rest of the page; the chart area holds **220px**; **no layout shift**; the chart appears when the chunk lands.
4. **Block the chart chunk** in DevTools request-blocking and reload: every money number still renders, the page stays usable, no crash, no error overlay.
5. Console shows **zero hydration warnings** (expected, since the tree never server-renders).
6. Confirm `/referrals` and the four admin/client pages are byte-unchanged, since none imports `EarningsChart`.

### The regression test that would catch a problem

A standalone Playwright script asserting, on `/earnings` with the chart chunk request blocked: the "available for payout" figure is visible and correct, the chart container measures 220px, and no console error is raised. **Honest caveat:** `playwright@^1.60.0` is a dependency but there is **no `test` or `e2e` npm script and no configured suite**, so this would be a new standalone script, not an addition to an existing harness.

### If the owner would rather not

**Then 269 KB is acceptable and he should decline.** It is not a broken page; it is a page carrying 111 KB it does not need on first paint. The honest trade is: **41% off the heaviest clipper page, against one import line in one file, on a pattern this codebase has used exactly once before, with a plain 220px box appearing in a slot that already shows a pulsing skeleton today.** If that is still more loading-behaviour change than he wants after two bad experiences, the cost of doing nothing is bounded and non-growing: clippers keep downloading 269 KB, roughly 1.4 s on slow 4G. **Nothing degrades, nothing breaks, and this can be revisited any time.** The recommendation stands, but declining it is a legitimate choice and not a risk to anything.

## What could NOT be measured

* **Whether clippers actually look at the chart** (PART 4). No page-view or interaction telemetry exists.
* **Real device timings** (LCP, TTI, main-thread parse cost of recharts on a low-end phone). Needs Lighthouse mobile or WebPageTest on real hardware; the 1.4 s figure is arithmetic from measured bytes.
* **The real-world race** between the 110.8 KB chunk download and the `/api/analytics/views-by-day` response, which decides how often the new placeholder is visible at all. Needs the throttled manual test in the proof list.
* **Whether Turbopack would split the chunk differently once the import is dynamic.** The 158 KB projection assumes the recharts chunk simply leaves the initial graph, which is the normal outcome; step 2 of the proof list verifies it rather than assuming it.
