## THE CRON IS NOT REGISTERED — it will NOT fire as merged (owner action required)
Railway self-schedules crons from the `CRON_JOBS` registry in `src/lib/railway-cron-scheduler.ts` (lines
63-85), gated by the `RAILWAY_NATIVE_CRON` env allowlist (`isJobEnabled`, :95). `retire-dead-clips` is in
NEITHER, so **the recurring retirement will never fire on its own** (the vercel.json crons do not run
because hosting is Railway, per BL-568). To activate it, BOTH are needed:
1. **A follow-up CODE round** must add an entry to `CRON_JOBS`, e.g.
   `{ name: "retire-dead-clips", path: "retire-dead-clips", due: (n) => n.getUTCHours() === 6 && n.getUTCMinutes() === 0 }`
   (daily 06:00 UTC). A merge-only round cannot add this, so it is NOT done here.
2. **The owner** must add `retire-dead-clips` to `RAILWAY_NATIVE_CRON` in Railway (comma-separated allowlist;
   `*` alone will not enable it until the CRON_JOBS entry above exists).
Until both land, the endpoint fires only when triggered by hand:
`GET /api/cron/retire-dead-clips?dryRun=1` with header `Authorization: Bearer <CRON_SECRET>` (drop `dryRun`
to run live). The recommended first action is a manual `?dryRun=1` to confirm the counts before scheduling.
