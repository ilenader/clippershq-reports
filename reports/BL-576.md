## PART 0 — why `consecutiveFailures` is pinned at 0 for gone clips (file:line; DELIBERATE)

A definitively-gone Instagram clip returns `stats:null` from `fetchClipStats` (apify.ts:2157, the
BL-275 gone-skip). It does NOT throw with the `.notFound` sentinel that only the TikTok deletion path
sets (captured at tracking.ts:1432). So in `processTrackingJob` the failure is classed `infra`:

- **tracking.ts:1475** — `const failureClass = fetchErrNotFound ? "not_found" : "infra";`
- **tracking.ts:1489-1508** — the INFRA_DEFER branch handles `failureClass === "infra"`, and its own
  comment at **:1497** reads *"consecutiveFailures intentionally NOT incremented (no strike)."*

This is **correct and must not be disturbed**: a `gone_404` is a definitive, correct provider answer,
not an infrastructure failure, so it must never strike a good clip. Consequently `consecutiveFailures`
can never accumulate for a deleted post and must NOT be repurposed. The new signal has a DIFFERENT
meaning — *"the platform has told us this post no longer exists, N times running"* — in a SEPARATE
counter.

Verified live: `consecutiveGone | integer | default 0 | NOT NULL`, both timestamps nullable; all 3,962
jobs read `sum(consecutiveGone)=0`, 0 first/last set — nothing moved.

**Mechanism.** The definitive-gone verdict is produced in the BATCH fan-out (apify.ts) and was
previously thrown away. It is now surfaced per-clip:

- `PlatformBatchOutput.goneUrls` (normalized URLs, optional) is populated from the SAME verdict that
  already drops the URL from the Apify fallback: IG `verdict==="gone"` (HikerAPI /v2 404,
  hikerapi.ts:701) and TT `goneSet` (LamaTok 404 + MediaNotFound, apify.ts:1049 — BL-559's
  urlClass-independent fix).
- `fetchClipStatsBatch` maps those URLs back to clipId and returns `FetchBatchResult.goneClipIds`.

**Capture point = the BATCH, on purpose.** The deleted reels are 10-31 days old, so they take the
`>=48h` age-fallback (`FALLBACK_AGE_LIMIT_H`, tracking.ts:1339) and NEVER reach the individual-fetch
path. The batch runs Hiker/LamaTok for **every** due clip regardless of age, so it is the only place
the whole dead population is observed.

**Counter maintenance** — `src/lib/gone-counter.ts` (`maintainGoneCounters`), called fire-and-forget
(`void`) from tracking.ts right after the global batch resolves. It writes ONLY the three new columns
and swallows every error, so it can never affect any tracking decision, timing, cadence, or money.

- **Increment ONLY on a definitive provider-gone verdict** (clipId in `goneClipIds`).
- **NEVER on ambiguity** — a transient 5xx, a timeout, an empty 200, or the cooldown short-circuit at
  hikerapi.ts:614 (returns `verdict="transient"` with NO network call, the leak-burst path BL-571
  proved fires during the burst) never set `verdict="gone"`, so they are never in `goneClipIds`.
- **RESET to 0 (+ clear both timestamps) the instant a clip resolves** (real stats in the batch this
  tick). A temporarily-private account that comes back loses its gone history entirely. Resolve wins
  over any stale gone verdict.
