**2026-07-19 · AUDIT ONLY. READ ONLY on code, config, data and money. Nothing changed, pushed, merged or flipped.**

## THE ANSWER

**Yes, all three Apify Instagram profile/bio call sites can be cut to HikerAPI with zero data loss. Proven by live probe, not read from code.** HikerAPI `GET /v2/user/by/username` returns every field all three sites consume: `biography`, `follower_count`, `is_verified` AND the avatar (`profile_pic_url`, HD via `hd_profile_pic_url_info`).

**But two premises in the brief are wrong, and honesty requires stating them:**

1. **Verification did NOT move off Apify.** BL-586 shipped bio persistence and link validation. It did NOT change the provider. Instagram verification still calls Apify FIRST today (`verify/route.ts:229` to `:295`). BL-586 PART 1 even says it stores "the clean structured Apify biography". So these calls are **not dead weight yet**: live IG verification actively depends on them right now.
2. **They are therefore not cuttable in this round, and must not be.** This round PROVES the HikerAPI replacement is complete and safe. The cut itself is a separate, small, non money code round, because a wrong cut locks clippers out of earning.

## PART 2 what the live verify path calls TODAY: Apify, not HikerAPI

Traced end to end on current main. When a clipper verifies an IG account: `POST verify/route.ts:603` calls `checkInstagramBio(profileLink, code)`. Inside (`:229` to `:295`) **ATTEMPT 1 is Apify** (`apify~instagram-profile-scraper`, `:238`), reading `d.biography` (`:266`) for the substring code match at `:154`. Only on Apify failure does it fall to Browserless (`:308`). **There is no HikerAPI anywhere in the IG verify path.** So the answer to "does it hit Apify or HikerAPI today" is: **Apify, first, every time.**

## PART 3 what breaks if cut: nothing. Field by field.

| Apify field consumed | Site | HikerAPI replacement | Covered? |
|---|---|---|---|
| `d.biography \|\| d.bio` | verify:266 | `user.biography` (empty string when no bio) | YES |
| profile pic capture | verify:238 / :884 | `user.profile_pic_url`, prefer `hd_profile_pic_url_info.url` for HD | YES |
| `d.profilePicUrlHD \|\| d.profilePicUrl` | account-profile:138 | same as above | YES |
| `d.followersCount ?? d.followers` | account-profile:139 | `user.follower_count` | YES |
| `d.verified` | account-profile:140 | `user.is_verified` | YES |

**No Apify only field.** Every value is present in the one Hiker response. Two shape notes, neither a data loss:
* **Naming differs** (Apify camelCase `followersCount`/`profilePicUrlHD`/`verified` vs Hiker snake_case `follower_count`/`profile_pic_url`/`is_verified`). The new helper remaps.
* **Input shape differs.** The Apify actor takes a profile URL; Hiker takes a username. `account-profile.ts` already has the `username` column; the verify sites extract it from `profileLink`. Trivial.
* **HD avatar:** Apify hands `profilePicUrlHD` directly; Hiker's `profile_pic_url` is a 150x150 thumbnail, with HD under `hd_profile_pic_url_info`. Read the HD field first to match today's quality.

## PART 4 the verdict and the exact next step

**Every Apify IG profile/bio call can be cut to HikerAPI with zero data loss.** Confirmed by the field map above against a live probe. Doing so removes the entire `apify~instagram-profile-scraper` actor from the bill (~$0.29/mo) and, as a bonus, makes these calls VISIBLE in the ledger for the first time.

**But this is the earning gate.** If the bio fetch fails, the clipper cannot verify and cannot earn. So the cut is phased, not a hard swap:

**Next step, BL-590 (a later round, code only, non money, no test gate needed):**
1. Add `fetchHikerInstagramProfileByUsername(username)` to `src/lib/scraper-providers/hikerapi.ts`: `GET /v2/user/by/username`, header `x-access-key`, map `biography` / `follower_count` / `is_verified` / `hd_profile_pic_url_info.url ?? profile_pic_url`.
2. **Ledger wiring:** call the existing `recordHikerUsage` (`hikerapi.ts:851`) inside the helper so profile calls log under a `hikerapi-profile` provider. Strictly better than today's invisibility. Cost stays ~$0 (prepaid).
3. **`account-profile.ts:132` (cosmetic avatar/follower display, not earning gating): cut to Hiker fully.** Failure here is harmless.
4. **`verify/route.ts:238` and `:884` (the earning gate): insert Hiker as ATTEMPT 0 BEFORE Apify, keep Apify as fallback for one release.** Measure that Hiker resolves the bio in production, THEN remove the Apify branch and the `APIFY_IG_PROFILE_ACTOR` const. Zero clipper risk during transition.
5. After step 4 telemetry is clean, delete the Apify IG profile actor and the stale `APIFY_INSTAGRAM_ACTOR` env var (BL-552 flagged it inert).

**No call genuinely cannot move.** All three are covered by the single probed endpoint.
