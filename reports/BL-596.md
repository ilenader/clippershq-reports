# BL-596 — Structured Rules + AI Pre-Check Foundation (DESIGN, build nothing)

Owner's ask: auto-reject only the clear violations (wrong sound, required caption token
literally absent, logo missing through the video), and NEVER auto-reject borderline work (a
merely bad caption goes to a human). R-6 proved the blocker: campaign rules are free text that
nothing reads because the form flattens per-rule items into one text blob at the DB boundary.
This doc designs the structured layer beneath the prose, the free caption/sound checks, the
cheapest honest logo check, and the ordered rollout. It is a design only. No code, no schema
change, no build was run (a markdown diff cannot move tsc).

## THE HARD LINE (BL-518, non-negotiable, governs every part below)
No machine silently changes a clip's status on a judgment call. A rule may AUTO-REJECT only when
the violation is machine-certain: a literal, deterministic, fetched-and-confirmed fact (sound id
present on both sides and different; a required token literally absent from a successfully fetched
caption). Everything else RANKS a clip for a human and never rejects. R-2 measured an 11.05%
false-rejection rate on a blind combined gate; at ~2,300 clips/month that is roughly 23 honest
clippers insulted every month. Auto-reject stays surgical or it does not ship. Every auto-reject
path FAILS OPEN: if the metadata was not fetched, the clip goes to a human, never to rejection.

Live baseline (queried today, cast ::text): 11 of 30 campaigns carry `requirements`, 86 rule
lines total, 4 campaigns with a non-empty `soundLink`, 3 with `hashtagRules`, 2 with
`captionRules`. This matches R-6's fill-rate measurements (11/30, 4 sounds, 3 hashtag).

## PART 1 — THE STRUCTURED SCHEMA (R-6's core fix)

### What exists now (verified in code)
The admin campaign form holds rules as real discrete items and then destroys the structure:
• `requirementsList: [""] as string[]` — one string per rule row (page.tsx:55).
• Each row renders as a plain `<Input>` (page.tsx:1191-1200); "Add requirement" pushes an empty
  string (page.tsx:824).
• On submit the array is flattened: `const requirements = form.requirementsList.filter(r =>
  r.trim()).join("\n")` (page.tsx:294) into the single `requirements String? @db.Text` column
  (schema.prisma:597).
• On edit it is split back: `c.requirements.split("\n").filter(...)` (page.tsx:178).

So the per-rule boundary already exists in the UI and is thrown away at `join("\n")`. Nothing
downstream can tell rule 1 from rule 4, or a sound rule from a taste rule. That single join is the
whole blocker. The fix is not a prose parser (R-2 proved that path: treating prose as literal
strings produced 9.47% false-rejection and "cannot be machine-checked without being redesigned
into a structured field first"). The fix is to keep the structure the UI already has and annotate
each row with a type.

### The change
1. Add a **type dropdown** to each requirement row (next to the existing `<Input>` at
   page.tsx:1193). Default `HUMAN_ONLY` so any un-typed / legacy rule behaves exactly as today
   (a human reads it).
2. Add a new nullable **`rulesJson Json?`** column on `Campaign`. Nullable, default null, so every
   one of the 30 existing campaigns is byte-identical until an owner opens the form and types a
   rule. It is display-invisible and never touched by any money-path file (same posture as
   `sounds`/`contentUrl`).
3. **Keep `requirements String? @db.Text` as the clipper-facing source of truth.** The verbatim
   sentence the owner typed is what the clipper always sees. `rulesJson` is the machine layer
   *beneath* the prose, never shown. The clipper sees the sentence, never the JSON (R-6).

### rulesJson shape (one object per rule, R-6's structure)
```
{
  "id": "r1",
  "text": "<verbatim rule, ALWAYS the sentence shown to the clipper>",
  "type": "REQUIRED_SOUND | REQUIRED_CAPTION_TOKENS | LOGO_PRESENT_THROUGHOUT | POSTING_WINDOW | VIDEO_LENGTH | EDIT_QUALITY | HUMAN_ONLY",
  "platforms": ["tiktok", "instagram", "youtube"],
  "params": { "tokens": ["#somesome"], "soundId": "...", "position": "anywhere|starts_with|ends_with", "coverage": "cover_frame|majority|entire", "minSec": 0, "maxSec": 0 },
  "enforcement": "auto_reject | rank | human_only"
}
```
`text` is mandatory on every rule; it is the sentence, and it is what a rejected clipper reads as
the reason. `enforcement` is derived from `type` (below) but stored explicitly so a future type can
be added as `rank`-only without a code change.

### Rule types, by what R-6's clustering of the 10 active campaigns actually found
Ranked by R-6 frequency. The middle column is the owner's absolute line: auto-reject requires
machine certainty.

| Type | R-6 rank / campaigns | Machine-certain? | enforcement | How it is stored + checked |
|---|---|---|---|---|
| `REQUIRED_SOUND` | rank 3, 5/10 ("specific sound, official only") | YES, by id | **auto_reject** | `params.soundId`; migrate `soundLink` → `soundId`. Reject ONLY when a soundId is present on the clip AND on the rule AND they differ. |
| `REQUIRED_CAPTION_TOKENS` | rank 2, 6/10 ("caption content") | YES iff literally absent | **auto_reject** | `params.tokens[]`; migrate `hashtagRules` → `tags[]`. Reject ONLY when a token is literally absent from a fetched, NFKC-normalized, case-folded caption. |
| `LOGO_PRESENT_THROUGHOUT` | rank 1, 8/10 ("on-screen visual") | NO (pixels) | **rank** | `params.coverage`; vision sampling (PART 3). Ranks the queue, never rejects. |
| `POSTING_WINDOW` | rank 5, 4/10 | YES, but already enforced | **rank / no-op** | Already blocked at submit (30-min window). R-2: 0% false, 0 violations to catch. Not a new auto-reject. |
| `VIDEO_LENGTH` | 0/30 use it today | YES iff duration known | **auto_reject (low priority)** | `params.minSec/maxSec`; duration is in the provider raw. Nobody sets it yet, so ship the type, not urgency. |
| `EDIT_QUALITY` / "vibe" | rank 4, 5/10 ("taste") | NO (subjective) | **human_only** | Verbatim `text` only. Never machine-scored. Always a human. |
| `HUMAN_ONLY` (default) | catch-all | n/a | **human_only** | The safe default for every legacy/un-typed rule and every external-rulebook rule (R-6: 4 rejections pointed to Google Docs). |

Auto-rejectable: `REQUIRED_SOUND`, `REQUIRED_CAPTION_TOKENS`, and (dormant) `VIDEO_LENGTH`.
Ranks-only: `LOGO_PRESENT_THROUGHOUT`, `POSTING_WINDOW`. Human-only: `EDIT_QUALITY`, `HUMAN_ONLY`.
A "merely bad caption" has no literal token rule attached, so it is `HUMAN_ONLY` and can never be
auto-rejected. That is the owner's borderline guarantee, encoded in the type, not in a heuristic.

## PART 2 — WHAT IS ALREADY FREE (measured in code, not assumed)

The submit path calls `fetchClipStats` (apify.ts:2186), the same function `actions/clips.ts:74`
uses. BL-552's live probe (`scripts/bl552-probe.ts`) confirmed LamaTok serves the TikTok submit
path and HikerAPI serves Instagram.

### The honest finding: the data is fetched but currently discarded
`fetchClipStats` returns `ClipStats` (apify.ts:280), which captures ONLY:
`views, likes, comments, shares, createdAt, platform, raw?`. There is **no** caption, soundId, or
hashtags field on `ClipStats`, and **no** `caption` / `soundId` / `hashtags` column on the `Clip`
model (grep of schema.prisma: only `captionRules` on Campaign exists). So today the caption, sound,
and hashtags are NOT parsed and NOT persisted anywhere.

But they are already in hand at zero extra vendor cost, sitting unread in the provider `raw`:
• **TikTok / LamaTok** (`lamatok.ts:273`): the parsed media object mirrors TikTok's
  `aweme_detail`. That object carries `desc` (the caption), `music.id` / `music.title` /
  `music.author` (the sound), and `text_extra[]` / `cha_list` (the hashtags). The provider already
  parses `statistics` off this exact object for views; the caption/sound/hashtag siblings are right
  there and are dropped into `ClipStats.raw` ("full response for debugging", apify.ts:287).
• **Instagram / HikerAPI** (`hikerapi.ts:753`): the result already carries `raw: res.raw`, whose
  media object holds `caption_text`, `usertags`, and `clips_metadata` / `music_info`.

So the caption-token check and the sound check are **FREE**: no new vendor, no new API call. What
they need is (a) parse those three fields out of the raw the submit path already receives, and (b)
persist them to a new column. This is exactly R-6 step 1 ("Persist caption cron already fetches,
$0, retroactive caption checks"). The cost is a small parse-and-store, not a fetch.

### The YouTube honesty caveat
R-2 measured YouTube at 43% of rule-carrying clips and **entirely inaccessible via TikHub**, and
overall metadata reachability at only 14.2%. So caption/sound checks are free for TikTok and
Instagram, and are a NO-OP (fail-open to human) for YouTube. This is why every auto-reject path
must fail open: no fetched caption means the clip ranks for a human, it is never rejected.

### Cover-frame price (confirming R-6)
R-6: the cover frame is already stored in Supabase, so template-matching + OCR against it costs $0
in vendor terms, and a Haiku 4.5 vision pass on it is **$4.48/month, $2.24 batched**. The prompt's
"$4.82/mo" is R-6's *combined* directional figure (cover-frame $2.24 + metadata $2.58 = $4.82),
not the cover-frame line alone. Both are R-6-measured and cited as such.

## PART 3 — THE LOGO-THROUGH-THE-VIDEO CHECK (owner's specific ask)

The owner wants the logo present through the MAJORITY of the video, not just the cover frame. The
cover frame alone answers "is the logo there at second 0"; it cannot answer "throughout".

### R-6's measured cost envelope
• Per-second sampling (30 frames on a 30-second clip): **$55.22/month** (Haiku batched) PLUS an
  unpriced video download of ~23 GB/month ingress. Misses the sub-$50 target.
• 3-frame sampling: **$5.98/month**, and R-6 found it answers "majority" and "entire" nearly as
  well as per-second.

### The design: sample enough frames to prove "throughout", no more
Sample at **0%, 25%, 50%, 75%** of the clip (4 frames) and require the logo present in the
majority of sampled frames for `coverage: majority`, or all of them for `coverage: entire`. Four
evenly-spread frames distinguish "logo only on the cover" from "logo throughout" cheaply; the
`coverage` param (`cover_frame | majority | entire`) is exactly R-6's temporal param, motivated by
the real rule "logo must stay fully visible at all times".

### Real cost at 77 clips/day (~2,310/month)
The only MEASURED figure is R-6's **3-frame sampling at $5.98/month** at this volume. Four frames
is one-third more images, so ~$8/month by linear extrapolation — this $8 is **UNVERIFIED**, an
extrapolation of R-6's 3-frame measurement, not a measured price. Recommendation: ship the 4-frame
0/25/50/75 sampler but treat $5.98/mo (3-frame, verified) as the floor and ~$8/mo (4-frame) as the
UNVERIFIED planning number. Frames beyond the cover are not stored, so this requires a video
download; the cover-only path stays $0. Railway bandwidth for that download is **UNVERIFIED** (R-6
could not read the billing from the machine).

### Accuracy: honest, because it is not certain
R-2 found on-screen-visual rules have **NO measured accuracy yet**. Vision is not machine-certain.
Therefore `LOGO_PRESENT_THROUGHOUT` **RANKS the queue for a human and never auto-rejects**, no
matter how confident the model sounds. It orders which clips a human looks at first; a human still
clicks.

### Ordered build (each step is reversible and ships behind the default)
| # | Step | Cost | Why this order |
|---|---|---|---|

| 2 | Parse `desc`/`music.id`/hashtags (LamaTok raw) + `caption_text`/`usertags`/music (HikerAPI raw) into new `Clip` columns | $0 (data already fetched) | Makes caption + sound checks free and retroactive. R-6 step 1. |
| 3 | Free auto-reject: `REQUIRED_CAPTION_TOKENS` (literal absence) + `REQUIRED_SOUND` (id mismatch), fail-open to human on any fetch miss | $0 | Only the two machine-certain checks. Ship after real data is flowing (step 2). |
| 4 | Logo ranking: cover frame ($2.24/mo) then optional 4-frame throughout (~$8/mo UNVERIFIED). RANKS only | $2.24/mo, +~$8 UNVERIFIED | Ranking never blocks a clipper, so it is safe to ship last and prove it beats random first (R-6 steps 5-6). |

Backfill (the afternoon) slots between step 1 and step 3: once the dropdown exists, classify the
86 lines, then turn on the auto-reject.

### False-certainty risk per auto-reject rule, and the appeal path
**`REQUIRED_CAPTION_TOKENS`** — risk: the clipper edits the caption after submit; a token is
present but Unicode-differs (emoji, zero-width, full-width); the caption fetch was stale or missed.
Mitigation: NFKC-normalize + case-fold before an exact substring test; reject ONLY on confirmed
literal absence in a successfully fetched caption; fail open on any fetch miss (→ human). R-2
measured 8.95% false-rejection when required-hashtag was a *blind* gate and found 58.8% of readable
clips broke the rule while humans approved 85% of them anyway — which is exactly why this rejects
only on the campaign author's explicit literal token, not on prose intent.

**`REQUIRED_SOUND`** — risk: TikTok remixes/reuses sound ids; official vs user-uploaded id drift;
the id is absent on one side. Mitigation: reject ONLY when a soundId is present on BOTH the clip
and the rule AND they differ; fail open otherwise (→ human).

**Appeal path (both rules):** the clipper never sees "machine rejected you". They see the same
human-style prose reason keyed to the rule's verbatim `text`, and can request re-review. Because
step 2 persisted the raw caption/soundId, a human verifies the call in a single look and overrides
if the machine was wrong. The override path already exists (owner overrides, R-5).

**Never a prose parser.** R-2 proved treating the freeform caption field as a literal string
requirement produced 9.47% false-rejection and that the field "cannot be machine-checked without
being redesigned into a structured field first". PART 1 is that redesign; the checks read
`params.tokens[]` and `params.soundId`, never the prose.

## BOTTOM LINE
Restore the structure the form already has (kill the `join("\n")`), add `rulesJson` and a per-row
type dropdown, keep the sentence as the clipper's source of truth. Two checks are free because the
submit path already fetches the caption and sound into `raw` and merely discards them; those two,
and only those two, may auto-reject, and only on a literal fetched-and-confirmed violation that
fails open. The logo-throughout check is a 4-frame sampler that RANKS and never rejects because
vision has no measured accuracy. Backfill is 11 campaigns / 86 lines / one afternoon. Everything
borderline stays with a human, by type, forever.
