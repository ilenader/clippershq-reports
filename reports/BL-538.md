## PART 1 — what a freeze does (verified against current main, not re-derived)

| What changes | Where | Undo reverses? |
|---|---|---|
| Resume row `toStatus=ACTIVE` + `fromPauseSource=AUTO` = **the era boundary** | `campaign-era.ts:63-80` | **Yes** (`eraExempt`) |
| Gate A `isAutoPausedCampaign` → `campaignStatusBlocks` → recompute gate | `tracking.ts:1863`, `1927-1932`, `1933` | n/a, resume clears it |
| Gate B BUDGET-LOCK (independent) | `tracking.ts:1959-1961` | n/a, resume nulls `lastBudgetPauseAt` |
| Gate C `isOldWindowClip` → same gate | `tracking.ts:1880-1881`, `1932` | **Yes**, follows from the boundary |
| Cadence `AUTO_PAUSED` / `OLD_WINDOW` → 4320 min | `tracking.ts:281-289` / `301-311` | **Yes, both** |

**The key fact:** after a resume, gates A and B are *already off* (`pauseSource` + `lastBudgetPauseAt`
nulled at `unfreeze/route.ts:121`). **The era boundary is the ONLY thing still holding the freeze** —
which is why neutralising it is sufficient, and why BL-530 removing it was so dangerous. Views were
never affected: ClipStat is written at `tracking.ts:1770/1782`, above the gate (confirms BL-534).

**Something BL-523 and BL-525 both missed.** Resetting `nextCheckAt` alone is **not enough**. BL-200
orders the due-jobs query `checkIntervalMin ASC` (`tracking.ts:3611`) under the 200-call/tick cap, on
purpose, so live clips beat ancient ones. Clips still stamped 4320 therefore sort **last** and can be
starved behind fresher work no matter how overdue `nextCheckAt` says they are. **The interval IS the
sort key.** The BL-145 unfreeze route resets only `nextCheckAt` (`157-168`). BL-538 moves interval +
nextCheckAt + clears `cadenceReason`. An incomplete undo looks fixed and is not.

## PART 2 — the undo

`GET/POST /api/admin/campaigns/[id]/undo-freeze` — OWNER only, rate-limited, never automatic.
Button "Freeze was a mistake" on any ACTIVE campaign in `/admin`.
Snapshot + exact rollback → exempt the boundary → reset **both** cadence reasons → guarded recompute
via the existing `recalculateClipEarningsBreakdown` + `writeClipEarnings`. Nothing hand-credited
(BL-522: HIGH double-pay risk). Blast radius in clips, clippers and dollars shown **before** acting,
produced by the *same code* that then executes.

* **Never decreases — guard in COMMITTED code** (`src/lib/earnings-never-decrease.ts`). BL-528 was
  right: the old guard lived only in an uncommitted script. **Also applied to
  `/api/admin/force-recalc-earnings`**, which wrote `newEarnings` unconditionally and exposed 123
  clips. ON by default there; explicit `allowDecrease:true` keeps its ability to repair clips that
  drifted too high. Blocked writes surface in the response + audit, never silent.
* **Never pays past a cap** — free, by routing through `writeClipEarnings`, which *is* the L1 budget
  hard-lock. A cap rejection is recorded as blocked, never reported as money that moved.
* **Idempotent** vs a second press and the live 10-min cron: the recompute is absolute, not a delta.
* **Walk-back guard:** POST demands the exact `statusChangeId` the preview returned. Without it a
  second press would exempt backwards through history, each press looking like a harmless no-op.
* **Scope guard:** only `MANUAL_OWNER` resumes are undoable. A real budget top-up is refused, with a
  message pointing at the resume choice instead. BL-522 asked for exactly this.

## Honest limits

* **AgencyEarning is not written by the undo.** BL-541 holds the owner-share accrual and
  `writeClipEarnings` documents that derived tables stay in caller code. The cron re-syncs it on the
  next poll (which the undo just pulled forward), so the owner-side number can lag the clipper-side
  by ~10 min on a CPM_SPLIT campaign. No money is lost either way.
* Marketplace clips are unfrozen and counted, but recomputed by the cron's marketplace fork (writing
  `Clip.earnings` via the single-share helper would break 60/30/10).
* 500-clip cap per CLAUDE.md. STRAENGE's 493 fits; the count is shown, so a bigger freeze is visible.
* It cannot invent views that did not happen.
* **This round touched 12 files, over the 5-file rule.** The 4-part spec was the plan; flagging it.
