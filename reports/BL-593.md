# BL-593 — Register the dead-clip retirement cron so it actually fires

## Problem
BL-590 built `/api/cron/retire-dead-clips` and BL-591 merged it, but it was never wired into the
scheduler, so it never runs on its own (the exact BL-568 trap: a built cron that silently does nothing).

## Change (ONE file, +8 lines)
`src/lib/railway-cron-scheduler.ts` — one new entry in `CRON_JOBS`, matching the existing entry shape:

```
{ name: "retire-dead-clips", path: "retire-dead-clips", due: (n) => n.getUTCHours() === 6 && n.getUTCMinutes() === 0 },
```

Daily 06:00 UTC (a quiet hour). No new pattern invented; identical field shape to the other 8 entries.
The cron's own logic was NOT touched: `retire-dead-clips.ts`, `route.ts`, and `cron-watchdog-policy.ts`
are byte-identical by blob OID on both refs.

## Watchdog interaction (BL-566 / BL-529) — checked
The watchdog only watches kinds returned by `resolveWatchedKinds()`, which iterates `CRON_JOBS` and
looks each job up in `JOB_NAME_TO_HEARTBEAT_KIND`. I deliberately did NOT add `retire-dead-clips` to
that map, so:
- The watchdog does NOT watch this cron at all. It appears in the logged `unwatchedJobs` list, not the
  watched set. So when the owner adds `retire-dead-clips` to `RAILWAY_NATIVE_CRON`, there is NO false
  `CRON_DOWN` alert, before OR after its first run.
- The critical money-cron (`tracking`) watch is undiluted. `tracking` is unioned into the watched set
  unconditionally and is the ONLY `critical` tier (the only one that emails). Nothing here changes that.

**Correct watch tier: none (not watched).** A daily housekeeping retirement missing one run is not an
emergency; the money cron is. Even if it were mapped, the route writes no `cronRun` heartbeat, so it
would forever be classed `never-seen` (info-only, never `went-stale`) — either way, no false alarm. Not
over-engineered: leaving it out of the heartbeat-kind map is the minimal correct choice.

## First-run safety — confirmed intact
The scheduler fires the bare path with NO query string, so the first live run uses the endpoint
defaults, all of which are safe:
- **Cap:** `maxPerRun` default 100, hard-clamped `Math.min(maxPerRun ?? 100, 500)`. Cannot mass-freeze.
- **Mandatory live re-probe gate:** a clip is retired ONLY on a FRESH HikerAPI HTTP 404 this run. HTTP
  200 leaves it alive/untouched (tracking auto-restores); transient/5xx/network is ambiguous and SKIPPED,
  never retired. An outage cannot freeze anything.
- **Reversible, earnings-neutral:** writes `videoUnavailable=true` + `videoUnavailableSince` only; never
  writes earnings (frozen, not zeroed), never `trackingJob.delete`.

BL-590's dry run found 6 of 30 candidates genuinely dead. The cap + re-probe gate mean the first live
run retires only the small, freshly-404-confirmed backlog. A `dryRun`-first first run is available
(`GET /api/cron/retire-dead-clips?dryRun=1` with the CRON_SECRET bearer) if the owner wants to eyeball
it first, but it is belt-and-suspenders; the cron is already capped and gated.

## OWNER STEP (in Railway, owner does this — I did not touch env)
Add `retire-dead-clips` to the `RAILWAY_NATIVE_CRON` env var. Prior value was `lifecycle,watchdog`.

**Exact final string to set:**

Until this is set, the entry exists in code but the scheduler's per-job allowlist keeps it off (default
OFF is preserved). No redeploy of code is needed for the value change beyond Railway's normal restart.
