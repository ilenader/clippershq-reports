# BL-598 — Multi-Clip, Multi-Platform Batch Submission (DESIGN ONLY, TEST CLIPPERS ONLY)

Date: 2026-07-20. Branch: `checkpoint/BL-598`. Scope: DESIGN a single submit that
takes several clips across TikTok, Instagram and YouTube at once, each link mapped
to the correct connected account, collapsible rows, within the per-campaign limit,
ONE campaign at a time, exposed to TEST CLIPPERS ONLY. READ ONLY: no code shipped,
no build run (a markdown only diff cannot change tsc/build). Every claim below is
grounded in the current code, cited by file and line.

## VERDICT (one line)

The batch flow is buildable as a thin fan out over the EXISTING single submit
logic: extract the per link body of `POST /api/clips` into a shared core, loop it
per row inside one campaign, reuse the proven `owner-submit-bulk` outcome
accounting, gate the whole surface on the existing `isTestUser` flag. No money,
tracking, or schema change is required.

## PART 1 — HOW SUBMIT WORKS TODAY (and what a batch can reuse)

### The single submit path (`src/app/api/clips/POST`, lines 677-1261)

Body today: `{ campaignId, clipAccountId, clipUrl, note? }` — exactly ONE clip,
ONE account, ONE campaign. The gate order:

1. Auth + ban (`getSession`, `checkBanStatus`), then clipper role via
   `canActAsClipperFromSession` (line 690).
2. Idempotency replay keyed on `userId + clipUrl` for 60s (line 717) so a double
   click returns the same created clip, never a duplicate.
3. Burst rate limit 12/hr (`clip-submit:<userId>`), or 120/hr for
   `clipSubmitBypass` users (lines 743-762).
4. URL hygiene: `sanitizeClipUrl` (line 710), length <= 2000, `http(s)` only,
   `isAllowedSocialHost` host allowlist (line 787), TikTok `/photo/` block.
5. Account: `db.clipAccount.findFirst({ id, userId, status:"APPROVED" })` (line
   807) — the account must be OWNED and APPROVED.
6. RULE 0 membership: the user must have JOINED the campaign with any of their
   accounts (`campaignAccount` lookup, line 821).
7. Campaign state: not DRAFT/COMPLETED/PAST/PAUSED, not archived, not a test
   campaign, budget not exhausted (lines 834-870).
8. **THE PER-CAMPAIGN LIMIT (lines 871-903).** It is a per (user, campaign) DAILY
   cap: `maxPerDay = clipLimitOverride?.maxClipsPerDay ?? campaign.maxClipsPerUserPerDay ?? 3`.
   A `clip_limit_overrides` row (unique on `userId_campaignId`) REPLACES the
   campaign default (schema note, line 2011). It counts today's clips for that
   user+campaign (`createdAt >= startOfDay UTC`, `isDeleted:false`,
   `isMarketplaceClip:false`); at/over the cap it 400s with `{ remaining:0, limit }`.
9. RULE 1 platform vs account: `detectPlatformFromUrl(clipUrl)` (which calls
   `socialPlatformForUrl`, host based, `src/lib/social-hosts.ts:57`) must equal
   `account.platform` (line 907). RULE 1b: the URL platform must be in the
   campaign's `platform` CSV (line 916). RULE 1c: that platform must have a CPM set
   (`getCampaignCpmForPlatform`, line 932).
10. RULE 2 freshness: a live Apify/YT fetch (`fetchClipFreshnessWithRetry`) must
    show the post is <= `MAX_CLIP_AGE_MS` (30 min) old; transient fetch failures
    are lenient (refund the slot, accept, let cron verify) — bad_url is a hard 400.
11. Duplicate checks on `normalizedUrl` (`normalizeClipUrlForMatch`): campaign
    scope and user scope, both over status IN (PENDING, APPROVED) (lines 1057-1081).
12. CPM freeze: stamp `cpmAtSubmissionDecimal` + `ownerCpmAtSubmissionDecimal`
    (lines 1084-1104) so later CPM edits cannot move this clip's earnings.

### The identical per-clip END STATE a single submit produces (lines 1107-1167)

One atomic `db.$transaction` writes three rows, and THIS is the exact state a batch
must reproduce per clip:

• `Clip.create` — `status` defaults to PENDING (schema line 839), with `clipUrl`,
  `normalizedUrl`, `clipAccountId`, `note`, and the two frozen CPM decimals.
• `ClipStat.create` — the first snapshot (fetched views/likes/comments/shares, or
  zeros on a lenient accept).
• `TrackingJob.create` — `isActive:true`, `nextCheckAt` rounded to the next :00
  (+1h, pushed a step if the gap < 30 min), `checkIntervalMin = 60` (day 0 floor).

Post commit, non blocking `after()` work: streak update, FIRST_CLIP_CELEBRATE
growth event (once per user), and `clip_submitted` analytics. None of this touches
the 6 money files; earnings are computed later by the tracking/earnings path off
the frozen CPM + ClipStat views.

### What already exists that the batch REUSES (the owner was right — most of it exists)

• **`POST /api/clips/owner-submit-bulk` (the whole file).** This is a working
  multi link batch: up to 30 links, per link independent try/catch + own
  transaction (one bad link never aborts the rest), bounded concurrency via
  `runWithLimit`, and BL-213's "never silently drop a pasted line" accounting —
  every input line yields exactly one outcome, `summary { submitted, skipped, total }`
  with `submitted + skipped == total`. This outcome model is the batch template.
• **`src/lib/owner-submit-core.ts`** already factors the owner path into
  `validateOwnerSubmitContext` + `processOwnerSubmitLink`. The clipper batch needs
  the SAME shape for the CLIPPER path (see Part 4, step 1).
• **`socialPlatformForUrl`** for auto detecting a link's platform to map it to the
  right connected account.
• **`clip_limit_overrides` + `maxClipsPerUserPerDay`** for the limit — reused verbatim.
• **`isTestUser`** flag (User column, `@default(false)`, propagated JWT -> session
  in `src/lib/auth.ts:675,716,734,745,755,801`) for the test only gate.
• Idempotency, `sanitizeClipUrl`, `normalizeClipUrlForMatch`, CPM freeze helpers.

The gap the owner bulk does NOT cover, and the batch must add: (a) it is OWNER only
and (b) it uses ONE account for all links. The clipper batch needs a PER ROW
account and MULTIPLE platforms in one submit.

## PART 2 — THE BATCH DESIGN

### Shape

New clipper endpoint `POST /api/clips/batch`. Body:

```
{ campaignId: string,
  rows: [ { clipUrl: string, clipAccountId: string }, ... ] }
```

ONE `campaignId` at the batch level (one campaign at a time, enforced by shape — a
row cannot carry its own campaign). Each row is one link + the connected account it
maps to. The client also sends the account so the server never guesses; the server
still re validates platform vs account (RULE 1) so a tampered pairing is rejected.

### Auto mapping a link to the right account (obvious, hard to get wrong)

When the clipper pastes a link into a row, the client calls `socialPlatformForUrl`
locally, detects TikTok / Instagram / YouTube, and:

• If the clipper has exactly ONE approved account on that platform that has joined
  this campaign, it is auto selected and the row shows a locked chip
  "@handle • TikTok (auto)". No decision to make.
• If they have MULTIPLE approved accounts on that platform, the account selector
  pre filters to just those and asks them to pick; the row cannot collapse to
  "ready" until one is chosen.
• If they have ZERO approved+joined accounts on that platform, the row shows a
  blocking error "No approved TikTok account connected to this campaign" with a
  link to Accounts / Join, and the row is excluded from submit.

The account picker only ever offers platform matching, approved, campaign joined
accounts, so a wrong pairing is structurally hard to create; the server RULE 1 / RULE 0
checks are the authoritative backstop.

### The limit, shown live and enforced server side

The header shows a live meter: `Used today {todayCount} / {maxPerDay} • {remaining} left`.
`remaining = maxPerDay - todayCount` where both come from the server (`maxPerDay`
already resolves the override). As the clipper adds rows the meter counts down;
the "Add clip" control disables at `rows.length === remaining` with the note
"You can submit {remaining} more to this campaign today."

Server side (authoritative, NOT UI trust): the endpoint recomputes `maxPerDay` and
`todayCount` at request time, then requires `validRows.length <= remaining`. Rows
are processed SEQUENTIALLY (concurrency 1 for the commit step) so the running
committed total is authoritative and the cap can never be overrun by a race; each
committed clip decrements the live allowance. If, mid batch, the live count would
reach the cap (e.g. a second tab submitted in parallel), the remaining rows return
`ok:false, error:"Daily limit reached"` and are reported as skipped — never a
silent half submit. (Freshness fetches, the slow part, may still run at bounded
concurrency BEFORE the sequential commit; only the cap sensitive create is serial.)

### Collapsible rows (screen space)

Each row has two states:
• COLLAPSED (default once valid): a single line — platform icon, `@handle`,
  truncated URL, and a status chip (Ready / Error / Submitted). Chevron to expand.
• EXPANDED: URL input, the account chip or selector, and any inline validation
  message. A row auto collapses to COLLAPSED the moment it validates clean, and
  auto expands if it later errors, so attention always lands on what needs fixing.

The whole list sits under the campaign header + limit meter; a sticky footer holds
the running "{n} ready, {m} need attention" summary and the single Submit button.

### Identical per clip state (money + tracking unaffected)

Each row is committed by the SAME core the single submit uses (Part 4 step 1), so
every batched clip ends in the exact three row state above (PENDING Clip + first
ClipStat + active TrackingJob + frozen CPM) and the same `after()` side effects.
Because the batch calls the shared core rather than a parallel copy, tracking,
earnings and verification behave identically to a one at a time submit — there is
no second money path to keep in sync.

## PART 3 — TEST CLIPPER ONLY GATING (exact mechanism)

Use the EXISTING `isTestUser` flag — no new flag, role, or table.

• **Server (authoritative):** `POST /api/clips/batch` reads
  `(session.user as any).isTestUser` (already on the session via the JWT
  propagation in `auth.ts`) and returns 404 for anyone whose flag is not `true`,
  BEFORE any work. 404 (not 403) so the endpoint is indistinguishable from
  non existent to a normal clipper — invisible and unreachable. The single submit
  `POST /api/clips` is left byte identical, so every non test clipper's path is
  unchanged.
• **Client:** the batch UI (the "Submit several clips" entry point + the row
  editor) renders only when `session.user.isTestUser === true`. Non test clippers
  see today's single submit form exactly as now.
• **Audience:** 2 users currently carry `isTestUser = true` (of 1178). The owner
  flips the flag on a user row to grant/revoke access; sign off = keep the flag on
  a widening set, and the eventual GA is a one line change of the gate (drop the
  `isTestUser` check) once approved. This mirrors CLAUDE.md's rule: new UI gated
  behind `isTestUser`, every non test user byte identical.

## PART 4 — DESIGN SPEC + ORDERED BUILD + EVERY EDGE CASE

### UX spec (states, copy, interactions)

Row states and the exact inline messages:
• EMPTY: placeholder "Paste a TikTok, Instagram, or YouTube link".
• DETECTING: after paste, a brief "Checking link…" while platform is detected and
  (on submit) freshness is fetched.
• READY (collapsed): "@handle • {Platform}" chip + truncated URL + green Ready chip.
• NEEDS ACCOUNT: "Pick which {Platform} account posted this" with the filtered picker.
• ERROR (expanded, red inline): the precise message per failure (below).
• SUBMITTED (collapsed, locked): green check, URL no longer editable.

Accessibility (baked into the spec, since this is user facing UI): each row is a
labelled group; the collapse control is a real `button` with `aria-expanded` and an
`aria-controls` pointing at the row body; validation messages use `aria-live`
"polite" and are tied to their input via `aria-describedby`; the limit meter is
readable text (not colour only); the status chip pairs an icon with a text label so
Ready/Error is never conveyed by colour alone; focus moves to the first errored row
on a failed submit. (A dedicated accessibility pass on the built component is a
build time gate, per the project's accessibility rule; this design fixes the
semantics up front.)

Collapse behaviour: valid rows collapse automatically; invalid rows stay/expand;
"Collapse all" / "Expand all" in the header; a collapsed errored row still shows its
red chip so nothing hides silently.

2. **Add `POST /api/clips/batch`** (test gated) that: validates `isTestUser`,

   bounded concurrency, then commits each row SEQUENTIALLY through

3. **Build the test gated UI**: campaign picker (single), limit meter, collapsible
   rows, per platform account auto mapping, client side pre validation, sticky

4. **Accessibility + mobile pass** on the component (375px, keyboard, screen reader)
   before sign off.

### Every edge case, each handled so nothing half submits silently

• **Wrong platform link for the row's account** (link is IG, account is TikTok):
  client blocks submit for that row with "This link is Instagram but the selected
  account is TikTok"; server RULE 1 (line 907) returns `ok:false` for that row —
  other rows still commit; the row shows red, is counted as skipped.
• **Account not connected / not approved / not joined**: client shows the blocking
  "No approved {Platform} account on this campaign" and excludes the row; server
  RULE 0 membership (line 821) + the APPROVED account lookup (line 807) reject it as
  `ok:false` if it slips through. Never a 500.
• **Exceeding the limit mid batch**: server counts once, requires
  `validRows <= remaining`, commits sequentially; if the live count hits the cap
  mid run the remaining rows return `ok:false "Daily limit reached"` (skipped, not
  committed). The meter and the up front `rows.length <= remaining` check make this
  rare; the sequential commit makes it impossible to overrun.
• **Partial failure (4 of 5 submit)**: each row is its OWN transaction (as in
  owner bulk), so the 4 good rows are fully committed to the identical per clip
  state and the 1 bad row returns its reason. Response:
  `summary { submitted:4, skipped:1, total:5 }` + `results[]` with each row's
  outcome. The UI marks 4 green Submitted and 1 red with its message; the clipper
  fixes and re submits ONLY the failed row (idempotency on userId+url makes a
  re submit of an already committed row safe).
• **Duplicate links**: within batch — deduped before processing (first wins, the
  rest return "Duplicate of another link in this batch", exactly like owner bulk
  lines 86-93). Against the DB — the campaign scope and user scope `normalizedUrl`
  checks (lines 1057-1081) plus the DB partial unique (`P2002` -> friendly 400,
  line 1256) reject it as `ok:false` for that row only.
• **Transient freshness failure** (Apify/YT blip): same lenient behaviour as single
  submit — the row is accepted, the slot is refunded, cron verifies later; only
  `bad_url` is a hard per row error.
• **Double click / resubmit**: the 60s idempotency cache (userId+url) replays the
  created clip, so a resubmitted batch never double writes.
