## The answer

**HikerAPI is not failing. It resolves everything that is resolvable, and Apify cannot resolve a
single thing HikerAPI cannot.** Probed live against all 1,268 tracked Instagram clips:

| | count | share |
|---|---|---|
| HikerAPI **resolved** (all via `play_count`) | **714** | 56.3% |
| HikerAPI `gone_404` (deleted/private) | **554** | 43.7% |
| HikerAPI resolved something Apify could not | 0 | |
| **Apify resolved something HikerAPI could not** | **0 of 37 tested** | **0%** |

Apify was asked for 37 of the `gone_404` clips. It returned `error: not_found` on **37 of 37**. On 10
clips HikerAPI resolved, Apify agreed on the view count **exactly, 10 times out of 10**
(2898=2898, 7743=7743, 1451=1451, …).

**So the "~669 Instagram clips a week that HikerAPI cannot resolve and Apify then resolves" is not a
real population. It is a measurement artifact.** Nothing was ever falling through, because the clips
Hiker returns `gone_404` for are genuinely gone and the code already skips Apify for them at $0
(`apify.ts:1351-1354`, `[SKIP-APIFY] reason=gone-404`).

## Where the artifact comes from (two defects, same family)

The ledger cannot see HikerAPI's work, so HikerAPI's successes are billed to Apify on paper.

**Defect 1 — `clipCount` counts the whole batch, not what Apify did.**
`tracking.ts:3853-3857` writes `clipCount: count`, where `count` is `globalBatchPlatformCounts[plat]`:
the number of clips **sent into** the batch, before the Hiker fan-out filters it. So a tick where
Hiker resolved 700 of 800 clips still records **800** clips against the Apify provider label.

**Defect 2 — `summarizeBatchProvider` has no Hiker branch (`apify.ts:1935-1958`).**
It only reads `apidojoHits` and `legacyHits`. Hiker's hits are invisible to it. This is **the same
defect the brief flags for LamaTok**, in the same function, and it is confirmed:

```ts
if (s.apidojoHits === 0 && s.legacyHits === 0) return noneTag;   // LamaTok-only batch -> "none-tiktok"
```

A batch served entirely by LamaTok has `apidojoHits === 0 && legacyHits === 0`, so it returns
`none-tiktok`. **Confirmed exactly as BL-549 described.** The IG side has the identical hole for
Hiker. The fix for both is one shape: `PlatformProviderStats` needs `hikerHits` and `lamatokHits`
fields, `summarizeBatchProvider` needs branches for them, and `clipCount` must record
`urlsForApify.length` rather than the pre-filter chunk size. **Specified, not built.**

**Consequence:** the "$77/month of Apify IG" in the ledger is an over-attribution of unknown size.
The `estimatedCostUsd` is derived from `callCount` (chunk count), so the dollar total is closer to
right than the clip attribution, but no number derived from `provider` + `clipCount` can be trusted
until Defects 1 and 2 are fixed.

## Why STEP 1 could not be answered as briefed

The brief asks to pull the failing clips and record "HikerAPI's recorded classification" per clip.
**That data does not exist.** Verified:

* `apify_usage_entries` has **no `clipId`** (12 columns, confirmed against `information_schema`). It
  records counts per batch, never which clip.
* **No `classification` column exists anywhere in the schema.** `hikerapi.ts` builds a
  `classification` on every `HikerResult` in memory and **never persists it**; `hikerapi.ts` performs
  no scraper-related DB writes at all.
* The fan-out numbers that would answer this (`hiker_hits`, `hiker_misses`, `gone_skipped`,
  `budget_abandoned`) are emitted **only as a `console.log`** at `apify.ts:1366`. They live in Railway
  logs and are unqueryable from here.

So I answered it by **direct measurement instead**, which is stronger than the ledger would have been:
probe all 1,268 live clips and observe the truth.

## The one thing I could not rule out

**The 200-second budget abandonment.** `apify.ts:1284-1341`:

```ts
const TICK_HIKER_BUDGET_MS = 200_000;
const tickStart = cronTickStartMs ?? Date.now();       // CRON TICK START, not fan-out start
if (Date.now() - tickStart > TICK_HIKER_BUDGET_MS) {
  return { idx, budgetAbandoned: true };               // -> missingForApify, Hiker never asked
}
```

The budget is measured from the **cron tick start**, not from when the IG fan-out begins. Work done
earlier in the tick eats the budget, and any IG clip reached after 200s is handed to Apify **without
HikerAPI ever being called**. Those clips are not Hiker failures; they are clips Hiker never saw.

**Measured relevance:** my sweep of all 1,268 clips at HikerAPI's concurrency of 10 took **137.5s**,
which is inside the 200s budget — so a dedicated fan-out would finish. Whether a production tick
reaches the IG fan-out with >137s of budget left is a Railway-log question I cannot answer from here,
and I will not guess. **`budget_abandoned` in the `[SCRAPER-DECISION]` log line is the one number
that settles it, and it is not persisted.**

Also worth noting: `HIKER_GLOBAL_CONCURRENCY = 10` while HikerAPI's own `/sys/balance` reports
`rate: 15`. We are self-limiting to two thirds of the allowed rate, which lengthens the fan-out and
makes budget abandonment more likely, for no stated reason.
