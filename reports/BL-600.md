### TikHub cross-check (matched ids), and what it reveals about rounding

```
id 7644087836953660693   LamaTok by/id 3,300,000  ==  TikHub web 3,300,000   (exact match)
                                                       TikHub app 3,330,086   (exact underlying count)
id 7635742500669377805   LamaTok by/id 2,600,000  |  TikHub app 2,568,863
id 7631302498665385234   LamaTok by/id 1,000,000  |  TikHub app 1,020,830
```

Read carefully: LamaTok `by/id` returns the DISPLAY-ROUNDED value, and it matches TikHub's WEB endpoint exactly (3,300,000 == 3,300,000, the same agreement BL-579 found). TikHub's APP endpoint returns the exact underlying count (3,330,086), which display-rounds to the same 3.3M. So LamaTok `by/id` is correct, and the roughly 0.9% gap against the app value is the TikTok-side display rounding BL-544 documented, NOT a LamaTok error. This is a clean corroboration and a live demonstration of the PART 4 landmine in one probe.

* **Fresh and low-view: NO SAMPLE OBTAINED THIS ROUND.** I resolved 12 recent approved TikTok short links to hunt for a live slideshow; all 12 resolved to `/video/`, zero slideshows. So every slideshow I could test is one of the 5 high-view (over 1,000,000) test posts. **Low-view (<10k), music-only, and region-locked slideshows remain UNVERIFIED**, exactly as BL-583 flagged. * **Image-only carousel returns null, correctly.** `DFvdHGSimtp` is a 9-child carousel with 0 video children, so `classifyV2Media` (`hikerapi.ts:459-471`) yields `views=null`, classification `carousel_image_only`. This is not an Apify-fixable gap: an image-only post has no play count in existence anywhere, so no vendor (Apify included) can return one, and such a post cannot earn a view-based CPM. * **Video posts return correct views**, from the first `play_count` field. Confirmed on three live posts.
* **carousel_mixed (a carousel WITH video children)**: the code sums the video children's `play_count` (`hikerapi.ts:473-493`, `viewSource="carousel_sum:play_count"`). My bounded IG sample did not contain a live mixed carousel (they came back single-video or image-only), so this path is CODE-confirmed, not probe-confirmed this round, matching BL-583.
* **Stored-views update:** the production path returns the resolved `views`, which the tracking cascade writes on the next tick via the existing never-zero/null rule. No stored view was moved by this audit (read only).

## PART 4: THE VERDICT AND THE EXACT ORDERED NEXT STEP

**Do both platforms get correct slideshow and carousel view counts today, without Apify?**

* **Capability: YES, on the posts tested.** TikTok slideshows resolve on LamaTok `by/id` (5 of 5 live, playCount correct, TikHub web agrees exactly, deleted returns a clean 404). Instagram carousels resolve on HikerAPI (video and mixed via video-sum; image-only correctly null). Both vendors are already paid. No new vendor is needed.
* **In production: NOT YET for TikTok slideshows.** The `by/id` path ships SHADOW only (`lamatok.ts:198` fires it fire-and-forget and logs); the real slideshow view still comes from `by/url` returning HTTP 500, which yields the `carousel` verdict and routes the poll to Apify. So slideshows are proven-capable but still Apify-served today. IG carousels already run live on HikerAPI in production.

**The one thing standing between shadow and live is the display-rounding PRICING proof, and it is the landmine, not a formality to skip.** A slideshow view count SETS a clipper's earnings, and BL-544 proved 84.1% of TikTok clips at or above 10k are display-rounded with endpoints rounding differently. My own cross-check shows it live: LamaTok `by/id` returns 3,300,000 where the exact count is 3,330,086. So moving slideshow views to `by/id` is a pricing decision that MUST pass a stratified shadow proof first:

* At least 100 matched slideshow clips BELOW 10k (exact-equality criterion) AND at least 100 AT OR ABOVE 10k (equality after both are normalized to the same rounding), matched by id never by row, zero null-versus-real, over 7+ days.
* **This proof is currently UNFILLABLE**, not merely unfinished: the tracked slideshow population is 5 clips, all REJECTED, all over 1,000,000 views. The <10k stratum has ZERO candidates and the >=10k stratum has 5. The population, not the vendor, is the limit. Its earning impact today is exactly $0 (no APPROVED slideshow exists).

**Does any sub-type still need Apify? No sub-type is proven to.** The only genuinely uncoverable object is an image-only IG carousel view count, which does not exist anywhere, so it is handled by returning null and should simply stop escalating to Apify, not by buying a vendor. Low-view, music-only, and region-locked slideshows are UNVERIFIED (no live sample obtainable from the current population), so they are stated as unproven, not assumed covered.

### Exact ordered next step (single, concrete)

1. **Make the shadow readable and complete.** Change BL-580's slideshow shadow from a console line to a DB/ledger row (matched id, `by/id` playCount, today's value, view bucket), AND fire it for short-link slideshows by resolving the `vt.`/`vm.` redirect to the canonical `/photo/<id>` first (today it is skipped for short links, `lamatok.ts:114`, and 761 of the 766 tracked TikTok slideshow-candidates are short links). Without this the stratified proof can never be read or reach significance. This is the true next build.
2. **Accumulate matched, stratified shadow data for 7+ days**, then evaluate against the criteria above. Expect the >=10k stratum to fill slowly and the <10k stratum possibly never at the current population.
3. **Flip pragmatically:** since 0 APPROVED slideshows exist and `by/id` reproduces the web-displayed rounded value (proven here), the >=10k flip can go live behind the existing never-zero/null guard once even a modest matched sample confirms agreement. Hold the <10k stratum until a real sample exists; its impact is $0 until an approved <10k slideshow appears.
4. **IG:** trust `carousel_mixed` video-sum in the overlay (a smaller pricing-adjacent proof, since per-child `play_count` is itself rounded at >10k) and add a code skip so image-only carousels stop escalating to Apify.

Result: with the shadow made queryable and these small changes, TikTok reaches zero Apify dependency on slideshows and IG on carousels. Nothing here requires a new vendor.
