## PART 1 — the fix

**One shared guard, three callers.** The BL-541 ambiguity test lived inline in agency-monitor. I extracted it VERBATIM into `src/lib/owner-share-guard.ts` (`decideOwnerGross`) and refactored agency-monitor to call it, so there is now a single source of truth. `decideOwnerGross` returns one of:

* **`gross`** — guaranteeOwnerSplit and the clip's stamped `oCpm/cCpm` agrees with `s/(1-s)`: amount is BL-162 gross off the clipper's base PLUS bonus.
* **`ambiguous`** — guaranteeOwnerSplit but a rate-era mismatch (somesome): the caller must NOT apply gross.
* **`not_guaranteed`** — flag off / degenerate s: the caller uses its legacy path.

**agency-monitor proven byte-identical.** Read-only run before and after the refactor: `checked=3146 violations=191`, STRAENGE 0, somesome 186 — identical. The extraction changed no behaviour.

**gamification.ts:871** now: `gross` → the gross amount; `ambiguous` or `not_guaranteed` → the exact pre-BL-563 base-only value. So STRAENGE gets gross and **somesome keeps base-only, byte-identical**, never a wrong-era gross. The downstream budget-cap and payout-reduction-cap logic is untouched.

**force-recalc:359** got the same treatment. PART 2 requires the credit to survive a force-recalc run, and it used base-only. It now uses `decideOwnerGross` off the post-cap clipper gross `newEarnings`, so an admin running force-recalc on STRAENGE writes gross, not base-only.
