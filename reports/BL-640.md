# BL-640 — Can a clipper see or take anything that is not theirs?

## VERDICT (one line): NO. Every clipper-facing route scopes to the session user, no route trusts a client-supplied id for authorisation, owner economics stay stripped, and the payout path is race-proof at four independent layers; the three items worth fixing before growth are an alternate cron auth header, a per-instance rate limiter, and wallet addresses still stored in plaintext.

**2026-07-23 · AUDIT ONLY. READ ONLY on code, data, schema, config and money.** Nothing changed, fixed, pushed or flipped. No payout created or altered. No destructive or state-changing action. No real clipper's data read beyond aggregate counts. Every handle, email and wallet address is redacted; no working exploit payload appears here.
**Base:** origin/main `9f5de583` · **Branch:** checkpoint/BL-640 · isolated worktree, no node_modules junction. Live reads via the sanctioned read-only runner. A markdown-only diff cannot change tsc or build, so none was run and none is claimed.

## PART 1 — HORIZONTAL: can clipper A reach clipper B's data?
**No route trusts a client-supplied id for authorisation. Count of clipper-facing routes that do: ZERO.**

The codebase uses one consistent and correct pattern: fetch the row by the client's id, then verify ownership against the session before returning anything, usually answering 404 rather than 403 so a stale URL cannot confirm that a hidden row exists.

| Surface | Scoping evidence |
|---|---|
| Marketplace strike dispute | `marketplace/strikes/[id]/dispute/route.ts:60` refuses when `strike.userId !== userId` |
| Marketplace chat thread | `marketplace/chat/threads/[id]/route.ts:70` 404s unless the viewer is userA or userB |
| Marketplace submission | `marketplace/submissions/[id]/route.ts:97` gates on isCreator or isListingOwner, 404 otherwise |
| Platform chat | `chat/conversations/[id]/messages/route.ts` defers to `canAccessConversation` (`chat-access.ts:78`), a participant lookup on the composite key |
| Own accounts | `accounts/[id]/route.ts:33` `findFirst({ where: { id, userId: session.user.id } })` |
| Own clips, payouts, earnings, notifications | all keyed `userId: session.user.id` |
| `clips/[id]` DELETE | OWNER only, not a horizontal surface |
| `clips/[id]/tracking` | `requireRole(["ADMIN","OWNER","CLIENT","REVIEWER"])`, CLIPPER excluded entirely |

**BL-615 batch draft: a draft cannot be read or overwritten by another clipper.** All three verbs key the composite `userId_campaignId` with the session id (`clips/batch/draft/route.ts:43`, `:69`, and DELETE via `deleteMany` with `userId` at `:87`). The client-supplied `campaignId` is only ever one half of the key, so it can never address another user's row. Structurally impossible, not merely unlikely.

**BL-618 removed the isTestUser gate; what remains still scopes to the caller.** `clips/batch/route.ts:116` resolves accounts with `where: { id: { in: ... }, userId }`, so an unowned id simply is not found. Each row then passes through the shared core, which independently requires the account be owned and APPROVED (`clipper-submit-core.ts:74`) and requires campaign membership through one of the caller's own accounts (`:78` to `:81`). Opening the gate widened who may call it, not what any caller can reach.

## PART 2 — VERTICAL: can a clipper reach an owner, admin or reviewer surface?
**No. A mechanical sweep of every route under `api/admin` and `api/marketplace/admin` for a missing role gate returned ZERO hits.** Nothing relies on a hidden button.

Authorisation is centralised in `src/lib/auth-guards.ts` and **fails closed**: `requireOwnerOrCapability` returns 403 when the DB is unavailable (`:148`) and again inside a bare `catch` (`:164`), so a surface whose capability could not be verified is never opened.

Spot-verified money-sensitive routes, all OWNER gated server-side: force-recalc (`admin/force-recalc-earnings/route.ts:43`), fix-earnings (`:56`), fix-budget (`:54`), payout adjust (`admin/payouts/[id]/adjust/route.ts:43`), payout approval (`payouts/[id]/review/route.ts:17`), track-all, reset-data, export, 1099 report. Agency monitor reads via `requireOwnerOrCapability("EARNINGS_VIEW")`. Campaign DELETE is OWNER (`campaigns/[id]/route.ts:876`); campaign PATCH admits ADMIN but scopes them to campaigns they created or were assigned (`:198` to `:213`) and routes their changes into a pendingEdit awaiting owner review (`:266`), so an ADMIN cannot silently edit a campaign at all.

**Cron:** all 12 routes require `CRON_SECRET` and fail closed when it is unset. Two accept an alternate header, which is finding 1 below.

`dev-auth` is double-gated and cannot exist in production: a `NODE_ENV === "production"` 404 at `dev-auth/route.ts:12` plus `isDevBypassEnabled()`, which itself requires non-production.

## PART 3 — can a clipper influence their own money?
**No client-supplied value reaches an earnings or payout calculation.** Traced through `src/lib/clipper-submit-core.ts`, the single path both submit routes use.

| Attack | Result | Evidence |
|---|---|---|
| Choose their CPM | **No.** CPM is never read from the request; it is derived server-side from the campaign row and stamped immutably at submission | `:173` to `:178` |
| Choose a view count | **No.** Views come from the provider fetch, never the body; a null resolve skips the first snapshot rather than writing a fabricated 0 | `:138`, `:197` |
| Choose a timestamp | **No.** Freshness compares provider `createdAt` against server `Date.now()` | `:148` to `:153` |
| Submit to an ineligible campaign | **No.** Membership required through the caller's own accounts, plus DRAFT, COMPLETED, PAST, PAUSED, archived, test and over-budget refusals | `:80`, `:93` to `:103` |
| Use an account they do not own | **No.** Account must be owned and APPROVED | `:74` |
| Beat the daily limit via the API | **No** (one narrow race, finding 6). Enforced server-side, independent of the UI | `:111` to `:114`, and again at `clips/batch/route.ts:140` |
| Force-approve or write earnings | **No.** The clipper core writes no earnings and cannot approve; status defaults to PENDING | file header, `:182` |

**Can they create a payout for more than their balance, or twice for the same balance by racing? No, on four independent layers.** Serializable isolation (`payouts/route.ts:601`), an in-transaction duplicate check refusing any second open payout per user per campaign (`:388`), a DB partial unique index as backstop, and a 3 per hour rate limit (`:255`). The over-balance gate compares integer cents against `effectiveCap = min(campaign available, global available)` (`:508`), so being over-paid on one campaign cannot fund a withdrawal on another. **Verified live: `uq_payout_open_per_user_campaign` is present on the production `payout_requests` table.** All three payout creation paths carry the BL-556 server-side method rule, including the currently dead server action, which fails closed by construction.

## PART 4 — what leaks in responses
**BL-531's strip is intact and still applied at all four clipper campaign routes** (`campaigns/route.ts:226`, `campaigns/[id]/route.ts:86`, `campaigns/past/route.ts:73`, and `campaigns/spend/route.ts:164` for the spend figure). `OWNER_SIDE_FIELDS` still drops `lockedOwnerShareDecimal`, `ownerCpm`, `ownerCpmDecimal`, `agencyFee`, `agencyFeeDecimal` and the three per-platform owner CPMs, so the owner's split is not derivable from a clipper payload.

Other classes checked: **bot and fraud signals are stripped, not leaked.** `clips/mine/route.ts:118` to `:127` destructures out every fraud, bot-alert, freeze and override field before responding, and masks FLAGGED as PENDING. The fields do appear in `clips/route.ts:413`, but that GET refuses CLIPPER outright (`:91`). BL-599's peer-relative rank exists only on the admin fraud-review page, with no clipper-facing API. **Wallet addresses** are returned only by the OWNER-gated payouts GET. **Emails** are selected in several routes but every occurrence feeds a server-side send and is never returned in a response (verified at `chat/conversations/[id]/messages/route.ts:391` and `community/channels/[id]/messages/route.ts:443`). **Other clippers' handles** appear only on deliberate leaderboards (finding 7).

## PART 5 — the basics that matter at scale
Rate limiting exists on login, magic-link, signup, account add, verification, clip submit, batch submit, payouts, campaign edit and chat. Two structural weaknesses at scale are findings 1 and 2. Verification additionally carries a **DB-backed per-account daily cap** that survives restart (`accounts/[id]/verify/route.ts:677`), which is the correct pattern and directly addresses the BL-564 retry behaviour. Mass account creation is bounded mainly by Discord OAuth requiring a real Discord account. **No secrets are logged**: every match is a secret name or a "not configured" warning, never a value. Cron endpoints are unreachable without the secret except via finding 1.

### Must fix before growth
Findings **1, 2 and 3**. One is an unauthenticated action on a live email path, one is an abuse control that degrades exactly as the platform scales, and one is unencrypted payout addresses for every clipper who has ever withdrawn.

### Hardening, not urgent
Findings **4, 5, 6 and 7**. None is exploitable by a clipper today; 4 and 5 are guards against a future edit reopening a closed hole, 6 is bounded by the budget cap, and 7 is a product choice.

### The platform is genuinely sound on the questions asked
Stated plainly rather than padded: on the four things the brief worried about most, the evidence is that they are closed. A clipper cannot read another clipper's data, because every route scopes to the session and none trusts a client id. A clipper cannot alter their own earnings, because CPM, views and timestamps are all server or provider derived and the clipper core writes no earnings at all. A clipper cannot reach an owner surface, because zero admin routes lack a server-side role gate and the shared guard fails closed. A clipper cannot double-spend a balance, because four independent layers stop it and the DB backstop index was verified present on production.

### What could not be measured, and why
• **Whether finding 1 is live** depends on whether the Railway origin is reachable from the internet, which is an infrastructure fact not visible in the repo. Confirming it by request would have fired a real marketing sweep, so it was deliberately not probed. Treat it as live until the ingress is checked.
• **Runtime behaviour of the marketplace paths**, which BL-617 recorded as structurally unexercised at zero submissions, so their scoping is read from code rather than observed.
• **The exact instance count** behind the rate limiter, which sets how much finding 2 actually weakens each limit.
