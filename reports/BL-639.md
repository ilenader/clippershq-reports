## VERDICT (one line)

**The site is fine for 10x on almost every axis, and the database is genuinely well indexed for clipper traffic. Three real problems are worth fixing before thousands of phones arrive, and all three are small: recharts costs every clipper 110 KB gzip on the earnings page, `/api/campaigns/spend` runs two unbounded whole-table aggregates on every campaign-list load, and every clipper page shows a blank spinner instead of content while five requests finish.** Nothing here needs a rewrite.

### The heavy library on a clipper page: recharts

**`/earnings` carries a 380.9 KB raw / 110 KB gzip chunk that `/clips` does not.** That single chunk is **41% of the earnings page's entire 269 KB gzip download**. It is shared byte-for-byte with `/admin/analytics` and `/referrals`, and it contains recharts (confirmed both by string match inside the emitted chunk and by the import chain).

Import chain, static, **not** lazy:
```
src/app/(app)/earnings/page.tsx:13
  -> src/components/earnings/EarningsChart.tsx:4
    -> src/components/ui/area-gradient-chart.tsx:4-6
       import { ResponsiveContainer, AreaChart, Area, XAxis, YAxis, Tooltip, CartesianGrid } from "recharts"
```

**The whole `src/` tree contains exactly one `next/dynamic`**, `VoiceRoom` in `src/components/layout/app-layout.tsx:40`. Nothing else is code-split, so every clipper downloads the chart engine before seeing a single number, even if they never scroll to the chart.

Transfer-time arithmetic from the measured 269 KB (JS only, ignoring HTML, images and API responses): about **1.4 s on slow 4G** (~190 KB/s) and about **5.4 s on a 400 Kbps connection**. **UNMEASURED:** real device timings (Largest Contentful Paint, Time to Interactive, main-thread parse cost on a low-end phone). Measuring that needs a Lighthouse mobile run or a WebPageTest profile on a real handset, which is a load-generating action and out of scope here.

### Requests on first load

Every clipper page is `"use client"` and fetches after hydration. None is server-rendered with data.

| Page | Fetches on mount | Parallel? | Extra |
|---|---|---|---|
| `/clips` | **5** (`/api/clips/mine`, `/api/campaigns?status=ACTIVE`, `/api/accounts/mine`, `/api/campaign-accounts`, `/api/gamification`) | one `Promise.all` (`clips/page.tsx:136-142`) | **plus one POST per clip card** (below) |
| `/campaigns` | **4** (`/api/campaigns`, `/api/campaigns/spend`, `/api/campaign-accounts`, `/api/campaigns/past`) | `Promise.all` (`campaigns/page.tsx:38-43`) | none |
| `/earnings` | **3** (`/api/earnings`, `/api/clips/mine`, `/api/analytics/views-by-day`) | 2 in a `Promise.all` (`:57-60`) + 1 separate effect (`:114-138`) | 3 requests per filter tap |
| `/progress` | **2** (`/api/gamification`, `/api/gamification?leaderboard=true`) | `Promise.all` (`:70-73`) | none |

On top of that, on **every** page, from the app shell: an **Ably realtime WebSocket** (`app-layout.tsx:236`) plus its `/api/ably-token` fetch, a **`/api/presence/heartbeat` POST every 30 s** (`:248-255`), and a **`/api/notifications/count` poll every 60 s** (`navbar.tsx:196-211`).

**A client-side request fan-out worth naming.** `src/components/clips/ClipCardNew.tsx:157-188` fires `POST /api/clips/{id}/thumbnail` **once per rendered card** that has no stored `thumbnailUrl`, deduped only by a `sessionStorage` key. A clipper opening `/clips` for the first time with N uncovered clips issues **N parallel POSTs**, each doing a DB lookup plus a provider scrape. The server rate limit is 40/min/user (`thumbnail/route.ts:43`), so **a clipper with more than 40 uncovered clips hits the limit on their own first page view**.

**Caching:** `/clips` requests all carry `cache: "no-store"` and a cache-busting `_t` timestamp (`clips/page.tsx:137-141`), so a phone on flaky data re-downloads every payload every time, and `useAutoRefresh(load, 60000)` (`:164`) re-fires all five **every 60 s and immediately on every tab re-focus**. `/api/campaigns/spend` is `force-dynamic` (`:5`), so it is never cached either.

## PART 2 — what is slow on the server?

### Real timings (EXPLAIN ANALYZE, live data, remote Supabase over the pooler)

| # | Query | Plan | **Measured** |
|---|---|---|---|
| A | `/api/campaigns/spend` clip aggregate | **Seq Scan on clips** | **2.3 to 4.2 ms** |
| B | `/api/campaigns/spend` agency aggregate | **Seq Scan on agency_earnings** | **1.3 ms** |
| C | `/api/earnings` clips for one busy clipper | Bitmap Index Scan `clips_userId_postedAt_idx` | 2.9 ms |
| D | withdrawal-gate earnings basis, one clipper | Bitmap Index Scan `clips_userId_postedAt_idx` | 0.4 ms |
| E | clips list, one clipper, newest first | Index Scan Backward `clips_userId_createdAt_idx` | 9.2 ms |
| F | **latest ClipStat per clip, busy clipper (237 clips)** | nested loop, **237 loops** on `clip_stats_clipId_idx` | **474 to 1,092 ms** |
| F3 | same pattern, 151-clip clipper | nested loop, 151 loops | 63 ms |
| G | campaigns list (28 rows) | Seq Scan on campaigns | 0.08 ms |
| H | unread notifications, one clipper | Index Scan `notifications_userId_isRead_createdAt_idx` | 1.2 ms |

**Slowest thing today: F, the per-clip latest-view lookup, at up to ~1.1 s for the heaviest clipper.** This is the shape `/api/clips/mine` uses (`clips/mine/route.ts:60-79`, `stats: { orderBy: { checkedAt: "desc" }, take: 1 }`) and that endpoint is fetched by **both** `/clips` and `/earnings`. **Honest caveat:** I measured the *pattern* (`DISTINCT ON` over the clipper's clips); the exact SQL Prisma emits for a nested `take: 1` include was not captured, so the precise Prisma cost is **UNMEASURED**. Capturing it needs Prisma query logging (`log: ["query"]`) against a real request. The cost clearly scales with the individual clipper's clip count: 63 ms at 151 clips, ~474 ms at 237.

### Indexes: genuinely good, and that is the main reason this scales

Every clipper-facing read is scoped by `userId`, and `userId` is well covered: `clips_userId_idx`, `clips_userId_createdAt_idx`, `clips_userId_postedAt_idx`, plus `notifications_userId_isRead_createdAt_idx`, `payout_requests_userId_idx`, and three `clip_stats` indexes including `(clipId, checkedAt DESC)`. With ~3.6 clips per clipper (4,132 clips over 1,155 clippers), a per-user query stays small no matter how large the table grows, **provided the index is used**. Plans C, D, E and H confirm it is. **No missing index was found on the columns clipper pages actually filter and sort by.**

### The real scale problem: two unbounded whole-table aggregates

`src/app/api/campaigns/spend/route.ts` runs, on **every** `/campaigns` load, for every clipper:
* `:63-71` `db.clip.groupBy({ by: ["campaignId"], where: { status:"APPROVED", isDeleted:false, videoUnavailable:false }, _sum:{ earnings:true } })` with **no campaign filter and no `take`** (whole `clips` table).
* `:94-97` `db.agencyEarning.groupBy({ by: ["campaignId"], _sum: { amount: true } })` with **no `where` at all** (whole `agency_earnings` table).

Both are Seq Scans (A and B above). They are invisible today at 4,132 clips. To project honestly I measured the **real seq-scan cost on this same database at larger row counts**:

| Table scanned | Rows | **Measured full aggregate** |
|---|---|---|
| clips (today) | 4,132 | **2.3 ms** |
| apify_usage_entries | 69,659 | **1,592 ms** |
| clip_stats | 162,669 | **733 ms** (parallel, 2 workers) |
| email_events | 377,532 | **2,440 ms** (parallel) |

**Caveat stated plainly:** these are different tables with different row widths and Postgres switches to a parallel plan above roughly 100k rows, so this is an order-of-magnitude curve, not a per-row constant for `clips` specifically. Measuring the exact curve for `clips` would require seeding a copy of the table at 10x and 100x, which is a data write and out of scope. What the curve does establish, unambiguously, is that **a full-table aggregate on this database moves from single-digit milliseconds at ~4k rows into the hundreds of milliseconds to multiple seconds somewhere between ~70k and ~380k rows.**

### N+1 and sequential-waterfall findings

* **`GET /api/gamification` is the worst offender on the critical path.** `gamification/route.ts:96-108` into `lib/gamification.ts:518-628` runs **roughly 8 to 11 sequential round-trips** on a cache miss: `user.findUnique`, a possible `user.update`, `await updateStreak(userId)` (**a write inside a GET**), a second `user.findUnique`, a `clip.aggregate`, a possible third `user.update`, `loadConfig`, `getStreakDayStatuses(userId, 3)` (another `findUnique` plus `findMany`), then the route calls `getStreakDayStatuses(userId, 60)` again and `user.findUnique` a **third** time for the timezone. It is one of the five blocking fetches on `/clips` and it re-runs **every 60 s**. It is not a per-item N+1, but it is a long serial chain where a `Promise.all` and one shared `user` read would do.
* **`/api/analytics/views-by-day`** (`:167-195` and `:198-215`) runs two **sequential** `$queryRawUnsafe` statements with **no LIMIT and no date window**, fetching every earning clip the clipper has ever had plus all their `clip_stats` rows, then applies the `days` window only in a JS zero-fill loop (`:261-269`). It re-fires on every timeframe or campaign-filter change on `/earnings`.
* **The genuine N+1 is on the client, not the server:** the per-card thumbnail POST fan-out described in PART 1.
* **No query-inside-a-loop was found** in `/api/clips/mine`, `/api/earnings`, `/api/campaigns` or `/api/campaigns/spend`. The `.map` blocks in those routes are pure JS. The `findMany` then `findMany` steps (`earnings/route.ts:197-200`, `campaigns/route.ts:239-242`) are batched single `IN` queries, which is the correct shape.

### Missing `take` limits (unbounded `findMany`)

| Route | Line | Bounded by? |
|---|---|---|
| `campaign-accounts/route.ts` | `:31` | user scope only, no cap |
| `accounts/mine/route.ts` | `:26` | user scope only, no cap, **and no `select`** |
| `earnings/route.ts` | `:197` | campaign id list |
| `campaigns/route.ts` | `:239` | campaign id list |
| `gamification.ts` | `:417` | date window only |

### Payload weight, not just query time

`/api/clips/mine` uses **`include`, not `select`** (`clips/mine/route.ts:60-79`), so **every Clip scalar** is fetched and serialized, then 14 fields are discarded in JS at `:117-126`. With `take: 5000` that is up to 5,000 full rows on the wire, and this endpoint is loaded by **both** `/clips` and `/earnings`. `campaigns/route.ts:131` and `accounts/mine/route.ts:26` have the same `include`-everything shape. Worse, `accounts/mine` returns `profileImageUrl`, which that route's own comment (`:30-33`) says can be a **base64 data URI averaging ~41 KB and reaching ~253 KB**, and the same field is selected per clip account inside `/api/clips/mine`.

## PART 3 — the mobile experience, honestly

**Does a page block on a slow request before showing anything? Yes, all four.** Each clipper page returns a bare spinner and nothing else until its fetches resolve:

| Page | Blocking line | What the user sees |
|---|---|---|
| `/clips` | `clips/page.tsx:265-271` | 8x8 spinner, blank page, until **all five** fetches resolve |
| `/campaigns` | `campaigns/page.tsx:73-79` | bare spinner |
| `/earnings` | `earnings/page.tsx:164-170` | bare spinner on first load |
| `/progress` | `progress/page.tsx:107-109` | bare spinner |

There are **no skeleton screens**. On a bad connection the slowest of the five `/clips` requests gates the entire page, and because the code does a second `Promise.all` over the five `.json()` bodies (`clips/page.tsx:143-149`), the slowest response also gates parsing of the other four. `/earnings` is better on **re**fetch (it keeps stale content) but still blanks on first load.

**Scroll jank in the submit modal.** `SubmitClipPremium.tsx:237-238` attaches `wheel` and `touchmove` listeners to `document` with `{ passive: false, capture: true }` and calls `preventDefault()` (`:235`). Non-passive touch listeners force the browser to wait for JavaScript before it can scroll, which is exactly the pattern that feels sticky on a low-end Android. It is active the whole time the modal is open.

**Images.** The campaign picker renders raw `<img src={img}>` with **no width, no height and no `loading="lazy"`** (`SubmitClipPremium.tsx:532`), and `next/image` is not used. That means layout shift as each thumbnail lands, and full-size downloads on a phone.

**Narrow screens.** The batch rows were built mobile-first in BL-622 (one tight line per row, reflowing to a full-width link under 640px) and BL-615's accessibility review covered focus handling, live regions and per-row error wiring. On the code, the flow is sound at 375px. **UNMEASURED:** actual rendering and one-handed reach on a real device. Confirming it needs a physical phone or a device-lab session; a desktop viewport resize is not the same test, and I did not run one.

**What happens to a half-filled batch if the connection drops?** **It survives, and BL-615's server-side drafts do cover this case.** The draft is stored server-side in `batch_submit_drafts` via `/api/clips/batch/draft` (GET load, PUT save, DELETE clear), autosaves debounced while editing (`BatchSubmitSection.tsx:221`), loads on campaign select (`:187`), and clears only on explicit cancel (`:232`) or successful submit. Because it is server-side rather than `localStorage`, it also survives a cleared cache and a device switch. On restore every row is **re-validated** against current campaign platforms, current connected accounts and the current remaining limit, and anything stale is flagged amber and blocks submit rather than being silently sent.

**Two honest qualifications on that.** First, the autosave is debounced, so keystrokes typed in the final moments before the connection drops may not have reached the server; the exact debounce window and worst-case data loss is **UNMEASURED** (it needs a throttled-network test against the draft endpoint). Second, and more important for this audit's framing: **the batch UI is still gated to real `isTestUser`, which is exactly three accounts** (two test clippers plus the owner, per BL-615). So the batch flow is not yet part of the experience the incoming thousands will get, and the single-submit path is what actually matters for them today.

## PART 4 — what breaks first on the clipper side

BL-628 named the tracking cron as the platform's first wall and it has been raised. **The next wall on the clipper-facing side is `/api/campaigns/spend`.**

**Why it is next:** it is the only clipper-facing query that is **not** scoped to the user. Everything else on these pages is `WHERE userId = me` against a good index, so it stays flat as the platform grows. The two `groupBy` calls in `spend/route.ts:63` and `:94` scan the **entire** `clips` and `agency_earnings` tables, uncached (`force-dynamic`), **once per `/campaigns` page view**. Cost therefore grows with total platform size **and** with the number of clippers viewing, at the same time.

**The number.** At today's ratio of ~3.6 clips per clipper:

| Clippers | Clips (projected) | Full-table aggregate, from the measured curve |
|---|---|---|
| 1,155 (today) | 4,132 | **2 to 4 ms**, invisible |
| ~5,000 | ~18,000 | tens of ms, still fine |
| **~10,000** | **~36,000** | **hundreds of ms, on every campaign-list load** |
| ~20,000 | ~72,000 | ~1 s or worse (measured 1,592 ms at 69,659 rows) |

**So it bites at roughly 10,000 clippers / ~36,000 clips, and it is painful by ~20,000 clippers / ~70,000 clips.** That is squarely inside the owner's stated one-to-two-month growth plan. It is also the compounding case: 10,000 clippers do not just make the scan bigger, they run it far more often.

**The runners-up, with their numbers:**
* **`/api/clips/mine` payload.** `take: 5000` with `include` (whole rows). Unbounded in practice per clipper; a clipper with 237 clips already pays ~474 ms for the nested latest-stat lookup alone. Bites hardest for the heaviest clippers, not the average one.
* **The thumbnail fan-out.** Bites at **>40 uncovered clips for a single clipper**, today, because that is the server rate limit. Already reachable: the busiest clipper has 237 clips.
* **Unbounded list rendering.** `ClipsPremium.tsx:173`, `CampaignsRedesign.tsx:293/306/321` and `EarningsPremium.tsx:180` render every row with no pagination or virtualization. On a low-end phone this is a DOM-size and main-thread problem well before it is a database problem, roughly in the **low hundreds of clips per clipper**.

## PART 5 — the ranked fix list

Ranked by clipper impact per unit of effort. **No rewrite is proposed.** Every item is a contained change.

### Do BEFORE the growth arrives

| # | Fix | What it is | Who feels it | Size |
|---|---|---|---|---|
| 1 | **Lazy-load recharts** | Wrap `EarningsChart` in `next/dynamic({ ssr:false })` so the 380.9 KB raw / **110 KB gzip** chart chunk is not in the initial `/earnings` bundle. Cuts that page's download by **41%** | every clipper who opens Earnings, worst on mobile data | **one import line** |
| 2 | **Scope or cache `/api/campaigns/spend`** | Add a `where: { campaignId: { in: <the ids being displayed> } }` to both `groupBy`s (`:63`, `:94`), or put a short TTL cache in front. | 3 | **`select` instead of `include` on `/api/clips/mine`** | `clips/mine/route.ts:60-79` already discards 14 fields in JS at `:117-126`. | 5 | **Flatten `/api/gamification`** | Collapse the 8 to 11 sequential round-trips (`lib/gamification.ts:518-628`) with a `Promise.all` and one shared `user` read, and move the `updateStreak` **write** off the GET path. It blocks `/clips` and re-runs every 60 s | every clipper, every minute | **medium, and it is a streak/level path so it needs care** |

### Said plainly

**The database is fine for 10x and mostly fine for 100x, and I am not going to manufacture work there.** Clipper reads are user-scoped and correctly indexed; measured plans came back between 0.4 ms and 9.2 ms. The two genuine exceptions are named above and both are contained: the unscoped aggregate in `/api/campaigns/spend` (item 2) and the whole-row payload of `/api/clips/mine` (item 3).

**The front end is where the cheap phone actually suffers, and item 1 alone is the single best ratio of clipper benefit to effort on this list:** one import line removes 41% of the heaviest clipper page's download. If only three things get done before the growth arrives, do 1, 2 and 3.

## What could NOT be measured from here

* **Real device performance:** LCP, TTI, main-thread parse and paint cost on a low-end phone. Needs a Lighthouse mobile run or WebPageTest on real hardware. Transfer-time figures in PART 1 are arithmetic from measured bytes, not observed page loads.
* **The exact SQL Prisma emits** for `include: { stats: { orderBy, take: 1 } }` in `/api/clips/mine`. I measured the equivalent pattern (474 to 1,092 ms for a 237-clip clipper); capturing the real statement needs Prisma query logging on a live request.
* **The precise degradation curve for the `clips` table** at 10x and 100x. The curve in PART 2 uses other real tables at 69k, 162k and 377k rows as proxies, which differ in row width and cross Postgres's parallel-plan threshold. An exact curve needs a seeded copy of `clips`, which is a data write.
* **One-handed usability and true 375px rendering** of the batch rows on a physical handset (PART 3).
* **Worst-case draft loss window** for the debounced batch autosave under a dropped connection.
* **Railway box headroom, Supabase pooler client limits and Postgres `max_connections`**, all plan-dependent and already flagged as unmeasurable by BL-628.
