## PART 1 — the cron header

### Every route that accepts it
Exactly **two**, found by grepping the whole `src/` tree:
- `src/app/api/cron/lifecycle/route.ts:31` and `:33` (the growth engine sweep, sends real marketing email)
- `src/app/api/cron/watchdog/route.ts:74` and `:76` (the CRON_DOWN alarm and the owner-bell alert path)

The other **ten** cron routes are CRON_SECRET-only (bearer, fail-closed if the secret is unset), verified individually. `tracking` and `retire-dead-clips`, the two the brief worried about most, are among the ten and do **not** touch this header.

### Why the header exists
`watchdog/route.ts:64-72` records it: BL-148 added it because Vercel Cron cannot send a custom `Authorization` header, so a Vercel-scheduled call could never match `Bearer <CRON_SECRET>`, and the watchdog was returning 401 on every real invocation (zero CRON_DOWN alerts during actual stalls). Vercel injects `x-vercel-cron: 1` on every verified cron call and strips it from external traffic, so the code treats the header's presence as proof the call came from Vercel's scheduler.

### Does a legitimate caller depend on it TODAY? YES. This is the crux, and it is evidenced, not assumed.
`vercel.json` still schedules exactly five jobs, and lifecycle and watchdog are two of them (`"*/15 * * * *"` and `"0,30 * * * *"`). I confirmed these are the live ingress from the cron heartbeat table:

| kind | fires in 3h | second-past-the-minute of each fire | reading |
|---|---|---|---|
| lifecycle | 12 | 0.54, 0.62, 0.69, 0.70, 0.71 ... | **exact top of minute = Vercel Cron** |
| watchdog | 6 | 0.85, 0.87, 0.87, 0.92 ... | **exact top of minute = Vercel Cron** |
| tracking | 18 | 1.0, 12, 16, 19, 26, 33, 51 (jittered), minutes 11 and 41 not multiples of 10 | **an external pinger sending the bearer** |

The pattern is unambiguous. Lifecycle and watchdog fire within one second of the scheduled minute, which is Vercel Cron's signature. Tracking fires at scattered seconds and off-cadence minutes, which is an external HTTP pinger (the "external trigger" BL-506 documents) sending `Authorization: Bearer <CRON_SECRET>`. The Railway in-process scheduler is **not** the live mechanism for these two: `instrumentation.ts:14-20` only starts it when `RAILWAY_NATIVE_CRON` is non-empty, and that variable is not set (the scheduler logs "disabled" on an unset flag).

**So BL-640's proposed fix is wrong for the live configuration.** BL-640 said "Railway already sends it, so nothing breaks." Railway is not sending it; Vercel is, and Vercel is using the header precisely because it cannot send the bearer. **Blindly deleting the `isVercelCron` branch would 401 every lifecycle and watchdog call, silently killing the growth engine and the outage alarm.** That is materially worse than the flaw.

### What the flaw actually is, and how real it is
The header is a trustworthy signal **only if every origin that can reach the app strips it from external traffic**. Vercel's edge does strip it. The concern is a second ingress: `railway.json` shows the app also builds and runs on Railway (`NIXPACKS`, `npm start`), and a directly reachable Railway origin would **not** strip an attacker-supplied `x-vercel-cron: 1`. If such an origin is publicly reachable, an attacker could trigger the growth engine (send a marketing sweep) or the watchdog on demand. **I could not determine the ingress topology from the repo**, which is the honest boundary here: whether clipershq.com is served exclusively through Vercel, or whether a Railway URL is also public, is an infrastructure fact not visible in code. Treat the flaw as real until that is checked, because the downside (spam sent to the whole user base) is reputational.

### The fix, and how to verify every cron still fires
The fix is **not** deletion. It is to give Vercel Cron a way to send a real secret, then remove the header trust:
1. Set `CRON_SECRET` in Vercel's project env (modern Vercel Cron automatically sends `Authorization: Bearer <CRON_SECRET>` when that variable is present, which did not exist when BL-148 was written; this is the change that makes the header obsolete).
2. Confirm from the heartbeat table that lifecycle and watchdog keep firing on their cadence for a full day **with the header branch still in place** (proves Vercel is now sending the bearer).
3. Only then remove the `isVercelCron` branch at both sites, so both routes are bearer-only like the other ten.
4. Verify again for 24h that all 12 crons fire (the `cron_runs` heartbeat by kind is the single source of truth; the watchdog itself alarms if tracking stalls).

Alternative if Vercel Cron secret injection is not wanted: move lifecycle and watchdog onto the **same external pinger that already fires tracking** with the bearer, delete their `vercel.json` entries, then delete the branch. This consolidates to one ingress and is the cleaner end state, at the cost of trusting one external pinger for all 12.

**What breaks if done carelessly:** removing the branch before step 1 or 2 → lifecycle and watchdog 401 on every call → no lifecycle emails and, worse, a blind watchdog that never warns of a tracking stall (the exact silent failure BL-148 was fixing). **Verification is non-negotiable: watch the heartbeat for 24h before and after.**

## PART 2 — the rate limiter

### What it protects and how it behaves
`src/lib/rate-limit.ts` is a single in-process `Map` (`:20`). Roughly 40 call sites use it. The overwhelming majority are keyed by `session.user.id` (chat, clips-mine, earnings-self, payout-req, campaign-edit, review actions), so they throttle an **already-authenticated** user and the per-instance weakness only lets an authenticated abuser get N times their intended burst, bounded and never a data-theft path. The genuinely pre-auth, scale-sensitive keys are few:
- `magic-link:${email}` (`request-magic-link/route.ts:22`), limit 3/hour, **keyed by email with no IP limit** — the email-bombing and address-enumeration gap BL-640 named.
- `magic-verify:${ip}`, `referral-click:${code}:${ip}`, `owner-ref-click:${code}:${ip}` — these already key by IP, which is the right shape.

### What actually happens at scale
The store is a plain Map with a 60s sweep. **On every restart or redeploy it is empty, so all counters reset. Across instances each has its own Map.** The behaviour is therefore **fail-open**: the worst case is that a limit is weaker than intended (an attacker gets up to instance-count times the limit, and a fresh window after every deploy). **It can never fail closed, so it can never lock a clipper out.** That is the current reality and it is the safe direction.

### The fix, per option
| Option | Cost / dependency | Store-unavailable failure mode |
|---|---|---|
| **Upstash Redis** (serverless, HTTP, pay-per-request) | New external dependency + secret; a few lines behind the existing `checkRateLimit` signature | Must be chosen explicitly; see below |
| **Postgres counter table** (a `rate_limit` row per key+window, atomic upsert) | No new dependency, reuses the DB the app already has; adds write load to the primary | If the DB is down the whole app is down anyway, so the question is moot for the app but not for the limiter path |
| **Vercel/platform WAF rate rules** | Platform feature, no app code; only helps at the edge, cannot see `userId` | N/A, it is infra |

### Fail-open or fail-closed? FAIL OPEN, deliberately.
If the shared store is unreachable, the limiter must **allow** the request, because failing closed would lock every clipper out of submitting clips, logging in and withdrawing the instant Redis or the counter table hiccups. A rate limiter is a defense-in-depth layer, not a correctness gate; the money invariants (budget cap, payout race protection) live elsewhere and do not depend on it. So: try the shared store, and on any error fall through to `allowed: true` (optionally with the existing in-memory Map as a same-instance backstop so a store outage degrades to today's behaviour rather than to no limiting at all). **No legitimate clipper is locked out by normal use**, because the limits themselves are generous (3 payout requests/hour, 12 batch submits/hour, 30 chat messages/minute) and a real clipper never approaches them.

### Recommendation
Do the **narrow, high-value slice only**: add an **IP-keyed limit alongside the email key on magic-link**, and keep it in the existing in-memory Map for now. That closes the one pre-auth abuse vector BL-640 flagged without taking on a new external dependency. The full shared-store migration is worth it only once there is genuinely more than one instance under load, which is not today. Threshold to revisit: when the app runs more than one instance in production, or when abuse is actually observed in the logs.

## PART 3 — the wallet addresses

### Who reads them, and do they need to be decryptable? YES, decryptable, not hashed.
- The **owner** reads them through the payouts GET (`payouts/route.ts:231-232`), rendered on `admin/payouts/page.tsx`, **to actually send the crypto**. They must be recoverable in plaintext at read time. Hashing is off the table.
- The **clipper** reads their own back on `payouts/mine/route.ts:45,60`.
Both already flow through `readWalletFields` (`wallet-crypto.ts:139`), which prefers the encrypted column and falls back to plaintext.

### The important finding: this is already built and the key is already live.
`src/lib/wallet-crypto.ts` implements AES-256-GCM (`:33-38`) with a per-row random IV, dual-write on create, and a graceful decrypt-to-plaintext fallback. A one-shot idempotent backfill exists at `scripts/migrations/F-SECURITY-WALLET-ENCRYPT-BACKFILL.ts` and aborts if the key is missing so it can never write nulls that look like success. **Live state confirms the key is set in production:** of 133 payout rows, **133 hold plaintext, 81 already hold ciphertext, 52 are plaintext-only, 5 are in-flight.** The 81 encrypted rows prove `WALLET_ENC_KEY` is live in Railway; the dual-write has been running. **The work that never finished is the backfill of the 52 stragglers and the drop of the plaintext column.**

### Key location and the consequence of loss
The key lives in the Railway environment as `WALLET_ENC_KEY` (64 hex chars). **If it is lost AND the plaintext column has already been dropped, every stored address becomes unrecoverable and the owner cannot send those payouts.** That is why the plaintext column must not be dropped until (a) the key is backed up somewhere durable outside Railway, and (b) 100 percent of rows are proven to decrypt. Today the plaintext column is the safety net (`readWalletFields` falls back to it), which is exactly why BL-640 found reads still work.

### The migration, with no address lost
1. Back up `WALLET_ENC_KEY` to the owner's password manager first. This is the single most important step; the key loss scenario is worse than the plaintext risk.
2. Run the existing backfill on the 52 plaintext-only rows (idempotent, per-row try/catch).
3. **Prove full coverage read-only:** every row's `decryptField(walletAddressEnc)` equals its plaintext, all 133, zero mismatches. This is the gate.
4. Only after a clean 133/133 proof, drop the plaintext columns in a later round.

### Is it worth it at 133 rows? Yes, but finish it, do not drop the net yet.
The honest answer is nuanced. The encryption is already 61 percent done and the key is already a live dependency, so the key-management risk is **already taken** whether or not the migration finishes. Given that, leaving 52 rows in plaintext is the worst of both worlds: the key risk without the full benefit. So: **finish the backfill and prove coverage now** (low risk, no new dependency, reversible), but **keep the plaintext column as the safety net until the key is durably backed up** and 133/133 decrypt is proven. Dropping the column is the only irreversible step and should wait for its own round with the owner present.

## PART 4 — rank, order, and spec

Ranked by real-world risk against implementation risk:

| # | Finding | Real-world risk | Implementation risk if careless | Fix | Verify | Rollback |
|---|---|---|---|---|---|---|
| **1** | Cron header (`x-vercel-cron` on lifecycle + watchdog) | **Highest.** Unauthenticated marketing-email trigger if a non-Vercel origin is public. Reputational | **Highest.** Deleting the branch first 401s the growth engine AND blinds the outage watchdog | Set `CRON_SECRET` in Vercel env, confirm 24h of fires, then remove both `isVercelCron` branches (`:31`, `:74`) | `cron_runs` heartbeat by kind for 24h before and after; all 12 crons still fire | Restore the branch (one revert); the header path is still there until step 3 |
| **2** | Wallet plaintext (52 of 133 rows) | Medium. DB-dump correlation of address to identity. Mitigated by RLS deny-all + Prisma-only access | Low, until the plaintext drop, which is a separate later step | Back up the key, run the existing backfill on 52 rows, prove 133/133 decrypt | Read-only diff: `decryptField(enc) == plaintext` for all 133 | Backfill only writes the enc column; plaintext untouched, so revert is a no-op |
| **3** | In-memory rate limiter | Lowest. Already fails OPEN, cannot lock anyone out; only pre-auth gap is magic-link email bombing | Low. Adding an IP key is additive | Add an IP-keyed limit beside `magic-link:${email}` (`:22`); defer the shared-store migration | Send repeated magic-link requests from one IP in staging; confirm the IP limit trips while a normal user is unaffected | Remove the added key check (one line) |

### Recommended order
**Do 2 first, then 1, and defer most of 3.** Reasoning:
- **Finding 2's backfill is the safest possible action**: it only writes the encrypted column, never touches plaintext, is idempotent, and the infrastructure and key already exist. It removes 52 rows of exposure with essentially zero breakage surface, and it is fully reversible because the plaintext net stays. Do it first to bank a real win with no risk. **Do not drop the plaintext column in the same round.**
- **Finding 1 is the highest real risk but demands the most care**, so it goes second, once there is appetite for the 24-hour verification window it requires. It must never be a blind deletion. The single most useful preliminary is an infrastructure check the owner can answer in a minute: is any Railway origin publicly reachable, or is the app served only through Vercel? If Vercel-only, the flaw is largely theoretical and this drops in urgency.
- **Finding 3's full migration should wait.** The limiter already fails open, so nothing is at risk of locking out, and a shared store is premature at one instance. Do only the tiny magic-link IP-key slice if email abuse is actually seen; otherwise the honest threshold is "more than one production instance, or observed abuse in the logs."

### What must not happen, restated because each is worse than the flaw
- Do **not** delete the `x-vercel-cron` branch before Vercel is confirmed sending the bearer. It stops tracking's alarm and the growth engine.
- Do **not** make the rate limiter fail closed. It would lock every clipper out the moment the store blips.
- Do **not** drop the plaintext wallet column before the key is durably backed up and 133/133 decrypt is proven. A lost key then means payouts are impossible.

### What could not be measured, and why
- **The cron ingress topology.** Whether a Railway origin is publicly reachable (which decides how real finding 1 is) is an infrastructure fact not in the repo. The timing evidence proves Vercel Cron is the live scheduler for these two routes; it cannot prove no other origin is reachable.
- **The production instance count**, which sets how much finding 2's per-instance weakness actually matters. No request-rate telemetry was read.
- **Whether the 5 in-flight payouts include any of the 52 plaintext-only rows.** Not probed beyond the aggregate, to avoid reading wallet values; the backfill covers them regardless of status.
