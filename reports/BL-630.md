# BL-630 — the ghost fee, given a runtime existence

**2026-07-22 · SHIPPED to branch `checkpoint/BL-630` (@ `18d178f`, verified on origin). Base:** origin/main `a9eb22de`. Tags `pre-BL-630` / `post-BL-630`. Closes the BL-627 latent leak: BL-625 subtracted a platform fee ON SCREEN but stored the full marketed budget as spendable, so a fee campaign would have let clippers earn the ghost ($550 on a $5,000 / 11%). Nothing was broken (no fee campaign existed); this fixes it before the first one is created.

## PART 1 — persist the fee
New additive nullable column `campaigns.platformFeePctDecimal Decimal(5,2)`, applied to prod via `ALTER TABLE campaigns ADD COLUMN IF NOT EXISTS "platformFeePctDecimal" DECIMAL(5,2)` (the sanctioned `run-schema-sql` path, NOT `prisma migrate`) plus `npx prisma generate`. I chose a PERCENT column (not an amount) so the ghost recomputes correctly if the budget is ever topped up (a selling platform takes its cut of whatever the budget is) and so it round-trips the calculator's own input. DEFAULT is NULL = no fee. Verified on live data: all 30 existing campaigns are NULL, so every one behaves exactly as before.

## PART 2 — the pool cap respects it (money files, loudly justified)
New PURE helper `src/lib/platform-fee.ts` `realBudgetFromFee(budget, feePct)` does the only fee arithmetic, on `Prisma.Decimal` (decimal.js, NEVER float), returning 2-dp numbers so the existing float cap math needs no other change. A NULL / 0 / >=100 fee returns `{ ghostFee: 0, realBudget: <the raw budget, unchanged> }`, the byte-identical guarantee.

Two money files changed (full diffs in the commit; the other 4 are BYTE-IDENTICAL by blob OID: clip-earnings-writer, earnings-calc, clip-earnings-invariant-middleware, money-decimal; campaign-era + earnings-never-decrease also identical):
* `balance.ts` getCampaignBudgetStatus: reads the fee, computes `{ ghostFee, realBudget }`, adds `ghostFee` to `spent` (so a fully-earned pool reads MARKETED fully spent), and derives `clipperPoolCap = (1-s)*realBudget` / `ownerReserveCap = s*realBudget` (was `*budget`).
* `tracking.ts`: the BL-162 guarantee per-clip cap and the auto-pause derive from `realBudget` (was `budget`); one `select` gains the fee column. Only the pool-cap lines moved; no earnings-write logic touched.
* `campaign-clipper-view.resolveClipperPoolCap` (display helper, not a money file): matches balance.ts to the cent so the clipper exhaustion test stays correct.

Every one of these is gated so `realBudget == budget` when the fee is NULL, i.e. byte-identical for every existing campaign. The BL-167 clamp and the L1 hard-lock in clip-earnings-writer consume the new `status.clipperPoolCap` / ghost-inclusive `spent` unchanged, so the enforcement path is fixed without editing that file. Clippers still see the MARKETED budget (BL-535 not regressed); the ghost is never a clipper-facing line.

## PART 3 — no existing campaign moves (proven on LIVE data)
`scripts/bl630-inert.ts` calls the NEW getCampaignBudgetStatus for every budgeted campaign and compares its caps to the OLD budget-based formula: **15 campaigns checked, 0 carry a fee, 0 caps moved.** No existing campaign's pool cap, owner share, clipper earnings or displayed budget changed by a cent. No earnings-write path was touched, so no clip's earnings can change or decrease (never-decrease guard byte-identical).

## PART 4 — the fee case, end to end
`scripts/bl630-prove.ts`, 22/22, flagship $5,000 marketed / 11% fee / 33% owner (CPMs clipper 1.00, owner 0.4925, stored s = 0.329983):
```
stored: budget 5000, platformFeePctDecimal 11.00
ghost $550.00 | realBudget $4,450.00
clipper pool cap $2,981.57 (was $3,350.08 -> ghost leak $368.50 CLOSED)
owner reserve  $1,468.43 (was $1,649.92)
ghost 550.00 + reserve 1468.43 + pool 2981.57 = $5,000.00 exactly (marketed fully spent)
```
The exact invariant `ghost + realBudget == marketed` holds on every awkward budget ($3,333.33, $9,750, $0.07, ...). The $550 ghost is unearnable (bounded by the realBudget pool cap and the L1 marketed gate) and never a clipper line. The stored fee equals the calculator's displayed fee, so screen and database agree on the fee. NOTE: the runtime clipper cap $2,981.57 is 7 cents above the calculator's headline $2,981.50 because the stored locked share is 32.9983% (4-dp CPM rounding) vs the display's target 33% — a pre-existing BL-625 artifact, clipper-favourable and immaterial; the MATERIAL $368.50 ghost leak is closed to the cent. A tiny odd realBudget can make the two independent round-half-up ceilings sum a cent over realBudget; the L1 hard-lock (ghost-inclusive spent vs marketed) is the hard backstop that bounds any actual payout at marketed.

## PART 5 — the edit path is guarded
The fee is IMMUTABLE after creation. The PATCH route BLOCKS any change to it with a `400` + `code: PLATFORM_FEE_IMMUTABLE` and the exact blast radius (clip count, clipper count); it is also absent from `CAMPAIGN_FIELDS`, so neither save path can write it. The calculator sends the fee on CREATE only; the edit form loads and drops it. A mid-campaign fee change therefore cannot happen silently (BL-570's lesson).
