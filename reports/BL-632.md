# BL-632 — daily submission counter counts UP (used), not DOWN (left)

**Shipped 2026-07-22.** branch `checkpoint/BL-632` @ `5c481ef` (verified push). Tags: pre/post-BL-632. Rollback: `git revert` the wording commit `0952c6e`.

## The change
The submit-clip allowance read "10 of 10 left today", which counts down and reads backwards. It now counts UP as clips are used, per BL-628 PART 4:
- Fresh day: **"0 of 10 used today"**
- After three: **"3 of 10 used today"**
- At the cap: **"10 of 10 used today, come back tomorrow"** (never a bare "0 left" that reads as an error).

`used = limit - remaining`, so the displayed count is exactly the server side used figure and can never disagree with what the server enforces.

## Proof
**Wording at each state (both the single modal and the batch line):**
- Fresh day (remaining = limit): single chip "0 of 10 used today"; modal line "0 of 10 submissions used today for this campaign."; batch line "Adding N clips. 0 of 10 used today for this campaign." Confirmed by the arithmetic `limit - remaining = 10 - 10 = 0`.
- Mid day (3 used, remaining = 7): "3 of 10 used today" on the chip and "3 of 10 submissions used today" in the modal and batch line. `10 - 7 = 3`.
- At the cap (remaining = 0, full = true): chip "10 of 10 used today"; modal + page "10 of 10 used today, come back tomorrow." No bare "0 left".

**Live update as rows are added (batch):** the line lives in `<p id="limit-meter" role="status">`, an existing aria-live polite region, and re-renders on every `rows.length` change (the effect on line 99 keys on `rows.length`). As the clipper adds rows the "Adding N clips" figure updates live and is announced; the "used today" anchor stays equal to the server side used figure. In this codebase adding a row never decremented the displayed remaining figure (it only incremented "Adding N clips"), so per the task's conditional the used figure correctly tracks server-side used, not a projection, keeping display and server in agreement.

**Boundary cases (arithmetic verified):**
- 0 rows: batch shows the base "used" = `limit - remaining`; no pending. Correct.
- 1 row: "Adding 1 clip." updates live; used figure = server used. Correct.
- Exactly at the cap: `remaining === 0` / `full` renders "10 of 10 used today". This is exactly when `addDisabled` (`rows.length >= remaining`) engages, so the display flips to fully-used precisely when the server would block the next add. Display agrees with server.
- Attempting to exceed: the unchanged block logic still fires. `blockReason` (`rows.length > remaining` -> "You can submit R more today; remove X") and `addDisabled` are byte-identical; the over-limit submit is still named clearly.

**Underlying limit logic byte-identical (the hard constraint):** the diff is display strings only. `getDailyRemaining`, the `remaining === 0` / `full` conditionals, `addDisabled`, `blockReason`, and the server-side per-campaign limit checks are unchanged.
