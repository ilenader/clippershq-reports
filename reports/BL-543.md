# BL-543 — kill the Instagram actor cascade

Measured against Apify's own run records, n=30 consecutive production runs each:

| Actor | Runs | Status | Usable results | Win rate |
|---|---|---|---|---|
| `apify/instagram-reel-scraper` | 30 | **SUCCEEDED 30/30 (billed 30/30)** | **0** | **0.0%** |
| `apify/instagram-scraper` | 30 | **SUCCEEDED 30/30 (billed 30/30)** | **0** | **0.0%** |

Both returned `error:"not_found"`, `errorDescription:"Post does not exist"`, every time. **~$45/month, buying nothing, ever.**

## PART 1 — the cascade, mapped

### Is it a genuine fallback chain? Yes. That was never the bug.

`fetchInstagramStats` (`src/lib/apify.ts:554`) is a strict `if (result !== null) return` chain. It stops on the first success. It does **not** fire unconditionally.

| Tier | file:line | Actor | Fires when |
|---|---|---|---|
| 1 | `apify.ts:601` → `:605` | `apify/instagram-api-scraper` | always |
| 2 | `apify.ts:564` → `apidojo.ts:144` | `apidojo/instagram-scraper-api` | tier 1 returned null |
| 3a | `apify.ts:672` (was) | `apify/instagram-reel-scraper` | tiers 1+2 returned null |
| 3b | `apify.ts:176` → `:178` | `apify/instagram-scraper` | tier 3a produced no usable item |

The batch path (`apify.ts:1195`) is also honest: HikerAPI per-URL first, then **only unresolved** URLs to api-scraper, then **only missing** to apidojo. The comment "no double-fetch, no double-pay" is accurate.

### So what actually went wrong

**The chain is correct. The legs were broken.** Two independent defects, both invisible because every run reports SUCCEEDED to Apify while returning junk to us:

1. **`apify/instagram-reel-scraper` was called with the wrong input shape.** `apify.ts:676` sent `{ username: [postUrl] }` — a **post URL in a field that takes usernames**. reel-scraper is a *profile* scraper. It searched for an account literally named `https://www.instagram.com/reel/ABC/`, found none, and answered `not_found`. Verified on a live run's INPUT record:
   ```
   INPUT : {"username":["https://www.instagram.com/reel/DZDEVE1o54_/?igsh=..."],"resultsLimit":1,...}
   OUTPUT: {"url":"...","username":"reel","error":"not_found","errorDescription":"Post does not exist"}
   ```
2. **`apify/instagram-scraper` is called correctly but cannot parse a reel URL.** It uses the right `directUrls` + `resultsType:"posts"` shape, yet it derives the username from the first path segment, so `instagram.com/reel/<code>` yields `username:"reel"` and the same `not_found`. The output literally says so.

**96% of our Instagram clips are `/reel/` URLs** (1,521 of 1,584 live; 63 are `/p/`). So tier 3 was two guaranteed-failure actors, in series, on nearly every clip that reached it.

### Which actor actually wins

Tier 1, `apify/instagram-api-scraper`, essentially always. Proven end-to-end on 4 live reels below: every one resolved at tier 1 with `provider=apify-instagram-api`. Tiers 3a/3b won **0 of 60 measured runs**.

### Why the ledger hid this (BL-540's 359-vs-1 finding, explained)

Two compounding reasons, both now visible in the code:

1. **The provider label is derived from HITS, not CALLS.** A leg that fires, bills, and returns nothing never appears in the label, so `apidojo` billed 359 runs/day while the ledger recorded ~1.
2. **`apify-ledger/route.ts:198` skips `!e.success` rows from the spend total entirely.** Apify bills a failed run exactly the same as a successful one. So the wasted legs were invisible **twice**.

## PART 2 — the ledger now records every billed call

**Constraint that shaped the design:** 5 of the 10 `logApifyUsage` call sites live in **`tracking.ts`**, which is one of the 6 money files and must stay byte-identical. The winner-label contract therefore could not be rewritten at the caller. So the truth is recorded at the only place that can know it: **the actor invocation itself**.

* **`logActorRun()`** (`apify-ledger.ts`) writes **one row per billed actor run, whatever the outcome**, with `provider = "actor:<name>"` and the live per-actor price verified against `/v2/acts/<actor>/pricingInfos`.
* Wired into every exit path of `fetchInstagramFallbackItem` (`apify.ts:174`) and `fetchInstagramStatsApiScraperSingle` (`apify.ts:601`) — HTTP error, empty, item error, silent-zero, and success all log.
* **No double count:** `apify-ledger/route.ts` splits `actor:` rows out of the legacy rollups and reports them as a separate authoritative `actorTruth` block. Every existing response key keeps its exact shape, so the dashboard renders unchanged.
* `actorTruth` deliberately sums **every** actor row regardless of `success` — which is what the invoice does — and surfaces `wastedUsd`: money spent on runs that returned nothing usable. That single number is what would have made this round unnecessary.

**Honest scope limit:** the 4 bio call sites BL-540 flagged as unlogged (`account-profile.ts:132`, `verify/route.ts:233` and `:836`, `verify-cascade.ts:205`) are **still unlogged**. They are $0.03 per 3.5 days (0.1% of spend) and wiring them touches the account-verify path for no measurable money. Reported, not done. See BL-544.

**Reconciliation:** `actorTruth` cannot be compared to Apify's billing API until the new rows accumulate in production — the table has none yet. The comparison route is `GET /v2/users/me/usage/monthly` → `monthlyServiceUsage.PAID_ACTORS_PER_EVENT` (BL-540 used exactly this; it read $156.35 of a $156.54 cycle). **I did not fabricate a post-deploy reconciliation I could not run.** What is proven now: the per-actor price model is exact (`$0.0033` for a reel-scraper run, `$0.003` for an api-scraper run, both matching Apify's own `usageTotalUsd` on real runs to the cent) and a 0-item `not_found` run still costs money in the ledger rather than $0.

## PART 3 — what was cut, and what deliberately stayed

**REMOVED: `apify/instagram-reel-scraper` (tier 3a), entirely.** It is a profile scraper fed a post URL. It cannot resolve any post URL, of any shape, ever. 0/30 usable.

**SKIPPED for reels only: `apify/instagram-scraper` (tier 3b).** `legacyActorsCanResolve()` (`apify.ts:153`) returns false for `/reel/` and `/reels/` — the shapes proven dead — and **true for everything else**, so `/p/` posts (63 clips) keep the exact pre-BL-543 path. It **fails open**: anything unparseable keeps the old behaviour. A cost fix must never be the reason a view goes missing.

**KEPT, and why:** tier 3b for `/p/` URLs. Measured 0/30, but that sample is dominated by reels, and I have **not** proven it dead for `/p/`. An unproven leg stays. That is the rule that decided every call here.

**Why this cannot change a stored view count.** For a reel, tiers 1+2 failing already produced `null` — because 3a and 3b returned `not_found` 100% of the time. The new code produces the same `null` without buying two guaranteed failures on the way. **Identical output, minus the invoice.**

### The saving, from Apify's real numbers

| Leg | Runs/3.5d | $/3.5d | Eliminated | $/month saved |
|---|---|---|---|---|
| `apify/instagram-reel-scraper` | 949 | $3.13 | 100% (leg removed) | **~$26.8** |
| `apify/instagram-scraper` | 940 | $2.16 | ~96% (reels only) | **~$17.8** |
| | | | **Total** | **~$45/month** |

Against Apify's actual **$5.59/day / $168/month** (BL-540, billing API): a **27% cut**, to roughly **$123/month**. No vendor change, no cadence change, no money-path risk.
