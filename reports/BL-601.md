# BL-601 — Verification of the BL-598 Multi-Clip Batch Design (READ ONLY)

Date: 2026-07-20. Branch: `checkpoint/BL-601`. Scope: pressure test BL-598's batch
submission design against the REAL code and REAL data before any build, hunting
specifically for wrong account attribution (a money and trust bug) and silent half
submits. READ ONLY: nothing built, no code/config/data/money change, no build run
(a markdown only diff cannot change tsc/build). Every claim is cited file:line.

## VERDICT (one line)

The design is DIRECTIONALLY sound but NOT build ready as written. Six gaps must be
fixed in the spec first; the top two are money traps (per row account resolution
and accidental reuse of the owner override core). Corrected build ready spec is in
Part 5.

## PART 1 — THE SINGLE SUBMIT CORE AS IT REALLY IS

**`processClipperSubmitLink` does NOT exist.** A repo wide grep for
`processClipperSubmitLink` and `clipper-submit-core` returns NO files. BL-598's
"thin fan out over processClipperSubmitLink" describes a core that must be BUILT
(BL-598 build step 1), not one that exists. Today the single submit logic lives
INLINE in `src/app/api/clips/route.ts` `POST` (lines 677-1261), entangled with
request scoped concerns that a loop cannot reuse as is:

• Body parse `req.json()` (698), session + ban + clipper role (678-692).
• Idempotency cache keyed `userId + clipUrl`, 60s (717).
• Per user burst rate limit `clip-submit:<userId>` 12/hr (743-762).
• `after()` post response hooks: streak, FIRST_CLIP_CELEBRATE, analytics (1192-1247).

The reusable per clip body IS present but not factored out: account lookup (807),
RULE 0 membership (821), campaign state + budget (834-870), **the per campaign
daily limit (871-903)**, RULE 1 platform vs account (907), RULE 1b campaign
platform (916), RULE 1c per platform CPM (932), RULE 2 freshness (946-1045), dup
checks (1057-1081), CPM freeze (1084-1104), and the 3 write transaction (1107-1167).

Answer to the question asked: the core is EXTRACTABLE but is NOT clean today, and
the LIMIT + platform + ownership checks live in the ROUTE, not in a helper. So the
design's premise ("thin fan out over a clean core") is not yet true; the extraction
(build step 1) is real work and the batch cannot be built before it lands and is
proven byte identical.

A precedent exists: `src/lib/owner-submit-core.ts` factors the OWNER path into
`validateOwnerSubmitContext` + `processOwnerSubmitLink`. It is the right SHAPE to
mirror, but it is the wrong CODE to reuse for clippers (Part 2, gap 1b).

## PART 2 — MAPPING INTEGRITY (the owner's core worry) — TWO REAL DEFECTS

### Gap 1a (CRITICAL, wrong account attribution): the reuse template binds ONE account to the whole batch

`owner-submit-bulk` resolves ONE account once via `validateOwnerSubmitContext`
(owner-submit-core.ts:99-108) and passes that SAME `ctx.account` to EVERY link
(owner-submit-bulk/route.ts:128-142). That is correct for the owner (one target
clipper, one account, many links) but is EXACTLY the cross contamination the owner
fears for the clipper batch: if the clipper batch copies this shape, row 2's
TikTok link and row 5's YouTube link all attach to row 1's account. The clipper
batch MUST resolve and validate an explicit `clipAccountId` PER ROW and pass it
into each row's processing independently. The body shape must be
`rows:[{clipUrl, clipAccountId}]`, never a batch level account.

### Gap 1b (CRITICAL, money + auto approve trap): do NOT reuse `processOwnerSubmitLink`

Because `owner-submit-bulk` already fans out over `processOwnerSubmitLink`, it is
the tempting thing to call. It must NOT be used for clippers, because it:
• force sets `status:"APPROVED"` + `isOwnerOverride:true` + `reviewedById`
  (owner-submit-core.ts:276-288) — clipper clips must be PENDING and go through
  review;
• writes earnings immediately via `writeClipEarnings` when views>0 (337-338) —
  the clipper path NEVER writes earnings at submit (tracking does it later);
• when no account is passed, picks the FIRST approved account OR MINTS A PLACEHOLDER
  account (173-183) — for a clipper this is silent wrong attribution to an
  arbitrary or fake account.
The clipper batch must call a NEW clipper core that mirrors `/api/clips` PENDING
semantics, and reuse ONLY owner-submit-bulk's REPORTING shell (BL-213 never drop a
line accounting, route.ts:59-171). Reporting shell = yes; money/create path = no.

### Gap 2 (HIGH, platform casing mismatch): auto mapping and RULE 1 disagree with real data

Platform strings are INCONSISTENT in production, verified by query:

`clip_accounts.platform`: `tiktok` (2), `TIKTOK` (14), `TikTok` (358),
`Instagram` (305), `YouTube` (321). `campaigns.platform` includes `TIKTOK` (15)
alongside `TikTok`. So TikTok exists in THREE casings on real accounts.

`socialPlatformForUrl` returns strictly `"TikTok" | "Instagram" | "YouTube"`
(social-hosts.ts:14,57-60). The single submit RULE 1 compares case SENSITIVELY:
`if (urlPlatform && account.platform !== urlPlatform)` (route.ts:907). So a TikTok
link against an account stored `"TIKTOK"` or `"tiktok"` yields
`"TIKTOK" !== "TikTok"` = true = REJECT with "your account is for TIKTOK". This is
already a latent defect for the ~16 mixed case TikTok accounts, and it directly
breaks the batch's two mapping steps:
• client auto map ("pick my approved account on the detected platform") filtered by
  exact string would MISS a `"TIKTOK"` account and falsely show "no approved TikTok
  account", or omit it from the picker;
• server RULE 1 would reject a correctly chosen `"TIKTOK"` account.

Note RULE 1b already lowercases both sides (route.ts:917-918) and owner core
lowercases both (owner-submit-core.ts:169), so the FIX is known and local: the
extracted core AND the client auto map MUST compare platform case insensitively
(`toLowerCase()` both sides). The extraction should correct RULE 1 to match RULE 1b
and owner core. Without this, honest clippers on mixed case accounts are either
blocked or shown an empty picker.

### What already prevents cross contamination (once per row resolution is honored)

With per row `clipAccountId`, each row is resolved by
`db.clipAccount.findFirst({ id: clipAccountId, userId, status:"APPROVED" })`
(the route.ts:807 pattern), so ownership + approval are enforced per row and one
row cannot borrow another's account. Multi account clippers: the server enforces
ownership + platform match but CANNOT tell which of two same platform accounts is
"intended" (both pass RULE 1) — so the UI MUST force an explicit pick when the
clipper has more than one approved account on the detected platform, never silently
default. This is a trust requirement, not just UX.

## PART 3 — SEEN CORRECTLY BY EVERYONE — the attribution keys, and the fields to set

Both views key off the SAME three Clip columns, verified:
• CLIPPER view `/api/clips/mine`: matches `userId === me` (route.ts:47-54) and
  renders the account via `clipAccount{username, platform, profileImageUrl}` and
  `campaign{name, platform}` (60-68). So a batched clip shows in the clipper's list
  under the right account IFF `Clip.userId` = the clipper and `Clip.clipAccountId`
  is the intended account.
• OWNER/ADMIN view `/api/clips` GET: attributes via `user{username}`,
  `clipAccount{username, platform}`, `campaign` (the select read in BL-598).

Therefore correct rendering everywhere reduces to setting `userId` +
`clipAccountId` + `campaignId` correctly PER ROW — which is exactly what gaps 1a/1b/2
threaten. Fix those and every view is correct by construction.

**Fields a single clipper submit sets that the batch MUST set identically** (so no
downstream view or job diverges), from route.ts:1107-1167:
• Clip: `userId`, `campaignId`, `clipAccountId`, `clipUrl`, `normalizedUrl`,
  `note`, `cpmAtSubmissionDecimal`, `ownerCpmAtSubmissionDecimal`; `status` left to
  DEFAULT `PENDING` (schema:839); `isMarketplaceClip` default false; `postedAt`
  null, `createdAt` now (NOT backdated).
• ClipStat: the first snapshot (fetched views/likes/comments/shares, or zeros on a
  lenient accept) — omitting it leaves tracking with no baseline.
• TrackingJob: `isActive:true`, rounded `nextCheckAt`, `checkIntervalMin` (60 day 0)
  — omitting it means the clip is NEVER tracked and NEVER earns.
Fields the batch must NOT set (owner only): `isOwnerOverride`, `reviewedById`,
`reviewedAt`, `backdateVerifiedAt`, `streakDayLocked`, and it must NOT call
`writeClipEarnings`. Mandating the shared clipper core makes both the "must set"
and "must not set" lists automatic; a hand rolled batch is where a field goes
missing.

## PART 4 — PARTIAL FAILURE, LIMITS, DUPLICATES, DISCONNECTS, TRANSACTION MODEL

**Transaction model: PER ROW atomic, batch NOT atomic — and that is correct.** Each
row runs its own 3 write transaction (clip+stat+trackingJob); the batch is a loop
with per row try/catch and full accounting, exactly like owner-submit-bulk. This is
the right call for the owner's "no clip silently lost or duplicated": all or nothing
would throw away 4 good clips because of 1 bad link.

• **4 of 5 valid, 1 bad**: the 4 commit to the identical per clip state; the 1
  returns `ok:false` with its reason. Response
  `summary{submitted:4, skipped:1, total:5}` + per row `results[]` (owner-bulk
  pattern, BL-213 never drop a line, route.ts:59-171). The UI marks 4 Submitted,
  1 Error; the clipper re submits only the failed row (60s idempotency on userId+url
  makes an accidental re submit of a committed row safe).
• **Mid batch limit breach**: enforce SERVER SIDE in two layers. (1) A pre check:
  recompute `maxPerDay` (override aware, route.ts:876-880) and today's count once,
  reject the whole batch if `validRows.length > remaining` BEFORE any write. (2) The
  authoritative guard: keep the per row daily count check INSIDE the clipper core
  and commit rows SEQUENTIALLY, so row N sees the N-1 already committed this batch
  and the (N)th over cap row fails cleanly with "daily limit reached" (reported, not
  silently dropped). Sequential commit is mandatory here; owner-submit-bulk's
  concurrency 5 (route.ts:26) must NOT be copied for the cap sensitive commit,
  because a concurrent count would race past the cap. Cross instance DB races are
  pre existing (no DB uniqueness on daily count) and not a regression — note only.
• **Duplicates**: within batch, dedupe first (owner-bulk `seen` set,
  route.ts:86-93). Against the DB, the campaign scope + user scope `normalizedUrl`
  checks (route.ts:1057-1081) plus the partial unique `P2002` friendly 400
  (route.ts:1256) reject the row only, not the batch.
• **Account disconnected between load and submit**: because the account is
  re resolved PER ROW at submit (findFirst APPROVED, owned), a row whose account was
  unapproved or deleted in the interim fails cleanly with the account error; other
  rows are unaffected. This only holds if gap 1a is fixed (per row resolution).
• **One campaign at a time**: enforced by body shape (single `campaignId`) + one
  membership validation; a row cannot carry its own campaign.

**Rate limit note (build detail):** a 5 row batch must not burn 5 slots of the
per user 12/hr burst gate. The batch endpoint should carry its OWN endpoint rate
limit (like owner-bulk's 20/hr, route.ts:40) and the extracted core should NOT
re run the per user burst gate per row.

## PART 5 — GATING VERDICT + CORRECTED BUILD READY SPEC

**Gating is airtight.** `isTestUser` is a User column (`@default(false)`) propagated
JWT -> session in `src/lib/auth.ts` (675, 716, 734, 745, 755, 801), so
`(session.user as any).isTestUser` is authoritative server side. The batch endpoint
returns 404 before any work when the flag is not true (invisible + unreachable), the
client renders the batch UI only when the flag is true, and the existing
`POST /api/clips` is left byte identical so every non test clipper is unchanged.
2 users currently carry `isTestUser=true` (of 1178) and can fully exercise it; the
owner grants/revokes by flipping the flag; GA is dropping the gate after sign off.

### Ranked gaps (highest = misassigns money)

1. CRITICAL 1a — per row account resolution is mandatory; never a batch shared
   account. Body `rows:[{clipUrl, clipAccountId}]`.
2. CRITICAL 1b — do NOT call `processOwnerSubmitLink` for clippers (it force
   APPROVES, writes earnings, can mint a placeholder account). New clipper core only.
3. HIGH 2 — platform comparison must be case insensitive in BOTH the client auto map
   and the server core; real accounts are `tiktok`/`TIKTOK`/`TikTok`. Fix RULE 1
   (route.ts:907) to lowercase both sides during extraction.
4. HIGH 3 — the clean core does not exist; extract it verbatim and prove
   `POST /api/clips` byte identical BEFORE any fan out.
5. MEDIUM 4 — reproduce the exact field set (Part 3) via the shared core; never
   hand roll the create; never set owner only fields; never write earnings at submit.
6. MEDIUM 5 — sequential cap sensitive commit + in core per row daily count check;
   batch level pre check is UX only. Own endpoint rate limit, not per row burst.
   (LOW 6 — FIRST_CLIP_CELEBRATE / streak: sequential commit + the existing
   prior clip count guard fire them once; or hoist to once per batch.)

### Corrected, build ready design

Endpoint `POST /api/clips/batch` (test gated, own 20/hr rate limit). Body
`{ campaignId, rows:[{clipUrl, clipAccountId}] }`. Flow:
1. `isTestUser` gate -> 404 if not true. Clipper role + ban. One `campaignId`.
2. Validate campaign open/not test/not archived + membership ONCE.
3. Dedupe rows in batch; compute override aware `remaining`; reject if
   `validRows.length > remaining`.
4. Run freshness fetches at bounded concurrency (read only, cap insensitive).
5. Commit each row SEQUENTIALLY through the NEW `processClipperSubmitLink`, which
   per row: re resolves + validates the row's own `clipAccountId` (owned, APPROVED,
   platform case insensitively matches the link and the campaign), runs RULES
   0/1/1b/1c/2 + dup checks + CPM freeze, and writes the identical PENDING
   Clip + first ClipStat + active TrackingJob in one tx. Accumulate BL-213 style
   `summary + results[]`.
6. Return the full per row report; UI renders green/red per row; failed rows are
   re submittable individually.

### Ordered build steps

1. Extract `src/lib/clipper-submit-core.ts` `processClipperSubmitLink` VERBATIM from
   route.ts:807-1167 (RULES, limit, dup, CPM freeze, 3 write tx); refactor
   `POST /api/clips` to call it; prove byte identical single submit behaviour. FIX
   RULE 1 to case insensitive here.
2. Add `POST /api/clips/batch` (test gated, own rate limit) per the flow above:
   per row account, sequential cap safe commit, owner-bulk reporting shell.
3. Build the test gated UI: single campaign picker, override aware limit meter,
   collapsible rows, case insensitive platform auto map with forced explicit pick
   when multiple same platform accounts, client pre validation, per row results.
4. Accessibility + mobile pass (375px, keyboard, screen reader, aria-expanded on the
   collapse control, aria-live validation) before sign off.

## SAFETY RECAP

READ ONLY: one doc, nothing built, no code/config/data/money change, no build run
(a markdown only diff cannot change tsc/build). This verification specifically hunted
