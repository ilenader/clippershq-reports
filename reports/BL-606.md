• TRAP 1 (one account for the whole batch): the batch resolves `clipAccountId`
  PER ROW and passes each to the core independently; there is no shared account.
  Proven live: a TikTok account + an Instagram link errors "URL is from Instagram
  but your selected account is for TikTok"; an Instagram account + a TikTok link
  errors "...from TikTok but your account is for Instagram" — a DIFFERENT account

• TRAP 2 (owner force-approve core): the batch (and the single route) call the new
  CLIPPER core, never `processOwnerSubmitLink`. Static: `processOwnerSubmitLink`
  appears 0 times in both `clips/route.ts` and `clips/batch/route.ts`. The clipper
  core force-approves nothing, writes no earnings, and mints no placeholder account.
• Casing: every platform comparison is case-insensitive. Core RULE 1 (line 118):
  `account.platform.toLowerCase() !== urlPlatform.toLowerCase()`.

## PART 1 — the extracted CLIPPER core

`src/lib/clipper-submit-core.ts` `processClipperSubmitLink(db, userId, campaignId,
clipAccountId, clipUrl, note)` was factored out of `POST /api/clips`. It does
account resolution (owned + APPROVED, per row) -> RULE 0 membership -> campaign
state + budget -> the per-campaign DAILY cap -> RULE 1 (case-insensitive) / 1b / 1c
-> RULE 2 freshness (lenient on transient) -> dup checks -> CPM freeze -> the
atomic transaction: a PENDING `Clip` + a first `ClipStat` + an active `TrackingJob`
+ the two frozen CPM decimals. It NEVER force-approves, NEVER writes earnings,
NEVER mints a placeholder account, and imports none of the 6 money files. It
returns a structured result (never throws) plus a `transient` flag so the caller
can refund a rate slot.

The single route was refactored to call the core and keep only its request-scoped
shell (idempotency cache, the 12/hr burst limit, the transient refund, the
streak + FIRST_CLIP_CELEBRATE + BL-602 shadow after() hooks). The single-submit
OUTCOME is unchanged: the core writes the identical Clip/ClipStat/TrackingJob/CPM
fields the inline body did, the response is still `NextResponse.json(clip, 201)`,
and every error returns the same status + message (the daily-cap error still
carries `{remaining, limit}`). tsc + next build pass on the refactor. (Verbose
diagnostic logs were consolidated; they are not part of the clip state or the
response.)

## PART 2 — the batch fan-out

`POST /api/clips/batch` (test-only). Body `{ campaignId, rows: [{clipUrl,
clipAccountId}] }` — ONE campaign at the batch level. Flow:
1. auth + ban + isTestUser 404 gate + clipper role + a batch-level rate limit.
2. shape + in-batch dedup (first wins; a duplicate row returns its own reason).
3. SERVER-SIDE daily-cap PRE-CHECK before any write: recompute `maxPerDay`
   (override REPLACES default) + today's count; if the processable rows exceed the
   remaining allowance, reject the WHOLE batch, write nothing, with a clear message.
4. SEQUENTIAL commit through the core (per-row `clipAccountId`), so the core's own
   per-row cap check sees the rows already committed this batch — race-safe.
5. per-row report + `summary {submitted, skipped, total}` (BL-213 never-drop-a-line).

TRANSACTION MODEL: per-row, each its own atomic core transaction, with a full
per-row result. 4 succeed + 1 fails -> the 4 commit, the 1 returns its reason.
Nothing is silently lost (every input row yields exactly one result) or duplicated
(in-batch dedup + the DB normalizedUrl campaign/user dup checks + the P2002
backstop, all inside the core).

## PART 3 — the UX (accessibility-lead reviewed)

`src/app/(app)/clips/BatchSubmitSection.tsx`: a single campaign picker + a live
limit meter ("Used today X / N, R left"), then rows. Each row has a platform
`<select>`, a connected-account `<select>` (filtered case-insensitively to the
clipper's accounts on that platform), a link `<input type=url>`, and a plain
reminder line "This link will attach to @handle on <Platform>". A "＋ Add clip"
button appends a row (disabled at the remaining limit); rows are collapsible; the
Submit button is blocked with a specific reason whenever any row is incomplete,
wrong-platform, has an unconnected account, or the batch exceeds the limit. On
submit it posts to the batch endpoint and renders a per-row result. The resulting
per-clip state is the core's, so tracking, earnings, verification, and the
owner/admin review queue render each clip exactly like a single submit.

Accessibility (per the 5-specialist review): rows keyed by a STABLE `row.id` (not
index); `role="group"` + `aria-label` per row; real `<label htmlFor>`;
`aria-describedby` for the per-row reminder and the submit-disabled reason;
`aria-disabled` + a click-guard (not native `disabled`) so the blocking reason is
reachable by screen readers; `aria-expanded`/`aria-controls` + a `hidden` collapse
panel; focus restored by id on add/remove; a polite atomic results summary + static
per-row text; 44px touch targets; a `flex-col md:flex-row` 375px reflow; all lucide
icons `aria-hidden`.
