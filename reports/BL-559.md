## THE FIX

**Three functional lines in `lamatok.ts`.** The gone verdict stops depending on how the URL is written.

```diff
- : res.status === 404 && urlClass === "video" && !isRegionBlock
-   ? "gone"
+ const isDefinitiveNotFound =
+   res.status === 404 && /MediaNotFound/i.test(text) && !isRegionBlock;
  ...
+ : isDefinitiveNotFound
+   ? "gone"
```

The old gate required `urlClass === "video"`. `classifyTiktokUrl` returns `"unknown"` for every
`vm./vt.` short link, and BL-557 measured **590 of 854 live TikTok clips (69%) as short links**. So a
definitive 404 on a short link was downgraded to `"transient"` and escalated to a billed clockworks
actor. BL-557 measured that at **77% of clockworks spend, $0.3089 of $0.4030, ~$9/month**, with
clockworks' own output returning `"Post not found or private."` 88 times.

**A 404 MediaNotFound is a fact about a POST, not a URL FORMAT.** It is now gated on two required
conditions, so nothing ambiguous is widened:

1. **status 404 AND the body says `MediaNotFound`.** This is the safety linchpin, see PART 2.
2. **no region / cross-border marker** (a geo-block might still be readable by Apify's residential
   proxies, so it stays transient).

**Unchanged, deliberately:** `urlClass === "photo"` still returns `"carousel"` first, and every non-404
(429, 5xx, timeout, empty 200) stays `"transient"`. The safety net survives every ambiguous case.

## PART 0 — proven safe BEFORE widening, two independent ways

**The risk: if a short-link 404 is ever transient rather than permanent, marking it gone stalls a live
clip.** So I proved permanence before touching the verdict.

### 0a. A second vendor already confirmed it, 88 times, and we paid for the proof

We have been paying clockworks to check exactly these clips. I read Apify's run log (free), pulled each
run's INPUT, and matched the short links clockworks was fed to its own OUTPUT:

```
of the short links clockworks was fed that LamaTok 404'd:
  15 x  "Post not found or private."   (clockworks, a SECOND vendor, agrees they are gone)
   1 x  RESOLVED_LIVE                  (the fresh clip, see below)
```

### 0b. Independent live probe, including resolving the short link to canonical

I probed all 8 unique short links live, and for each also resolved `vt.tiktok.com/...` to its canonical
`/video/` URL and re-checked, in case the classification was the only thing wrong:

```
  7 x  lt=404 mediaNotFound=true  canon=404   <- genuinely gone, canonical also 404
  1 x  lt=200 mediaNotFound=false canon=200   <- LIVE, resolves normally
```

**SHORT LINKS THAT RETURNED A 404 AND ARE ACTUALLY LIVE: 0.** The one live short link returned LamaTok
**200**, not a 404 — it is `vt.tiktok.com/ZSXPCSW3W`, the 92-second-old fresh clip BL-557 identified.
**The verdict gate only runs on `!res.ok`, so it never sees a 200.** My PART 0 script initially printed
STOP on that single live URL; I did not act on it blindly, I disambiguated status, and a live 200 is
not a false 404. The widening is safe.

### BL-550's sample DID cover this population

its 99 `gone_404` were not urlClass-filtered and Apify resolved 0 of 12. That is supporting evidence;

## PART 2 — the slideshow gap is untouched, and the distinguisher is clean

BL-557 measured 21% of clockworks spend as the legitimate slideshow rescue: LamaTok cannot resolve
TikTok slideshows and Apify rescues them. **A slideshow is a LIVE post; classifying one as gone would
zero a real clip.**

**How a slideshow presents from LamaTok, probed live:** HTTP **500 `AppException`**. Never a 404, never
`MediaNotFound`. A deleted post presents as HTTP **404 `MediaNotFound`**. **That single field cleanly
separates them**, and it is exactly why the fix requires `MediaNotFound` in the body rather than trusting
status 404 alone. Even a slideshow arriving as a short link (classed `unknown`) cannot be marked gone,
because it is a 500, not a 404.

Belt and braces: `urlClass === "photo"` still hits the `carousel` branch **first**, before any 404
logic, so a canonical `/photo/` URL is protected by shape too.

**Does LamaTok have a slideshow endpoint?** Not one I found on the `v1/media/by/url` surface (it 500s).
Not investigated further; it is out of scope and left for its own round.

## PART 3 — fail-open honoured, proven

A gone verdict returns `stats: null` and skips Apify at $0. **It writes no 0 anywhere.** The verdict
function performs no DB write at all; tracking keeps the last-known value on null (BL-543). Proven in
PART 4: a deleted short link returns `stats: null`, and the stored view counts of live short-link clips
are untouched.
