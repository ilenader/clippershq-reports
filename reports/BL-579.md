# BL-579 — Cheapest way to fully replace Apify, including slideshows (READ ONLY)

**Verdict up front:** Apify CAN be driven to zero including slideshows, and the cheapest method is a
vendor already paid for. LamaTok's `/v1/media/by/id` endpoint (which we do NOT currently call)
returns a TikTok slideshow's `playCount` with HTTP 200, on the exact posts where the currently-used
`/v1/media/by/url` returns HTTP 500. No new vendor is needed for TikTok slideshows. Instagram has no
genuine slideshow gap: HikerAPI already reads carousel video views (the carve-out to Apify is
self-imposed, not a capability limit), and image-only carousels have no view concept to resolve. The
one real caveat is PART 4: moving slideshow views to a different endpoint is a PRICING change under
BL-544's display-rounding rule, so it must pass a stratified shadow proof before any flip.

## Probes run (all disclosed; NO Apify actor run)
1. Supabase reads via `scripts/run-select.js` (our DB, read-only, $0). Timestamps cast `::text`.
2. LamaTok `/v1/media/by/url` on 2 real slideshows: both HTTP 500. LamaTok bills SUCCESS only (server
   errors are free, per `lamatok.ts:163`), so these cost $0.
3. LamaTok `/v1/media/by/id` on 2 slideshows plus 2 alternate-path attempts (4 successful 200s + 2
   404s) from the prepaid quota. At roughly $0.0006/request (balance/remaining, see PART 3) about
   $0.003 total.
4. LamaTok `/sys/balance` once (status, prepaid, effectively $0).
5. TikHub `/api/v1/tiktok/web/fetch_post_detail` on the SAME 2 slideshow ids: about $0.0012 each,
   about $0.0024 total. Well under the owner's TikHub ceiling (view probes, a handful of clips, no
   profile calls).
6. HikerAPI `/v2/media/info/by/code` on 2 IG `/p/` posts (prepaid, effectively $0).

Total external probe spend well under $0.01. No Apify actor was run. Code read from `main` @ `94f3cb17`
in an isolated worktree; nothing was modified. Only this doc was added.

`imagePost=true` confirms both are genuine slideshows; `stats.playCount` carries the view count. The
media id is present verbatim in a `/photo/<id>` URL, and for a short link it is recoverable by
following the redirect first. Alternate paths `/v1/media/info/by/id` and `/v1/photo/by/id` both 404

**TikHub `/api/v1/tiktok/web/fetch_post_detail` on the SAME ids: YES, and it AGREES exactly.**
```
id 7644087836953660693  HTTP 200  play_count=3300000   (== LamaTok by/id)
id 7635742500669377805  HTTP 200  play_count=2600000   (== LamaTok by/id)
```
Both vendors return byte-identical view counts on both slideshows (matched ids, not row counts). At
these magnitudes (millions) the value is display-rounded to two significant figures on BOTH vendors,
i.e. the rounding is TikTok-side here, not vendor-specific.

**HikerAPI on Instagram `/p/` posts:** probed 2. One was a single video (`media_type=2`,

HikerAPI already covers every IG shape that HAS a view count. The IG carousel carve-out to Apify is a

## PART 4 — the display-rounding landmine (the shadow proof any swap MUST pass first)

BL-544 proved 84.1% of TikTok clips at or above 10k views are display-rounded and different endpoints
round differently. A clipper's earnings are computed from the view count, so moving slideshow views
from clockworks to LamaTok `by/id` is a PRICING decision, not a neutral swap. Our 2-clip probe is
reassuring (LamaTok `by/id` == TikHub exactly, both at millions where rounding is TikTok-side) but 2
clips is not a proof.

**Required shadow proof BEFORE any flip (no blind endpoint change):**
1. Run LamaTok `by/id` in SHADOW next to the current clockworks value for slideshows, matched by
   media id (never by row position, BL-550's trap).
2. Stratify by the 10k boundary: at least 100 slideshow clips BELOW 10k (where exact values matter
   most) and at least 100 AT OR ABOVE 10k. If the live slideshow population cannot supply 100 per
   stratum (PART 1 suggests it may not), run the maximum available and HOLD the flip until the strata
   fill, rather than flipping on a thin sample.
3. Pass criteria fixed PER STRATUM in advance (e.g. below 10k: exact equality on the shadow set; at
   or above 10k: equality after both are normalized to the same rounding, since the divergence there
   is TikTok-side). Zero null-versus-real: LamaTok `by/id` must never return null where clockworks
   returned a real number for the same id.
4. Only after both strata pass does the `by/id` result become the write value for slideshows.

## PART 5 — verdict and exact ordered next step

**Can Apify reach zero including slideshows? Yes.** TikTok slideshows resolve on LamaTok `by/id`
(already paid, same per-call as every TikTok poll, verified live on 2 ids and corroborated exactly by
TikHub). Instagram needs no new vendor (HikerAPI already reads carousel video views; image-only
carousels have no views). So the last structural reason Apify exists (the ~21% TikTok slideshow gap)
is closable with a vendor already in the stack.

**Is it worth doing versus leaving the sliver on Apify?** The dollar alone is marginal (slideshows are
well under $1/month today). But the owner's goal is Apify GONE entirely, the fix reuses an already-paid
vendor at roughly one seventh of clockworks' per-poll price, and it removes the only remaining
TikTok dependency on Apify. So it is worth doing, gated on the PART 4 proof. It is NOT worth
integrating a NEW vendor (TikHub) for slideshows when LamaTok already answers.

**Exact ordered next step (build-prompt ready), probing the already-paid option first:**
1. In `lamatok.ts`, add a `by/id` FALLBACK that fires ONLY when `urlClass==="photo"` (or when
   `by/url` returns 500 AppException on a resolved `/photo/` id): extract the numeric id from the
   `/photo/<id>` URL (for a short link, resolve the redirect to the canonical `/photo/<id>` first),
   call `/v1/media/by/id?id=<id>`, and map `stats.playCount` under the SAME never-zero/null rule the
   file already enforces. Do NOT change the `by/url` path for videos.
2. Ship it SHADOW first (log LamaTok `by/id` slideshow views next to the clockworks value; write
   nothing to earnings) and run the PART 4 stratified proof. This interacts cleanly with BL-576's
   gone counter: a slideshow is 500-never-404, so it never enters `goneClipIds` and the counter is
   unaffected either way.
3. Only after the proof passes, flip slideshows to LamaTok `by/id` as PRIMARY (Apify no longer
   called for them). At that point TikTok has zero Apify dependency; IG's residual Apify spend is a
   SEPARATE problem (the single-fetch `null_views_on_200` path, BL-577), not slideshows, and is the
   next target for driving Apify to zero.

Note: the IG single-fetch `null_views_on_200` residual (BL-577) and this TikTok slideshow gap are the
two distinct remaining Apify dependencies. Closing both drives Apify to zero. This round closes the
slideshow one with an already-paid vendor.
