**The code computes his streak to 15, the same as stored.** I traced his real data through the real evaluation (`gamification.ts:261 updateStreak`, `:211 evaluateDayByBounds`, `:402 getStreakDayStatuses`) with a read-only reproduction that issues the identical queries and never writes:


**Why 15 and not 16:** his two clips dated 2026-07-02 were submitted at 18:50 UTC, which is 2026-07-03 00:20 IST. In his own timezone they land on 07-03, not 07-02. So his streak legitimately runs 07-03 through 07-16 (14 days) plus today 07-17 (todayActivity true) = **15**. This is exactly correct, and it is the streak he says he had.

**There is no line at which his streak becomes zero.** Every candidate the brief listed, checked against his data:

* **Timezone consistency between eval and grid.** Both `evaluateDayByBounds` (`:211`) and `getStreakDayStatuses` (`:402`) read `createdAt` and gate through the same `streakEligibleWhere()` fragment (`owner-share-policy.ts:103`), both using `user.timezone`. My independent Postgres recompute (in his tz) reproduced the stored value written by the JS code, so the two implementations agree. No drift.
* **The backdateVerifiedAt gate leaking onto ordinary clips.** It does not, and it cannot. `streakEligibleWhere()` returns `{ OR: [{ isOwnerOverride: false }, { backdateVerifiedAt: { not: null } }] }`. An ordinary clip (`isOwnerOverride = false`) matches the first branch **regardless of any timestamp**. The gate only ever ADDS eligibility for a platform-verified backdated override; it never REMOVES eligibility from an ordinary clip. His clips are all `isOwnerOverride = false`, so all are eligible.
* **HikerAPI's null timestamp breaking an ordinary clip.** Irrelevant to the streak. `backdateVerifiedAt` is stamped only at owner-submit for override clips; ordinary clips never carry it and never need it for the streak. HikerAPI returning null (or, as I found, a real `taken_at`) changes nothing for an ordinary clip's streak eligibility.
* **owner-submit-core.ts:234's createdAt overwrite.** That overwrite applies only to owner-override clips (the `postedAt` backdate path). His clips are ordinary with `postedAt = null`, so their `createdAt` is the genuine submit time. Untouched.
* **Which field the streak reads.** Both the eval and the grid read `createdAt`, consistently. There is no path reading `postedAt` or `approvedAt` for the streak. No divergence.

**Why the app shows him 15, live.** The display path `computeGamificationState` (`gamification.ts:518`) calls `updateStreak(userId)` (`:556`) and then re-reads `currentStreak` (`:559-563`) on every load. So each time he opens the app it recomputes from his full history and shows 15. There is no code path that yields 0 for him while those clips exist, because his yesterday (07-16) has an APPROVED, streak-locked clip, which evaluates to "passed."

**Most likely explanation for what he saw:** a transient client-side or PWA cache showing a stale value, or a momentary view during the BL-541 deploy window that the self-healing recompute-on-read immediately corrected. His stored value could never have been written as 0 by `updateStreak` given his clip history. This is a perception or cache artifact, not a data loss.

## PART 4: the root-cause fix

**There is no root cause to fix, because there is no defect.** The boundary the brief wanted protected is already intact and proven:

* An ordinary self-submitted clip never needs a platform timestamp to count for a streak. `streakEligibleWhere()` includes every `isOwnerOverride = false` clip unconditionally. Verified in code and in data: all 3,826 ordinary clips carry `isOwnerOverride = false` (the column is `Boolean @default(false)`, non-nullable; zero nulls exist), so none can fall through the `{ isOwnerOverride: false }` match.
* BL-541's anti-manufacturing rule is intact and I did not weaken it: a backdated override clip still counts only when `backdateVerifiedAt` is set by a platform-confirmed post date. The 22 clips backdated more than a day still cannot fabricate a streak.
* The eval and the grid still share the one fragment (`streakEligibleWhere()`), so they cannot drift. BL-563, which I built on top of, touched only the owner-earnings gross gate (lines 26 and 867 to 899) and did not go near the streak code (221, 402, 641), so that shared-fragment guarantee survives BL-563.

Making a code change here would be inventing a fix for a defect that does not exist. The most defensible protection against future regression is a test asserting that an ordinary clip is always streak-eligible regardless of `backdateVerifiedAt`; I did not add it in this round because it is a net-new test-infra decision and the boundary is already correct and proven, but I recommend it as the one cheap hardening if the owner wants a permanent tripwire.
