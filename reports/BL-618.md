**2026-07-22 · SHIPPED to branch `checkpoint/BL-618` (@ `9f7e1c4`, verified on origin). Base:** origin/main `c784ec33`. Tags `pre-BL-618` / `post-BL-618`. This touches the submit path, so every BL-601 money trap was re-proven on merged main BEFORE the gate came off.

## PART 0 — the root cause, proven before any change
**The flag was NOT on the wrong account.** Queried the live database: three users hold `isTestUser=true` right now, and one of them is the account the owner logs in as:

| id | role | handle | isTestUser |
|---|---|---|---|
| cmoobbldj… | CLIPPER | testclipper_26758 | true |
| cmobu3f02… | CLIPPER | 1jackclips | true |
| cmn4m6lh6… | OWNER | Ankara | true |

testclipper_26758 (the owner's TEST CLIPPER login) has `isTestUser=true` and has joined 4 campaigns with 3 approved accounts, so BatchSubmitSection's own `joinedActive.length === 0` null-guard is not the cause either.

**The real cause is placement plus a session/deploy dependency.** `/clips` is the only submit surface (no second submit route). It renders two DISTINCT things: (1) the single-link submit **Modal** (`SubmitClipPremium`) that opens when the clipper taps "Submit a clip"; and (2) `BatchSubmitSection` — the collapsible "Add several clips at once" accordion, rendered BELOW the clip list, that holds the plus button. On main these were separate, and the accordion was wrapped in `{realIsTestUser && (…)}`, where `realIsTestUser = Boolean(session.user.isTestUser)` reads the **session JWT**, which is stale until the user logs in again, and additionally depends on the just-merged BL-616 deploy being live on Railway. So a clipper who taps "Submit a clip" gets the single-link modal with no plus button, and the batch accordion only appears when the session flag happens to be fresh AND the page is scrolled to it. That is exactly "sees only the single-link form with no plus button." (I cannot read Railway's deployed commit from here; the code is correct on main, so the fix below makes the feature independent of both the session flag and deploy timing.)

**Every gate that had to be true for the plus button (before this change):** (a) `isTestUser` truthy in the session JWT; (b) the clipper has joined an active campaign (`joinedActive.length > 0`); (c) the accordion expanded; (d) a campaign selected with at least one eligible account. Removing (a) is the fix.

## PART 1 — opened to every clipper
The `isTestUser` gate is removed in all three places, leaving a normal authenticated-clipper check scoped to the caller's own accounts and campaigns:
* **UI** (`page.tsx`): `BatchSubmitSection` now renders unconditionally in the live premium layout every clipper hits; the unused `realIsTestUser` const is deleted. It still returns null on its own when the clipper has joined no active campaign.
* **`POST /api/clips/batch`**: the test-user 404 gate at line 48 is gone; the only gate is `canActAsClipperFromSession` (any authenticated clipper). Every row is still scoped to the caller's OWN accounts + OWN campaigns by the shared clipper core.
* **`/api/clips/batch/draft`**: the same 404 gate removed; drafts stay scoped per (clipper, campaign).
* **Single-submit is untouched** — `SubmitClipPremium` is not in the diff — so anyone who prefers the single-link form keeps it exactly as-is. The batch UI is an addition, never a replacement.

## PART 2 — the BL-601 money traps, RE-PROVEN on merged main (before opening the gate)
A live read-only core trace against real testclipper accounts (failure cases return before the write transaction, so nothing was written — `clips before=6 after=6`):
```
A) TikTok account + Instagram URL  -> "This URL is from Instagram but your selected account is for TikTok"
B) Instagram account + TikTok URL  -> "This URL is from TikTok but your selected account is for Instagram"
C) Instagram account + IG URL on a TikTok-only campaign -> "This campaign only accepts TikTok clips. Your link is from Instagram"
   clips for testclipper BEFORE=6 AFTER=6  (UNCHANGED — no write)
```
* **TRAP 1 (per-row attribution):** A resolves the TikTok account, B resolves a DIFFERENT (Instagram) account — each row resolves its OWN `clipAccountId`, never one account bound to the batch. Case-insensitive RULE 1.
* **Server-side platform constraint:** C is rejected by RULE 1b before any write (the batch route also runs its own `campaignAllowedPlatforms` guard before the core).
* **TRAP 2 (clipper core only):** `processOwnerSubmitLink` is referenced **0 times** in both `batch/route.ts` and `batch/draft/route.ts`; `processClipperSubmitLink` is the only submit path (no force-approve, no earnings write, no placeholder minting).
* **Server-side per-campaign limit before any write** (the `todayCount` pre-check), one campaign per batch; **in-batch dedup + DB dedup** reject duplicates so nothing double-posts.
* **Byte-identical per-clip state:** the clipper core, `clips/route.ts`, the 6 money files, `tracking.ts` and `campaign-era.ts` are byte-identical by blob OID vs origin/main (not in the diff), so a batched clip lands in the identical PENDING Clip + first ClipStat + active TrackingJob + frozen CPM state a single submit produces, and tracking, earnings, verification and the admin review queue all render it the same.

Every trap holds, so the gate was safe to remove.

## PART 3 — what a real clipper now sees
The plus button ("Add clip") under the rows adds a row; each row has a platform select (constrained to the campaign's accepted platforms), a connected-account select (constrained to that platform), and a link field, plus a plain "This link will attach to @handle on {Platform}" reminder. Rows are collapsible. Submit is blocked with a specific reason on any incomplete, wrong-platform-for-its-row, disallowed-platform, unconnected-account, duplicate, or over-limit row. Drafts persist per clipper per campaign and re-validate every restored row against the current platforms/accounts/limit (BL-615), clearing only on explicit cancel or a successful submit. `BatchSubmitSection` is unchanged from BL-615 (accessibility-lead reviewed there; byte-identical here), so this ships no new UI markup, only the gate removal.
