## PART 1 — the endgame, traced
Bonuses (level+streak+PWA, capped 25% / 30% override) are added ON TOP of the per-CLIP `maxPayoutPerClip` (earnings-calc.ts:169-185), but are strictly INSIDE the pool cap. Two independent pool mechanisms bound the crossing tick:
1. Per-tick budget cap, inside a Serializable tx (tracking.ts:2335-2534): computes committed `spent`, `remaining = budget - otherSpent`, then TRUNCATES the crossing clip. `remaining<=0` keeps current (no increase, 2521-2524); `total>remaining` ratio-caps clipper+owner to exactly `remaining` with a residual adjust so the sum never exceeds it (2525-2534). Guarantee path caps clipper at `(1-s)*budget` headroom, trimming BONUS FIRST (2494-2507), then re-derives owner via the BL-163 owner-lock `owner = clipperFinal * s/(1-s)` (2563-2588) so both pools exhaust together at exactly `budget`.
2. L1 hard-lock in writeClipEarnings (clip-earnings-writer.ts:199-257) reads COMMITTED spend and THROWS (rejects the whole increment) if already over budget or `projected>budget`. The BL-167 clamp (311-382) additionally trims any non-tracking write to `clipperPoolCap` headroom, scaling base+bonus together.

Cap is applied PER CLIP, PER TICK, against committed state. Tracking is per-campaign SEQUENTIAL, so no same-campaign concurrency; the crossing clip is truncated, never overpaid. Overshoot is impossible in the cron path beyond sub-cent float; residual risk is only a rare non-cron burst (<= one clip's CPM), auto-paused next tick. Ghost fee is NOT a runtime entity (no fee column exists) — see risk #1. Live evidence: STRAENGE landed $2,998.10 / $3,000 (under $1.90, NOT over); 0 of ~180 campaigns show realized spend > budget.
