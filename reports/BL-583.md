## PART 4 — the display-rounding landmine and the exact proof required

BL-544: 84.1% of TikTok clips >=10k are display-rounded, endpoints round differently (16,466 exact vs
16,500 vs 16,300 stored), and <10k is served exact. Moving slideshow views to by/id is therefore a
PRICING decision, not a neutral swap.

**The stratified shadow proof that must pass before any flip:**
* >=100 slideshow clips in the <10k stratum AND >=100 in the >=10k stratum, matched by id (BL-550's
  trap: a probe once returned an id never requested).
* Pass criteria fixed PER stratum in advance: <10k requires exact by/id-vs-today equality (exact
  numbers matter there); >=10k requires equality after both are normalized to the same rounding (the
  divergence there is TikTok-side, and my 5 samples plus BL-579's TikHub agreement suggest by/id
  reproduces today's rounded value).
* Zero null-versus-real: by/id must never return null where today's path returned a real number for
  the same id.
* 7+ days of matched data.

**Is there enough data yet? No, and it cannot be reached at the current population.** The shadow logs
to console (unreadable here) and fires for only 5 REJECTED clips, all >1,000,000 views. So the <10k
stratum has ZERO candidates in the tracked slideshow population and the >=10k stratum has 5. Even
running for months, ~5 low-cadence clips will never approach 100 per stratum. **The proof, as
specified, is currently unfillable, not merely unfinished.** The population, not the vendor, is the
limiting factor.

## PART 5 — the verdict and the ordered path

**Can Apify reach TRUE zero on slideshows and carousels using only already-paid vendors, with no
view-count quality loss?**
* **Coverage: yes.** LamaTok by/id resolves the TikTok slideshow class (5/5, deleted -> 404 clean) and
  HikerAPI reads IG carousels (mixed via video-sum in code; image-only correctly null because no
  metric exists). Both vendors are already paid (prepaid quotas, balances effectively unmoved).
* **Quality/pricing: yes for the >=10k stratum, unproven for <10k.** by/id reproduces the same
  display-rounded value today's path yields at high views (5 samples plus BL-579's exact TikHub
  match), so the >=10k flip is very likely earnings-neutral. The <10k stratum is unproven but currently
  drives $0 (there are no APPROVED slideshows at all, let alone <10k ones).
* **The one genuine non-coverage item:** image-only IG carousels have no view count in existence. That
  is not an irreducible Apify sliver; it is a "the metric does not exist" case whose correct fix is to
  STOP escalating them to Apify (they cannot earn a view CPM), not to buy a vendor.

**Projected final Apify cost for slideshows and carousels: ~$0.** No irreducible vendor sliver
remains; the only residue is the pricing-proof formality for FUTURE approved slideshows and a code
skip for image-only carousels.

**Exact ordered path:**
1. **Make the shadow readable and complete.** Change BL-580's shadow from console-only to a ledger/DB
   row (matched id, by/id playCount, today's clockworks value, view-bucket), AND fire it for
   short-link slideshows by resolving the redirect to the `/photo/<id>` first. Without this the proof
   can never be read or reach significance. This is the true next build.
2. **Accumulate matched-id, stratified shadow data for 7+ days** and evaluate against the PART 4
   criteria. Because the population is tiny, expect the >=10k stratum to fill slowly and the <10k
   stratum possibly never.
3. **Flip condition, pragmatic:** since 0 APPROVED slideshows exist today, the earning risk of flipping
   the >=10k slideshows is currently zero, and by/id reproduces today's rounded value, so the >=10k
   flip can go live behind the existing never-zero/NULL safety once even a modest matched sample
   confirms agreement. Hold the <10k stratum until a real sample exists; its impact is $0 until an
   approved <10k slideshow appears.
4. **IG:** widen the HikerAPI overlay to trust carousel_mixed video-sum (a smaller pricing-adjacent
   proof, since summed per-child play_count is itself rounded at >10k), and add a code skip so
   image-only carousels stop escalating to Apify (no metric exists to fetch).
5. **Result:** with the shadow made queryable and the two small code changes, Apify reaches zero on
   slideshows and carousels. State plainly: no slideshow sub-type in the probed set is uncoverable; the
   only genuinely uncoverable object is a view count that does not exist (image-only carousels), which
   is handled by not polling, not by paying.

Honest caveat: low-view, music-only, and region-locked slideshow sub-types are UNVERIFIED (no sample
in the tracked population). The path above surfaces them via the queryable shadow before any flip
rather than assuming them covered.
