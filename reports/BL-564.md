# BL-564: why 114 clippers cannot verify their accounts, settled by live probe

**AUDIT ONLY. READ ONLY on code, config, data and money. Nothing changed, fixed, pushed, or merged.**
**Base:** origin/main @ `9e13650a` (post BL-538) · **Branch:** checkpoint/BL-564 · **Probe cost: ~$0.01, all LamaTok prepaid quota. No Apify actor run. No rate limit hit.**

---

## THE ANSWER, IN ONE PARAGRAPH

It is a MIX, and both halves are now proven from live data, not guessed. **The verification code is genuinely in the bio for real stuck clippers, and our cascade cannot see it** (the top escalator's entire bio is literally their code `AA5Z`, and the 4-tier HTML cascade missed it 14 times). **A structured bio source that reads it exactly already exists and we already pay for it**: LamaTok's media response carries `author.signature`. But **it is not the whole story**: many escalating accounts do not have the code in their bio, 12 of 151 accounts have a profile link so malformed it can never verify, and 78% of escalated accounts eventually reach APPROVED on their own. So this is partly our defect and partly people not adding the code or entering a broken link. The single highest-value first move is not a scraper: it is to persist the fetched bio at attempt time so this stops being unanswerable, validate the profile link at entry, and check one free thing on YouTube. Full reasoning below.

---

## PART 0: BL-552's diagnosis: verified, and it holds, with one correction

BL-552 said Instagram reads a structured `biography` field while TikTok and YouTube scrape HTML and cannot reliably see a client-rendered bio, and that the fix is a structured source, not more tiers. **Measured against current main, this holds:**

* **Instagram bio is a structured Apify/HikerAPI field.** `verify/route.ts:233` calls the IG profile actor directly; BL-552 proved HikerAPI `GET /v2/user/by/username` returns `biography`. Instagram escalates **zero** times.
* **TikTok tier-1 is a plain HTTP GET that gets a bot wall.** I fetched two stuck TikTok profiles with a browser User-Agent and got back **exactly 1462 bytes** each: a login/stub page containing neither the bio nor the code. That is why the scrape fails at tier-1.
* **YouTube tier-2 reads a structured field but through a key that appears unset.** `verify-cascade.ts:337` reads `process.env.YOUTUBE_API_KEY`; it is **absent from `.env.local`**. See PART 2.

**The one correction:** BL-552 framed the fix as "a structured bio source for TikTok, exactly like Instagram already has." That is only half-available. LamaTok's structured bio field exists **on a media (video) response**, not on a profile lookup: LamaTok's user endpoint is `NotImplementedYet` (PART 2). So the structured source exists, but reaching it at verify time is harder than BL-552 implied, because verification happens against a profile URL, often before any video exists.

R-5's diagnosis (`cappedAtTier1`, missing TikTok tier-2) was already disproved by BL-552 and I did not re-inherit it. Confirmed: `cappedAtTier1` is a rounding error (7 of 326) and TikTok runs 3 to 4 tiers.

## PART 1: SETTLED: is the code actually in their bio today?

I took the accounts that escalated most and probed each profile **live, right now**, reading the structured bio where a video existed (LamaTok `author.signature`) and the raw profile otherwise.

| account | code | escalations | status | structured bio today | code present? |
|---|---|---|---|---|---|
| **abhijot4993** (TikTok) | `AA5Z` | 14 | APPROVED | `"AA5Z"` | **YES, exact. Our cascade missed it 14 times.** |
| coldmind.exe (TikTok) | `6R6U` | 13 | APPROVED | `"discipline over feelings\nstay cold"` | no |
| cariaga.velasco (TikTok) | `V7U8` | 6+ | APPROVED | `null` (empty bio) | no |
| topfrag_tv (TikTok) | `WXHN` | 8 | APPROVED | plain fetch returned a 1462-byte bot wall | undeterminable by scrape |
| pitchhoney (TikTok) | `F638` | 7 | REJECTED | link is **malformed** (see below) | cannot fetch at all |

**The abhijot4993 case is the proof the defect is real.** Their entire TikTok bio is the four characters `AA5Z`. LamaTok's structured field returns it instantly and the production match function finds it. Yet the account logged 14 `NEEDS_ADMIN` escalations, meaning the live 4-tier HTML cascade (`tiktok-plain-fetch`, `tiktok-apidojo-profile`, `tiktok-browserless-content`) failed to see four plain ASCII characters, over and over. This is not the clipper's fault. At least some of the 114 are stuck through a genuine platform defect.

**But it is genuinely a mix.** coldmind.exe's live bio is an unrelated motto with no code; cariaga.velasco's bio is empty. Both are now APPROVED, so they may have added the code, passed, then removed it. The honest limit, exactly as BL-552 warned: **the bio text is not persisted at attempt time, so for an APPROVED account I cannot distinguish "never added it" from "added it, passed, removed it."** For abhijot4993 the ambiguity does not apply, because the code is still there.

### Does the 78%-eventually-APPROVED pattern support transient failure or eventual compliance?

Both, and they are the same story. Of **151 escalated accounts** (107 distinct clippers, 336 escalations in 30 days): **118 are now APPROVED (78%)**, 26 PENDING, 7 REJECTED. **No admin approval action exists in `audit_logs` for any of them** (queried directly: zero rows), so these self-resolved through the automated cascade eventually succeeding, not through a human. The signature of one clipper retrying **14, 21, 13, 11 times** and then passing is exactly what you get when the code IS in the bio but the cascade sees it only intermittently (the browserless tier is flaky), so the clipper hammers retry until a tier happens to catch it. abhijot4993 is the worked example: code present throughout, 14 failures, eventual pass, code still there.

## PART 2: is there a structured bio source for TikTok or YouTube? Probed live.

### TikTok: YES on a video, NO on a profile

LamaTok's media endpoint (`/v1/media/by/url`, already used in production for view counts) returns a full `author` object. Probed live against abhijot4993's video:

```
GET https://api.lamatok.com/v1/media/by/url?url=<abhijot4993 video>   ->  HTTP 200
  author.signature = "AA5Z"          <-- the bio, structured, exact
  author.uniqueId  = "abhijot4993"
  author.nickname  = "Abhijot"
```

`author.signature` **is** the TikTok bio, returned as a clean structured string with newlines intact (coldmind.exe came back `"discipline over feelings\nstay cold"`). This is the structured source BL-552 wanted, and **we already pay for it and already call it.**

**The catch, probed and confirmed:** LamaTok has NO working profile lookup. Every user-endpoint shape I tried:

```
GET /v1/user/by/username?username=topfrag_tv   ->  HTTP 501  {"exc_type":"NotImplementedYet"}
GET /v2/user/by/username?...                    ->  HTTP 404
GET /v1/user/web_profile_info?...               ->  HTTP 404
GET /v1/user/info?...   /v1/user/username/...    ->  HTTP 404
GET /v1/user/medias/by/username?...              ->  HTTP 404
```

So the structured TikTok bio is reachable **only through one of the clipper's videos**, not through their profile URL. Verification today runs against the profile URL, often before the clipper has posted anything, which is why the currently-stuck TikTok accounts I pulled all had zero clips in our DB. **The structured source exists; wiring it into verification means verifying against a video URL, not a profile URL.** That is a design change, not a drop-in.

### YouTube: a structured field is read, but through a key that looks unset

`verify-cascade.ts:333-425` (`youtube-data-api-channels`) calls Google's **free** Data API `channels.list?part=snippet` and searches `item.snippet.description` for the code (`:416`). So a structured field IS read. Two reasons it still escalates 108 times:

1. **`YOUTUBE_API_KEY` is absent from `.env.local`.** When unset, the tier returns `{ found:false, transient:true, error:"no YOUTUBE_API_KEY" }` (`:337-340`) and never calls Google at all. **If it is also unset in Railway, every YouTube verification runs with tier-2 effectively dead**, leaning entirely on the plain fetch and browserless tiers. I could not read Railway's env, so I state this as a strong hypothesis to check, not a fact. It is the first thing to check because it is free.
2. **`snippet.description` is the channel's long "About" text, not the short tagline under the channel name.** A clipper who puts the code in the visible bio line rather than the About box would not be found even with the key set. I could not probe the Data API (no key available to me), so I did not verify which field these clippers used.

I fetched the three top YouTube channels' pages directly: `@Static Harbor` returned **HTTP 404** (the handle has a space, `youtube.com/@Static Harbor`, which is not a valid handle), and the other two returned full pages that did not contain the code in the raw HTML. Consistent with either "no key" or "code in the wrong field."

**Honest verdict on PART 2:** a structured TikTok bio exists but only per-video, not per-profile, so it cannot be dropped into profile-based verification unchanged. A structured YouTube description is already wired but may be inert for want of a key. Neither is the clean "flip one endpoint" fix the framing hoped for.

## PART 3: the match function is NOT the bug

`verify/route.ts:131-161` (`searchHtmlForCode`) uppercases both sides and does a substring match, with a unicode-unescape fallback (`:158`). Tested against the real bios from PART 1: abhijot4993's `"AA5Z"` matches cleanly (`full-text`). The code alphabet is deliberately unambiguous (`ABCDEFGHJKLMNPQRSTUVWXYZ23456789`, no I/O/0/1), so look-alike confusion is already designed out.

**The match works whenever it is handed the bio.** The failure is entirely upstream: tier-1 is handed a **1462-byte bot wall**, not the bio, so there is nothing to match. A match-function fix would be free, but there is no match-function bug to fix. The one residual risk the function does not cover (a zero-width space or emoji inserted *between* the code characters) did not appear in any real bio I fetched; abhijot4993's was clean and still failed, which points at fetch, not match.

## PART 4: what the 114 actually experience

The in-app message is **not** a worded dead end. On a clean miss (`verify/route.ts:926-930`):

> "We couldn't find code AA5Z in your TikTok bio. Make sure it's saved and your profile is public, then try again. Our team will also review it."

That is clear and it points somewhere. **The problem is the last sentence is an empty promise.** It queues a `NEEDS_ADMIN` audit row, but R-5 established the admin queue is unworked: support has migrated to Discord (41 open tickets vs 8 resolved, 12 clipper messages in 30 days, **zero owner replies**). So "our team will also review it" resolves to nobody. The only channel that actually does anything is the **"try again"** button, so the clipper presses it 8, 14, 21 times. The retry storm is not irrational behaviour; it is the only working control on the page.

**What they should see:** either an honest "we could not read your bio automatically, paste a link to one of your videos and we will check it" (which would also hand us the structured `author.signature`), or a worked review queue behind the promise. Right now they get a promise with no owner behind it and a retry button that mostly, eventually, works.

## PART 5: the verdict and the fix

### Are the 114 stuck because of our code or because they never added the code?

**Both, measured.** From live probes of the highest-signal accounts and the full escalated population:

* **At least one, and by inference many, ARE stuck by our defect.** abhijot4993's code is in the bio right now, exact, and the cascade failed 14 times. The 78%-eventually-pass-with-no-admin pattern only works if the code is genuinely present and seen intermittently, which means the "present but unseen" population is large.
* **Some never had the code in their bio** (coldmind.exe, cariaga.velasco), though for APPROVED accounts this cannot be cleanly separated from "added then removed."
* **12 of 151 accounts (8%) have a profile link that can never verify** regardless of vendor: doubled URLs (`tiktok.com/@https://www.tiktok.com/@king_b3rto`, `tiktok.com/@tiktok.com/@cariaga.velasco`) and handles with spaces (`@viral holli`, `@Static Harbor`). Seven of these have a literal space. This is a data-entry bug, not a scraping bug.
* **Duplicate accounts abound** (kie_lyricxz + kiel_yricxz, zebraclips + zebraclipssss, dhalialyrics + dhalialyrics16), the fingerprint of clippers giving up and re-registering with a new handle.

So the honest answer is not the comfortable "they just did not add it." A real defect exists and is proven. But it is entangled with people not adding the code, entering broken links, and an unworked review queue.

### The fixes, ranked by value

1. **Persist the fetched bio and the match result at attempt time.** One column, written from data we already fetch. It costs almost nothing, it makes "never added" versus "cannot see" permanently answerable instead of a live-probe archaeology exercise, and it is the prerequisite for justifying any scraper investment. This is the thing BL-552 and this round both had to work around.
2. **Validate and normalize the profile link at entry.** Strip a pasted full URL, reject spaces, collapse the doubled `@https://` case. Free, pure win, clears 8% of the stuck population who can never pass today.
3. **Verify TikTok against a video URL, reading LamaTok `author.signature`.** Proven exact on abhijot4993, uses a vendor already paid, reliable where the profile scrape is not. Clippers are people who post videos, so asking for one is natural. This is the real engineering fix and it is sound, but it is a flow change (verify a clip, not a profile), so it is not first.
4. **Check `YOUTUBE_API_KEY` in Railway; set it if missing.** Free, no code. If unset, it may clear a large share of the 108 YouTube escalations at once. If set, the next question is whether `snippet.description` is the field clippers actually use.
5. **Put an owner behind the review queue, or stop promising it.** The message says "our team will review it" and no one does. Either work the `NEEDS_ADMIN` queue or change the copy so it does not promise a channel that is dead.

### Recommend ONE thing first

**Persist the bio at attempt time (fix 1), together with entry-time link validation (fix 2).** Not a scraper. Here is why this and not the structured-source project: the structured TikTok source is real but per-video-only, so shipping it is a verification-flow redesign that should not be started until we know how many of the 114 are the abhijot case (code present, unseen) versus the coldmind case (no code) versus the malformed-link case. Persisting the bio answers that within days of real traffic, at almost zero cost and zero risk, and link validation immediately unblocks the 8% who literally cannot pass. Then, with the split measured rather than inferred from a handful of live probes, decide whether the video-based structured check is worth the flow change. **Do not start the engineering project before the one-column measurement that tells you how big the problem it solves actually is.**

The one thing to say plainly to the owner: **this is not purely "people are not adding the code." A genuine defect is proven** (abhijot4993, code present, 14 failures). But the cheapest correct first step is still measurement plus link validation, because the current data cannot size the defect precisely and a scraper built on a guess is how R-5 got the diagnosis wrong in the first place.
