# BL-571 — the third gone-leak: diagnosed, NOT patched (and why the patch was wrong)

**2026-07-17 · READ ONLY on code, config, data, money. Nothing shipped. The one code edit I drafted

## Bottom line
BL-567's leak MECHANISM is correct: during a HikerAPI rate/cooldown burst the per-tick gone-skip is
suppressed and due deleted reels escalate to billed Apify actors. But BL-567's decomposition into a
"low-risk PART 1 worth most of the money" + "risky PART 2 retirement" is WRONG. **The PART 1 gone-check
at apify.ts:616 cannot be shipped as an effective standalone: any live HikerAPI check there is silenced
by the SAME cooldown that caused the leak, so it is redundant with the gone-skip already at apify.ts:2182.
An effective 616 skip must read a PERSISTED gone-state — which is PART 2. PART 1 and PART 2 are one fix,
not two.** I shipped nothing and I am not claiming a saving I did not make.

## PART 0 — the mechanism is real (BL-567 confirmed)
The 4 prior fixes are on main and deployed (BL-543 cascade, BL-551 ledger, BL-554 IG batch gone-skip,
BL-559 LamaTok TT). The residual is the per-clip single fallback `fetchInstagramStats` (apify.ts:616),
reached only via `fetchClipStats` (the sole caller, apify.ts:2191; the function is not exported). A
deleted reel reaches it only when the overlay gone-skip at apify.ts:2182 did NOT fire — i.e. when
`tryHikerForInstagram` returned verdict≠"gone". Live-ledger confirms the overlay is ON and working:
**3,091 hikerapi-v2 IG single calls in 48h** (1,944 ok / 1,147 miss). So gone reels only slip past 2182
during a HikerAPI cooldown burst, exactly as BL-567 said (bursty by hour: 17:69, 01:29, 08:19).

## PART 1 — why the 616 gone-check is a no-op (the correction)
The cooldown short-circuit is `hikerapi.ts:614`: when a cooldown is active `tryHikerForInstagram`
returns `{ useResult:false, verdict:"transient" }` **without a network call**. The overlay at 2169 hits
this, gets "transient" (not "gone"), skips 2182, and falls to 616. The drafted PART 1 called
`tryHikerForInstagram` AGAIN at 616 — same process, same active cooldown, same line-614 short-circuit,
same "transient". It skips nothing. Two ways to "fix" that, both bad:
- Reuse the live verdict (what the brief specified) → redundant with 2182. Recovers ~$0.
- Bypass the cooldown with a raw by/code 404 call → recovers money BUT hammers HikerAPI during the very
  429 burst that set the cooldown, deepening the rate-limit. Self-defeating on a money cron.

The only correct 616 skip reads a REMEMBERED gone-state set on a prior clean tick, so a cooldown tick
skips Apify from memory. That memory is PART 2. **Conclusion: do not ship a 616 live-verdict check.**

## PART 2 — retirement is the real fix, and it needs a NEW signal + sign-off
BL-567's own PART 4 says this: retire the dead reel so it leaves the rotation and the cooldown burst has
nothing to escalate. The constraints, verified against schema:
- **consecutiveFailures is NOT the signal.** Gone/null is classified INFRA_DEFER (a correct answer, not
  a failure), so `TrackingJob.consecutiveFailures` stays 0 for deleted reels (confirmed: field exists,
  and there is no `lastSuccessAt` and no gone-counter). Retirement needs a NEW `consecutiveGone` column
  incremented ONLY on verdict="gone" (definitive /v2/ 404) and reset to 0 on any real resolve.
- **You cannot identify the dead set from today's DB.** The escalated-cadence reasons (DEAD_72H,
  BRACKET_ACTUALLY_DEAD, DORMANT_14D) are VIEW-PLATEAU heuristics, not 404 confirmations. BL-567 had to
  probe HikerAPI (8/8 sampled truly gone) to confirm deletion. So retirement must let the new counter
  accumulate over a burn-in BEFORE any write — no one-shot back-fill off the plateau flags.
- **Blast radius is large; a plateau-based write would be catastrophic.** Active IG jobs at ≥24h cadence,
  videoUnavailable=false: **1,410 (1,076 APPROVED).** Only a probe-confirmed minority (BL-567: ~34 in a
  90-min window, ~267 stale) are actually deleted. Retiring on the plateau flags would freeze ~1,000 live
  clippers' view growth. This is why the counter + burn-in + dry-run + owner sign-off are mandatory.

### The safe retirement design (for a dedicated round, Opus, with sign-off)
1. Additive `TrackingJob.consecutiveGone Int @default(0)` + `Clip.videoUnavailableSince DateTime?` via
   `run-schema-sql.js` (ADD COLUMN IF NOT EXISTS only). No `prisma migrate`.
2. In tracking, split DEFINITIVE-gone from transient miss (mirror BL-559): on verdict="gone" increment
   `consecutiveGone`; on any real resolve reset to 0. WRITE-ONLY the counter first; retire NOTHING yet.
3. Burn-in ≥7 days. Then dry-run: list clips with `consecutiveGone ≥ K` (K≥3, ~a week at 28h cadence),
   with id, clipper, savedEarnings, age. LOUD warning if hundreds. Owner signs off on the LIST.
4. Retire the signed-off set: `Clip.videoUnavailable=true` + `videoUnavailableSince=now`, AND add
   `clip: { videoUnavailable: false }` to the dueJobs WHERE at cron/tracking/route.ts (BL-567: marking
   the clip alone does NOT stop polling — the cron filters only isActive+nextCheckAt). Never
   `trackingJob.delete`; keep the row so it is reversible and visible.
5. **Earnings-safe by construction:** videoUnavailable freezes earnings at last-known (savedEarnings),
   never zeros them (BL-543 fail-open), and tracking auto-restores if a reel ever returns 200.
   writeClipEarnings is the only writer; the 4 invariant fields are untouched. Reversible: clear the two
   fields to un-retire.
