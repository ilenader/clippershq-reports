# What is actually deployed, and why LamaTok looks dark (BL-549)

AUDIT ONLY, 2026-07-16. Read only: no code, config, data, or money changed. Nothing pushed to main,
nothing merged, no env var flipped, no provider switched. All times UTC.

**The headline: LamaTok is not dark. It never was.** It serves about 68.5% of TikTok polls over 7 days
and 79.6% over the last 24 hours. The ledger cannot say so, because its label vocabulary predates
LamaTok and reports LamaTok's successes as `none-tiktok`. BL-547's 1.4% is an artifact of that label,
not a measurement of LamaTok.

## A measurement trap that nearly cost this audit its answer

`scripts/run-select.js` renders `timestamp without time zone` columns through a local +02:00 offset,
so every timestamp prints **two hours earlier than it is stored**. On first pass this made the newest
ledger row look like 19:01 against a wall clock of 21:06 and the tracking cron look **dead for two
hours**. It is not. Casting with `::text` gives the stored value: DB `now()` = `21:08:34`, newest
ledger row = `21:01:33`, GitHub's own clock = `21:08:26 GMT`, machine `date -u` = `21:08:30Z`. All four
agree. **Any future round reading this ledger must cast to `::text` or it will read a two-hour hole
that does not exist.**

## PART 2 — why LamaTok looks dark: the ledger cannot pronounce its name

The owner eliminated the obvious answer, and was right to. **`SCRAPER_TIKTOK_PROVIDER` is not the
cause, and setting it to `lamatok` was already redundant**: `tiktokScraperMode()` (apify.ts:364-378)
returns `"lamatok"` for *any* value that is not exactly `"apify"` or `"shadow"`, including unset. BL-488
made LamaTok the default. The env var was never the gate.

**The real cause is a labelling gap, not a routing failure.**

`fetchTikTokStatsBatch` at **apify.ts:1025-1027**, when LamaTok serves every URL in the batch:

```js
if (missing.length === 0) {
  console.log(`... result=lamatok-only`);
  return { results: lamatokMap, apidojoHits: 0, legacyHits: 0 };
}
```

`summarizeBatchProvider` at **apify.ts:1935-1958** then labels the ledger row. Its entire vocabulary is
`apidojoHits` and `legacyHits`. **It has no LamaTok branch.** So at **apify.ts:1951**:

```js
if (s.apidojoHits === 0 && s.legacyHits === 0) return noneTag;   // → "none-tiktok"
```

**A batch that LamaTok served perfectly is recorded as `none-tiktok`.** The label means "no Apify tier
hit", and a reader naturally reads it as "nothing worked".

### Each hypothesis, answered with evidence

| Hypothesis | Verdict |
|---|---|
| `LAMATOK_KEY` missing in production | **FALSE.** `isLamatokConfigured()` (lamatok.ts:33-35) gates the branch; LamaTok-only batches cannot occur unless the key is set. 5,690 clips prove it is set. Note it is **absent from this machine**, so this cannot be local leakage. |
| Branch entered but failing through | **NO.** 121 of 126 `none-tiktok` rows in 24h are `success = true`. Only 5 failed. |
| `tiktokScraperMode()` returning something else | **NO.** Proven by behaviour: the LamaTok-only branch is the only code path that yields `apidojoHits=0, legacyHits=0` with results. |
| Shadow mode sampling rather than serving | **NO.** Shadow is fire-and-forget (apify.ts:418-420) and returns Apify's stats; the ledger would read `legacy-tiktok-only`, not `none-tiktok`. |
| Batch path diverges from single path | **YES, but only in labelling.** Both prefer LamaTok. The single path writes `lamatok-tiktok` (apify.ts:396); the batch path cannot. |

**The genuine remaining gap: LamaTok misses about 30% of TikTok URLs** and Apify clockworks catches
them at $7.80 per 7 days (about **$33/month**). That miss rate, not the env var, is the real TikTok
cost. Nobody has yet classified why those 2,600 fall through.

## PART 3 — can Apify be retired? Not on evidence I can produce, and I could not run the probe

**I could not probe.** `HIKERAPI_KEY`, `LAMATOK_KEY` and `APIFY_TOKEN` are **all absent from this
machine** (`.env`, `.env.local`); only `TIKHUB_API_KEY` is present. **I ran zero live probes this
round, and therefore incurred zero probe cost.** The question the round was built to answer, whether a
different HikerAPI endpoint could resolve the clips Apify catches, **cannot be answered from here**. I
will not fabricate a result.

What I can establish by reading, and it matters:

**The obvious second endpoint is already the fallback.** `hikerapi.ts:3-8` documents that the client
already tries **both** documented v2 media endpoints: `/v2/media/info/by/code?code=<shortcode>` as
primary, then `/v2/media/info/by/url?url=<cleaned-url>` as fallback. The older `/a1/` and `/a2/` media
paths are **documented-deprecated and 404 even for accessible posts**, which is why BL-114 moved off
them. So "try a different HikerAPI endpoint" is not an unexplored idea; it is already implemented, and
the deprecated alternatives are known dead.

That is not proof that no other surface exists (HikerAPI also publishes profile and GraphQL surfaces I
have not tested), but it does mean the cheap hypothesis is already spent. **On today's evidence, Apify
cannot go to zero.** BL-544 and BL-545 both concluded it must stay, and nothing here overturns them.

**To actually answer this**, a probe round needs `HIKERAPI_KEY` available to it, a sample of the real
fall-through clips, and a disclosed per-call cost. That is a real round with a key, not an inference.

### The landmine: do not promote LamaTok to serve TikTok views at volume yet

**LamaTok already serves 68.5% of TikTok polls, so this is not hypothetical.** BL-544 proved **84.1% of
TikTok clips at or above 10k views are display-rounded**: the same video in the same minute read
16,466 exact from one endpoint, 16,500 rounded from another, and 16,300 rounded as stored. Because
earnings are `(views / 1000) x clipperCpm`, **which endpoint answers is a pricing decision, not a
technical one.**

The existing 8-sample shadow run **sits entirely below 10k views and therefore proves nothing about
the 84% cohort that would be repriced.** Before any further LamaTok promotion, a shadow proof must
pass, **stratified across the 10k boundary**:

* **Stratify explicitly**: at least 100 clips below 10k and at least 100 at or above 10k, sampled from
  live tracked clips. The 10k boundary is where rounding starts, so an unstratified sample hides the
  entire effect.
* **Pass criteria, fixed before looking**: exact match on 99% or more **within each stratum
  separately**; **zero** cases where LamaTok returns 0 or null while Apify returns a real number (that
  is the earnings-freezing failure); and **no systematic bias in either stratum** (median delta 0, not
  "LamaTok reads 20 lower on average").
* **Rounding is the thing being tested, not noise.** If LamaTok returns rounded values at or above 10k
  where Apify returns exact, that is a real repricing of every clip in the 84% cohort, and it must be
  the owner's explicit decision with the dollar impact quantified first, not a side effect of a
  provider flip.
* Read-only by construction: the run writes nothing to `Clip.earnings`.

**Switch nothing until that passes.**
