# BL-636 — Analytics Screenshot Reader: Audit (READ ONLY)

**Question the owner asked:** does the analytics screenshot reader actually work, and how accurate is it? He is right to worry: a misread decimal (4% instead of 42% retention) could get an honest clipper accused of botting.

**One line verdict:** it is NOT safe to rely on today. It is TikTok only with a real number parsing bug (K/M suffixes are silently truncated), and its real accuracy is **UNMEASURED** because OCR has never run on a single production screenshot. It should not be trusted for any platform until platform detection, the parser fix, and a real accuracy measurement are done. The one thing that IS solid: nothing it produces can change a clip's status or earnings.

No code, config, data or money was changed. This is a markdown only diff, so no build was run (a doc cannot change tsc). Probes disclosed in the last section.

---

## PART 1 — What exists today, honestly

**The path, end to end:**

1. Owner (or a `CREATOR_SCAN_REVIEW` holder) uploads a Studio screenshot in the Creator Scan Review Panel → `POST /api/admin/creator-scan/[accountId]/screenshot` (`src/app/api/admin/creator-scan/[accountId]/screenshot/route.ts:28`).
2. The route sniffs magic bytes (PNG / JPEG / WebP only, 5 MB cap), stores the image in Supabase as durable evidence via `storeScreenshot` (`src/lib/creator-scan/screenshot-store.ts:27`).
3. It then calls `runScreenshotOcr` (`src/lib/creator-scan/screenshot-ocr.ts:132`), which POSTs the image to **OCR.space free tier, engine 2** (`api.ocr.space/parse/image`, line 145). The text it returns is parsed by the pure function `parseStudioMetrics` (line 87) against one flat `EXTRACTORS` list (lines 30 to 48).
4. The parsed numbers come back as `ocr.fields` and the panel **pre-fills** the same form fields the reviewer would otherwise type, marking each one `autoFilled` with a "please confirm" note (`CreatorScanPanel.tsx:513`, `:522`).
5. When the reviewer saves, `POST /api/admin/creator-scan/[accountId]/review` recomputes a 0 to 100 bot likelihood score (`review/route.ts:77`, `scoring.ts:364`) and writes it to `creator_reviews` / `creator_review_videos`. **It never reads or writes `Clip`.** It is READ ONLY on all clip, tracking and money data.

**What extracts the numbers:** OCR (OCR.space cloud), then regex label matching. Not a vision model, not a human (the human confirms afterward).

**Where the output goes:** into a reviewer form field, then into an owner only review record. It informs a human; it does not act.

**Has it ever been used in production? No.**

| Table | Rows | First | Last |
|---|---|---|---|
| `creator_review_screenshots` (the screenshot/OCR path) | **0** | never | never |
| `creator_reviews` (a saved review) | 1 | 2026-07-11 | 2026-07-11 (single, looks like a test) |
| `creator_scans` (Layer 1 auto scan on approval) | 75 | 2026-07-10 | 2026-07-23 (still firing) |

The Layer 1 profile scan runs on its own. The **screenshot reader specifically has zero production uses**: no image has ever been uploaded, so OCR has never once run on real data. On top of that, `OCR_SPACE_API_KEY` is **absent from `.env.local`**; with no key the feature degrades to store only and auto read is off (`screenshot-ocr.ts:134`). So the honest status is: **half built and effectively unused.** We are auditing a feature that has never actually read a screenshot in anger.

## PART 2 — The platform problem

The Layer 1 profile scan IS platform aware (`scan.ts:143` routes by `acct.platform`). **The screenshot reader is not.** `EXTRACTORS` is one flat set of TikTok Studio labels applied to every image, with no platform detection (confirmed by BL-597). Concrete failure modes when an Instagram or YouTube screenshot passes through it:

**Two kinds of failure:** most fields fail SILENTLY (label does not match → field left blank → reviewer types it manually, relatively safe). The dangerous ones are the **wrong slot captures**, where a number lands in the wrong field and then feeds the score.

**YouTube:**
* "Average view duration" contains "duration", which matches the `videoLengthSec` label (`/\bduration\b/i`, line 38). The **average view duration gets read into video length**. That is a wrong slot capture, and video length is what picks the retention norm (`scoring.ts:195`), so a wrong length flips a healthy retention into a flagged one.
* "Impressions", "Impressions CTR", "Watch time (hours)", "Subscribers" → no matching label → all blank.

**Instagram:**
* "Accounts reached" does NOT match `/\breach\b/i` (the word boundary fails on "reached"), so reach is blank.
* "Accounts engaged", "Plays", "Interactions" → no matching label → blank.
* "Profile visits" is the only field that lands, and only by luck (it matches `profileVisits`, `screenshot-ocr.ts:31`).
* Multi column IG layouts invite cross column bleed: `extractValue` reads up to 40 characters after a label (line 61) with first match wins across the whole text, so a value from the wrong column can bind to a field.

**Net:** for IG and YouTube the reader mostly produces blanks (which the human then fills) plus a few actively wrong numbers. Blanks are recoverable; the wrong slot captures are the ones that could mislead a verdict.

## PART 3 — Accuracy: measured, not assumed

**The real accuracy rate is UNMEASURED, and I will not invent one.** OCR has never run in production (0 screenshots), `OCR_SPACE_API_KEY` is not set locally, and I have no real screenshots at 720p or 480p to test against. An invented accuracy figure would be worse than none, so the honest answer is: **unknown until measured.**

What I CAN state with certainty, because it is in the code rather than a guess, is a set of deterministic parsing defects. These are the exact mechanisms behind the owner's fear:

* **K / M / B suffixes are silently truncated (severe).** `toNumber` does `parseFloat` after stripping commas only (`screenshot-ocr.ts:50`), and the count regex is `/(-?\d[\d,]*)(?!\s*%)/` (line 72), which stops at a period. So "12.4K views" is read as **12**, and "1.2M" as **1**. TikTok Studio routinely shows abbreviated counts. This alone can misread views by roughly 1000x. It is a certainty, not a probability.
* **A period used as a thousands separator inverts the number.** OCR often emits "1.234" for "1,234". `toNumber` only strips commas, so it parses **1.234**, a 1000x under read.
* **The decimal percentage misread the owner named is a live path.** If OCR reads "42%" as "4%", the value passes the 0 to 100 sanity clamp (`scoring.ts:105`) as perfectly valid, lands in `retentionPct`, and the score reasons it as "Retention 4% ... views likely partly fake" (`scoring.ts:199`). The clamp catches "420%" but does nothing about a low misread. This is precisely the honest clipper false accusation route, and it is **unguarded**.
* Sanity clamps exist (pct 0 to 100, counts non negative, seconds under 36000) but they only stop wild values, not the plausible wrong ones that do the damage.

**Image quality (720p / 480p):** UNMEASURED. Note that OCR.space free tier caps file size at 1 MB while the route accepts 5 MB (`route.ts:17`); larger screenshots would be rejected by OCR.space and silently fall back to manual entry. The 720p vs 480p reliability threshold cannot be stated without running the reader on real images.

**What the owner must supply for a real measurement:**
1. Roughly 15 to 20 real screenshots **per platform** (TikTok, Instagram, YouTube), each paired with a human typed ground truth of every number on it.
2. Both resolutions he actually receives: a 720p set and a 480p set (and any 480p he sees), plus a couple of blurry / compressed ones.
3. Both single column and multi column Studio layouts.
4. `OCR_SPACE_API_KEY` set (or approval to trial a vision model) so the reader can actually run.

With that, accuracy per field per platform per resolution can be measured properly. Without it, any number is fiction.

## PART 4 — Cost and speed

* **Current design (OCR.space free tier):** $0 up to the free monthly cap. Latency is typically a few seconds per image (the code allows up to 25 s before timing out, `screenshot-ocr.ts:144`). Free, fast, but the weakest on low resolution text, and the 1 MB free cap bites on larger screenshots.
* **A vision model alternative (highest quality):** a single analytics screenshot is on the order of low thousands of input tokens plus a small output, i.e. roughly a fraction of a cent to a couple of cents per image depending on model tier. At the owner's realistic volume (currently zero, plausibly low tens of investigations per month) that is a few cents to a couple of dollars per month. **Cost is not the real constraint here; accuracy and platform awareness are.**

**Free, fast, highest quality: pick two.** Free + fast is what exists now (OCR.space), and it is the weakest on 480p. Fast + highest quality is achievable with a paid vision model at trivial cost given the volume. You cannot get all three. **Best real option:** because volume is tiny, a vision model is the strongest choice on quality for near zero money, but it is not free. If "free" is non negotiable, keep OCR.space but fix the parser and add platform detection first, and accept lower accuracy on 480p.

## PART 5 — The safety rule that must hold

**A misread number must never produce an accusation. Today that rule holds structurally, and the presentation should be hardened.**

Confirmed safe by construction:
* The review route only INSERTs into `creator_reviews` / `creator_review_videos` (`review/route.ts:84` onward). It **never** updates `Clip.status`, earnings, or any money field. The reader cannot change a clip's status.
* Every surface is gated by `requireOwnerOrCapability("CREATOR_SCAN_REVIEW")` and is never clipper visible.
* This reader is a separate system from the BL-599 peer engagement signal. It cannot reach or bypass that signal's rank not reject discipline, and like BL-599 it never freezes, rejects or accuses. BL-518 (nothing auto accuses or auto rejects; a human always decides) is upheld.

What it does well already: OCR values are pre filled into the form as `autoFilled` and labelled "auto read N fields, please confirm" (`CreatorScanPanel.tsx:522`), and the raw image is kept as evidence, so a human sees the source.

What must be added before anyone leans on it:
1. **Show each extracted number next to its source region of the image** (a crop or a highlighted box), not just the full screenshot elsewhere on the page, so the reviewer verifies the read at a glance.
2. **Flag low confidence extractions explicitly** (OCR.space returns per word confidence; surface it, and always flag any value carrying a K / M / B suffix or an ambiguous separator).
3. **Never feed an auto read number straight into a verdict.** Keep the "please confirm" gate, and consider requiring the reviewer to touch each auto filled field before the score is trusted.

## PART 6 — Verdict and the plan

**Usable today? No, not for reliance.**
* **TikTok:** the only platform it targets, but even here accuracy is UNMEASURED and the K / M truncation bug will misread common abbreviated counts. Usable only as a rough draft a human fully re checks.
* **Instagram and YouTube:** not usable. No platform detection, wrong labels, silent blanks plus a few wrong slot captures.
* It has never been used in production and OCR is not even enabled, so nothing is at risk right now, but it must not be trusted until the work below is done.

**Ranked plan:**
1. **Fix the number parser first** (K / M / B suffix expansion, locale aware thousands separators). This is a correctness prerequisite even for TikTok and is the single most direct guard against the false accusation the owner fears.
2. **Platform detection** (signature match on the OCR text) so an IG or YouTube screenshot is identified before any label mapping.
3. **Per platform label maps** to unified fields (TikTok, Instagram, YouTube each get their own label set).
4. **Confidence flagging plus number beside its source region**, with the confirm gate kept.
5. **Measure accuracy** on the owner supplied sample (PART 3), per field, per platform, per resolution. Only after this may any auto read be treated as more than a draft.

**Bottom line for the owner:** the screenshot reader should not be trusted until platform detection, the parser fix, and a real accuracy measurement are done. Until then it is a convenience that pre types numbers for a human to fully verify, and that is exactly how it must stay. Its saving grace is that it is structurally incapable of accusing anyone on its own.
