# R-6 — the bridge from prose rules to checkable rules: the structure already exists, the database destroys it

**2026-07-17 · RESEARCH ONLY. READ ONLY on production. Nothing built, changed, pushed or merged. One doc.**
**Base:** main @ ee1f5b1f · **Branch:** checkpoint/R-6

---

## THE ANSWER

**The structure already exists. It is thrown away at the database boundary.**

The campaign form does not ask for prose. It renders a **repeatable list, one input per rule**, with an
"Add requirement" button and the helper text *"Add each requirement as a separate item."*
(`admin/campaigns/page.tsx:1021-1035`). The owner already authors rules one at a time. Then:

```ts
// page.tsx:290 — SAVE: the list is flattened into a blob
const requirements = form.requirementsList.filter((r) => r.trim()).join("\n");
// page.tsx:174 — LOAD: the blob is split back into a list
const reqs = c.requirements ? c.requirements.split("\n").filter((r) => r.trim()) : [""];
```

So R-5's diagnosis is right that the rules are prose in the database, but the cause is narrower and far
cheaper to fix than "campaign rules are free text". **Each rule is already a discrete item in the UI. What
is missing is a type on each item.** This is not a prose-parsing problem. It is a `join("\n")`.

**Correcting R-1 on a detail that matters for this design:** R-1's examples render rules separated by
`|`. That was R-1's own markdown-table formatting, not the data. Measured: **`pipe_count = 0` on all 11
campaigns**; the real delimiter is newlines (4 to 12 per field). One rule per line, exactly as the form
writes it. Anyone building a parser off R-1's rendering would split on the wrong character.

### The structured schema, every field motivated by a real rule

One JSON array on the campaign, one object per rule. Each row keeps its human text so nothing is lost.

```jsonc
{
  "id": "r1",
  "text": "Caption must credit STRAENGE and end with #STRAENGE.",  // verbatim, always shown to the clipper
  "type": "caption_contains",
  "platforms": ["tiktok", "instagram"],   // scope; omit = all
  "params": { "value": "STRAENGE", "position": "ends_with" },
  "enforcement": "check"                  // check | warn | human_only
}
```

| `type` | Motivated by (real campaign, real line) | Params |
|---|---|---|
| `sound_is` | STRAENGE [1] *"official 'Read My Mind' sound only, never a re-upload"* | `soundId`. **Id, not link.** A re-upload has a different id, so id-compare enforces "no copies" for free. Today `soundLink` holds a `vt.tiktok.com` short link, which R-2 proved cannot be compared. |
| `caption_contains` | STRAENGE [5] *"credit STRAENGE and end with #STRAENGE"* | `value`, `position: anywhere \| starts_with \| ends_with`. **`position` exists because STRAENGE says "end with".** A plain contains-check cannot express it. |
| `caption_from_list` | CROCS [4] *"Use one of our approved captions. Tiny tweaks only."* | `options[]`, `similarity: 0.9`. **Fuzzy, not exact: "tiny tweaks only" is literally a fuzzy-match instruction.** |
| `hashtag_required` | somesome, GainzAlgo, WinGram `hashtagRules` | `tags[]`. Two of three are already bare tags. |
| `hashtag_forbidden` | somesome `captionRules` *"Do not include other app mentions or extra hashtags"* | `tags[]` or `allow_only: true`. **This is what `bannedContent` was for; it is used 0/30 times.** |
| `mention_required` | CROCS [2] *"Tag @crocsshop_us and @wisdm8 every time"* | `handles[]`. **A list, because CROCS names two.** |
| `platform_allowed` | STRAENGE [6] *"TikTok and Instagram only"*; CROCS [5] *"Post on TikTok"* | `platforms[]` |
| `duration_between` | Nothing. **No campaign has ever set it.** | `minSec`, `maxSec`. Included only because the column exists; it has no live motivation. |
| `overlay_present` | somesome *"logo must stay fully visible at all times"*; CROCS [1] *"Show the Crocs in every video"* | `assetRef`, **`coverage: cover_frame \| majority \| entire`** |
| `overlay_not_overlapping` | somesome, 14 rejections: *"must be placed in the clean middle split area only… do not place over captions, text, subtitles"* | `assetRef`, `forbiddenRegions: ["text"]` |
| `burned_text_equals` | GainzAlgo [3] *"'Indicator name: GainzAlgo V2 Alpha' text overlay"*; somesome, 8 rejections: *"top right should say 'What should I put on the cards?'"* | `value`, `region: top_right \| bottom_left \| any`, **`coverage`** |
| `no_watermark` | GainzAlgo [4]; STRAENGE [3] *"no other watermarks"* | `deny: ["tiktok_logo", "other_brands"]` |
| `source_from_set` | GainzAlgo [1] *"only from the approved profiles list"* | `profileRefs[]` or `phashIndexRef`. **pHash, not a model (R-3).** |
| `audience_tier` | GainzAlgo [8] *"Minimum 30% Tier 1 audience"* | `tier`, `minPct`. Partly exists as `targetAudience`. |
| `account_setup` | GainzAlgo [7] *"logo as PFP, trading-related name, bio with gainzalgo.com"* | `pfp`, `bioContains` |
| **`human_only`** | STRAENGE [2] *"old-money luxury edit, that's the only vibe"*; [4] *"cut it to the beat"*; CROCS [3] *"hook them in the first 3 seconds"*; somesome *"low-effort descriptions will be rejected"* | `text` only. **Carries no params by design.** |

**Two params exist only because real rules forced them, and both would be missed by a naive schema:**

1. **`platforms` per rule.** somesome's `captionRules` says *"(If its Instagram)"* **three times**, and
   *"(This is for Youtube Instagram and Tiktok)"* once. **Rules are already platform-conditional in
   prose.** A schema with one rule set per campaign cannot express somesome's actual policy.
2. **`coverage` on every pixel rule.** somesome says *"at all times (for majority of the video)"*;
   GainzAlgo says *"on screen the entire video"*. **These are temporal claims.** A cover-frame check can
   only ever prove presence at t=0. This single param is what decides whether frame 1 is enough, and it
   is the honest answer to the owner's per-second question (PART 2).

### What prose expresses that structure cannot, honestly

**Four things, and they are not edge cases.**

* **Taste.** *"High-end old-money luxury edit. That's the only vibe."* No schema holds this. It is 5 of
  10 campaigns (rank 4). `human_only` does not solve it; it **names** it, which is the most a schema can
  do.
* **Compound intent in one sentence.** *"Hook them in the first 3 seconds and make it your own. Never
  post a raw clip."* is three rules: a temporal attention rule, a subjective originality rule, and a
  deny-rule. Structuring forces the owner to split it, and something is lost in the splitting: the
  emphasis, and the reason.
* **The reason behind the rule.** *"No Crocs, no pay"* is a rule plus a consequence plus a tone. A
  clipper reads intent from that sentence. `{"type": "overlay_present"}` communicates nothing.
* **The rulebook that is not in the system at all.** R-1 found four rejection reasons pointing clippers
  at external Google Docs, and two campaigns say *"Read the full requirements right below this"*.
  **Structuring the database cannot structure a Google Doc.** Any machine reading only the database is
  reading a partial rulebook, and it will confidently enforce a subset.

**The mitigation is the design above, not a claim that prose is obsolete:** every rule keeps `text`
verbatim, `enforcement: human_only` is a first-class value rather than a failure, and the clipper is
always shown the sentence, never the JSON.

**Two facts from R-2 that dominate this tier and are easy to skim past:**

* **Metadata reachability is 14.2%.** 85.8% of clips, no check could speak at all. "Free" checks that

**And the rule is dead anyway:** 58.8% of readable clips break the required-hashtag rule and **humans

**Cost, measured by R-2, not estimated:** $0.00112/clip (TikHub), i.e. **~$2.58/month** at 2,301 clips.

### Tier 3: EXPENSIVE, pixels over time — the owner's specific ask, costed

The owner asked for *"a screenshot every single second, and compare them"*. That sentence has two
readings and they differ by four orders of magnitude.

**Reading (a): sample the clip once per second and judge each frame.**
A 30s clip at 1 fps = 30 frames × ~1,600 tokens = 48,000 input tokens.

```
Haiku 4.5 input $1/MTok  ->  48,000 x $1/1,000,000        = $0.048 / clip
                        x 2,301 clips/month               = $110.45 / month
                        Batch API (documented 50%)        = $55.22 / month
```

**Against the ~$50/month target: it misses, on the API alone, before any infrastructure.** And the API
is not the step-change. **Frame 1 is free (already stored); frame 2 requires downloading and decoding
the video.** R-3 estimated ~10 MB/clip × 2,301 ≈ **23 GB/month plus ffmpeg CPU**.

**That 23 GB is an estimate, and whether it costs anything is UNVERIFIED.** R-3 called it "egress", but
pulling video *into* Railway is ingress; Railway's bandwidth billing is not something I can read from
this machine and **I will not invent a price for it**. What is certain is the *shape*: once the video is
downloaded, more frames are cheap. The cliff is the download, not the frame count.

| Frames | Haiku batched | Needs video download? |
|---|---|---|
| 1 (cover) | **$2.24** | **No, already stored** |
| 3 (first/mid/last) | $5.98 | Yes |
| 30 (per second) | $55.22 | Yes |

**So per-second buys almost nothing over 3 frames, at 9x the price.** R-3 established that all four of
the highest-count pixel rejection reasons are answerable from the cover frame alone.

**But the owner is not simply wrong, and this is the one place I disagree with a flat "cover frame is
enough".** Two real rules are **temporal**, and PART 1's `coverage` param names them:

* somesome: *"logo must stay fully visible **at all times** (for **majority of the video**)"*
* GainzAlgo: *"text overlay on screen **the entire video**"*

**A cover-frame check proves presence at t=0 and nothing more.** A clipper who shows the logo for one
second and removes it passes. So sampling is genuinely needed for `coverage: majority | entire`.
**Three to five frames answers "majority" and "entire" nearly as well as thirty**, because the rule is
about persistence, not about any particular second. Per-second is paying for redundancy that video has
by construction.

**Reading (b): compare every second against a source podcast.** R-3 costed this as asked:
**~$26,300/month**, about 157x the entire Apify bill. **Done properly it is ~$0**: index the source once,
then match with **perceptual hashing** (pHash, Hamming distance, CPU-only, sub-millisecond, the technique
behind Content ID). An LLM asked *"is this the same footage"* is slower, costlier and less reliable than
a hash built for exactly that. **The idea is right; the LLM is the wrong tool.**

### The per-second verdict, plainly

**No. Per-second sampling does not land inside ~$50/month.** $55.22 on the API alone at Haiku with the
batch discount, before an unpriced download-and-decode pipeline, and it buys ~nothing over 3 frames at
$5.98. **The affordable answer that actually serves the rules is: cover frame at $2.24, plus 3 frames at
$5.98 only for the two campaigns whose rules say "entire video" or "majority".**

## PART 3 — the business case, tested against the real alternative

**R-5 measured the alternative and it costs zero.** Review is 14.9 of 19.6 owner hours/month (76%), and
it is **already 52.6% delegated** to one REVIEWER doing 1,206 decisions in 17.0 h/mo. Four campaigns
(BAD BITCH ANTHEM x2, Deja Shoe, CROCS) have **no reviewer scoped**, forcing 85 owner clips/month. The
other 1,045 are in campaigns the reviewer **already covers**. So the constraint is **capacity and habit,
not tooling**, and the fix is a scope change with zero code.

**Pricing AI against that, honestly:**

| Option | Cost/month | Owner hours recovered | Build |
|---|---|---|---|
| **Scope the 4 unscoped campaigns + a 2nd reviewer** | **$0 engineering** | **up to 14.9** | **none, exists, proven at 52.6%** |
| Cover-frame shadow ranking | $2.24 | **0 until proven** | small |
| Structured rules | $0 running | **0 alone** | **real build, see PART 4** |
| Per-second sampling | $55+ and unpriced infra | 0 until proven | large |

**AI review does not beat scoping more reviewers, and it is not close.** Row 1 recovers the same 14.9
hours this week, at zero cost, reversibly, with machinery already carrying more than half the load.
Anything AI does is **strictly additive to that**, not an alternative.

**At what volume does AI flip ahead?** Reviewer capacity is the variable, not clips. R-5: 2,336 decisions
took 32.5 human hours/month across two people (~50s/decision, median 25s). AI flips ahead **only when
reviewer hours run out and hiring another is harder than building** — i.e. when volume outgrows the
people willing to click, not when it outgrows a cost line. At 2,301 clips/month with a queue R-5
measured as **healthy (17 pending of 1,150, median 9.2h to decision)**, that point is not close. **The
queue is clear. Nobody is drowning.** The honest trigger is roughly a 3x to 5x volume increase, or the
reviewer leaving, and neither is today.

**The $300/campaign comparison, stated fairly.** R-5 measured **2 campaigns created in 30 days**, so a
$300/campaign human is on the order of **$600/month**. Cover-frame + metadata at **$4.82/month** is
~124x cheaper. **That arithmetic is real and it favours the owner's instinct.** But it prices a thing
the AI cannot yet do: R-2 measured the metadata gate at 11.05% false rejection with 14.2% reachability,
and R-3 did not benchmark any pixel check for accuracy. **You cannot buy a $600 outcome for $4.82 until
something proves it produces that outcome. Nothing has.**

**`fraudScore` is unusable and this matters more than it sounds.** R-5 measured score 0 clips rejecting
at **19.0%** and score 40+ at **19.5%** — statistically identical, so the ranking carries no information.
**There is no existing signal to gate an auto-approve on, and none to seed a sampling order with.** Any
AI reviewer must earn its own signal from scratch and prove it against human decisions before it is
trusted with anything. **Do not inherit the fraud queue's ranking. It does not work.**

## PART 4 — the honest design

### The smallest first version that genuinely saves time

**It is not a checker. It is the `join("\n")`.**

**Step 1 — stop destroying the structure the form already has.** Add a `type` dropdown to each existing
requirement row and store `Campaign.rulesJson` alongside the untouched `requirements` text. The form
already has the list, the add/remove buttons, and the helper text. **The owner's authoring workflow
barely changes: they type the same sentence and pick a type from a dropdown.** Keep `requirements` as the
rendered source of truth for clippers, so nothing they see changes and there is no migration risk.

**Step 2 — backfill 11 campaigns by hand.** Not a parser. **Eleven campaigns, ~70 rule lines total**
(4 to 12 lines each). Reading them and assigning a type is an afternoon, and a human doing it will get
somesome's *"(If its Instagram)"* right where a parser would silently drop it. **Never write a prose
parser for this: R-2 already proved what that produces.**

**Step 3 — a shadow ranking, deciding nothing.** Run the cover-frame check ($2.24/month) plus the
metadata checks on every clip, store the verdict, **change no status, notify nobody, show no clipper
anything**. After 30 days, measure whether the verdict correlates with human decisions. This is R-2's
shadow-checker pattern applied to pixels, and it is the only way to earn the signal `fraudScore` never had.

**What Step 3 unblocks:** if and only if the ranking is measurably better than chance, it can order the
owner's queue so the 10 to 15 spot-checks land on the likeliest problems. That is the owner's sampling
model, and it is the right instinct **because it never has to be perfect**.

### What must be TRUE before any AI decision touches a clip's status

**For an auto-reject: effectively nothing clears this bar, and the honest advice is to stop aiming at it.**

BL-513's fraud scorer was refused at **8.33%** false rejection. R-2's combined gate measured **11.05%**
and was refused. So the threshold is **below 8.33%** by precedent. But do the arithmetic on what even a
good number means: **1% false rejection at 2,301 clips/month = ~23 honest clippers wrongly rejected every
month**, against R-5's **127 distinct clippers**. That is ~18% of your entire clipper base insulted per
month, **at a rate ten times better than anything measured**.

The governing rule says a false rejection destroys an honest clipper's trust and a missed violation costs
a few dollars. **Those are not commensurable, and the asymmetry does not improve with a better model.**
So: **never auto-reject. Rank only.** A wrong rank costs a human ten seconds; a wrong rejection costs a
clipper their trust and you a clipper.

**BL-518's rule stands and this design is built inside it:** no automatic system may change a clip's
status, and **the clipper must never see a machine's suspicion.** Everything above is owner-facing
ordering. The clipper sees the same review they see today, from the same human, with the same prose
reason. The machine's opinion never reaches them and never touches `status`.

### The ordered plan

| # | Step | Cost | Unblocks | Human stays? |
|---|---|---|---|---|
| **0** | **Scope the 4 unscoped campaigns to the existing reviewer** (R-5's answer) | **$0** | up to 14.9 h/mo **now** | yes |
| 1 | Persist the caption the cron already fetches and discards (R-2's ask) | $0 | every future caption question, retroactively | yes |
| 2 | `type` per rule row + `rulesJson`; keep `requirements` text | small build | everything below | yes |
| 3 | Hand-backfill 11 campaigns, ~70 lines | an afternoon | real rules to check | yes |
| 4 | `soundLink` -> `soundId`; `hashtagRules` -> `tags[]` | small | the two checks R-2 found `NA` | yes |
| 5 | Shadow ranking: cover frame + metadata, decides nothing | $2.24/mo | the signal `fraudScore` lacks | yes |
| 6 | **Only if step 5's ranking measurably beats chance:** order the owner's queue | $0 | the sampling model | **permanently** |
| 7 | 3-frame sampling for the two `coverage: entire/majority` campaigns | $5.98/mo | the temporal rules | yes |

**Where the human stays permanently: the decision.** Steps 5 to 7 order a queue. A person still clicks.
That is not a limitation to engineer away later; it is the design, and it is what makes the owner's
sampling model safe.

## PART 5 — the verdict

**Can a ~$50/month AI reviewer replace a $300/campaign human? No, not as stated. But the owner is
directionally right, and the truthful version is better than the one they asked for.**

**What is achievable, and what it costs:**

* **The price is not the problem and never was.** Cover frame + metadata is **$4.82/month**, ~124x
  cheaper than a $600/month human. **The owner's $50 budget is roughly 10x more than the useful work
  costs.** They are anchored on the wrong constraint.
* **Accuracy is the problem.** R-2 measured 11.05% false rejection and **14.2% metadata reachability**.
  R-3 benchmarked **nothing** for pixel accuracy. The most-requested rule class (on-screen visual, 8/10
  campaigns) has **no measured accuracy at all**. Today the honest quality claim is: **unknown, and
  unknown is not shippable against money.**
* **Per-second is out**: $55.22/month batched on the API alone, before an unpriced download pipeline, to
  buy ~nothing over 3 frames at $5.98. **The podcast-matching idea is right and is ~$0 via pHash**, not
  $26,300 via an LLM.
* **Structured rules must come first, and it IS a real build** — but a much smaller one than R-5's row 11
  implies. **It is not a prose-parsing project.** The form already collects a list; the database
  flattens it. The build is: a type per row, a JSON column, and **an afternoon of hand-backfill across 11
  campaigns and ~70 lines.** Call it days, not months.

**The sampling model works. Full automation does not.** The owner's instinct that one human spot-checking
10 to 15 clips beats one human watching 100 is correct, and it is correct precisely because it never
needs the AI to be right, only to be **better than random at ordering a queue**. **Nothing has proven it
is yet**, and `fraudScore` is a live warning that a plausible ranking can carry zero information: 19.0%
versus 19.5%, built, shipped, and worthless.

**They are also right that other agencies do this**, and I am not dismissing it. The difference is that
those agencies wrote their rules as rules. **Ours are sentences about vibes, and four of them point at a
Google Doc.** That is the actual gap, and it is closeable.

**The single next action, and it is not this project:** do R-5's step 0 first. **Scope the four unscoped
campaigns to the reviewer who already handles 52.6% of the volume.** It recovers the same 14.9 hours this
week, for zero, reversibly. Then, if you still want the machine, start with step 1 and step 2, which cost
almost nothing and make every later option possible. **Do not build a reviewer before the rules are rules.**
