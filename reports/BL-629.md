# CLIPS_PER_TICK safety — is it safe to raise 30 to ~120? (BL-629)

Audit only, read only. Measured 2026-07-22 from live prod data, git history, the BACKLOG, and the reports repo. No load test was run against production. Where a number could not be measured it is marked NOT MEASURABLE FROM HERE. The owner will make any change himself after this.

## One line verdict (PART 5 up front)
**Raising CLIPS_PER_TICK to 120 is SAFE for money and data (it changes tracking freshness only, never earnings, and NULL never 0 holds under overload), but the ONE real risk is the Railway compute box, which took the site down before and cannot be measured from here. Do it STEPWISE (30 to 60 to 90 to 120, watching the [TRACKING] run duration each step), and confirm the Railway compute budget has headroom FIRST.**

## PART 1 — why is it 30? (the history, found)
It is a **deliberate throttle installed after two real incidents**, not an arbitrary default. But the incidents were caused by a fetch path that no longer exists.

• **BL-116-P3 (2026-06-17) introduced the cap.** Root cause on record: a cron tick collected up to **500** due clips, fetched them all in **~90 to 158s** (the Apify / clockworks era), then ran the per campaign Serializable write loop; total wall clock passed the Railway HTTP/proxy kill window, so the route was cut off **before persisting clip_stats**. Result: **no auto clip_stat for ~2h45m**. The kill was TIME, not memory (the batch path returns ClipStatsSlim with no raw payload; memory trim was found unnecessary). Cap default was set to 60.
• **BL-198 / BL-199 (2026-06-19) hardened it.** A clockworks TikTok batch (**~108 to 126s**) blew the per chunk timeout, every clip in it became a miss, and up to **25 serial ~18s individual fallbacks** cascaded past the 300s route ceiling into a SIGKILL that left a **15 minute stale lock**. BL-199 capped cron individual fallbacks at **5** so this can never recur regardless of cap.
• **BL-197 (2026-06-19) lowered the default 60 to 30.** At 60 the top of hour run "runs long (multiple ~108s clockworks batches), hits the 300s ceiling, holds the lock, and blocks the off-hour drain." Also, Railway already had `CLIPS_PER_TICK=30` set; the code default 60 was overriding it because the env was not reaching the process. So 30 was made the code default to match the intended production value. Stated capacity then: ~30 x 6 = ~180/hr > ~142/hr demand.

**Conclusion, stated plainly:** 30 is a real, deliberate throttle, and its driver was **the slow Apify path**, specifically ~108 to 126s clockworks TikTok batches and ~18s serial Apify single fetches. **That path is now DEAD** (BL-611/612/613: Instagram is HikerAPI, TikTok is LamaTok, Apify is off). So the *specific* reason for 30 is now **stale**. What is NOT stale is the underlying constraint it protects: a tick's total wall clock must stay under the Railway kill window, and that wall clock is now dominated by the per campaign Serializable **write loop**, not the fetch. That is why this is a "measure the write loop, raise carefully" question, not a "just bump it" one.

## PART 2 — what actually happens at higher values (measured, then extrapolated)
Measured today at cap 30:
• **Tick wall clock:** the clip_stats checkedAt spread for busy (25 to 30 clip) ticks is **10 to 17s** on the median tick; BL-199b independently recorded a full run at **~74s** (which includes fetch dispatch plus the once per tick budget sweeps and monitors). A small number of ticks on the evening of 07-21 showed spans of **79 to 486s**; these are outliers against the 10 to 17s norm and could not be fully attributed from here (likely a provider slow spell or write contention, or a bucketing artifact spanning a killed then recovered tick). They are the tail the safety nets exist to contain.
• **Provider calls per tick / rate:** HikerAPI peaked at **14 calls/second** in the last 24h at cap 30. This is bounded by the concurrency 15 semaphore, NOT by cap size, so raising the cap makes ~14/s last **longer**, not go **higher**. The 15/s ceiling is respected at any cap.
• **Campaigns per tick:** ~6 distinct campaigns per 27 clip tick (the write loop is per campaign sequential Serializable, so this is the real write cost driver).

Extrapolation to 60 / 100 / 150 (honest, write loop scales roughly linearly; fetch is semaphore bounded and near flat; fixed per tick overhead of budget sweeps + monitors does not scale):
• **Does a tick still finish before the next (ticks are every 10 min = 600s)?** Yes, comfortably. Even a fully loaded tick yields at the **210s loop budget** (TRACKING_LOOP_BUDGET_MS) and then has the reserved ~90s tail to finalize, so ~300s worst case against a 600s gap. A healthy 120 clip tick is estimated at **~150 to 200s** (scaling the ~74s baseline, most of which is clip linear), which sits under the 210s graceful yield. Margin is real but tightening at 120, which is exactly why the stepwise method below matters.
• **Does the HikerAPI budget overflow path re-engage?** Only in the benign sense. Post BL-610 the per tick HikerAPI budget IS the 210s loop budget; if a tick runs long, overflow clips are **left absent from primaryMap and keep last known (NULL never 0), re-probed next tick**. Apify is dead, so overflow means "this clip does not update this tick," never "pay Apify." No money path, no wrong value.
• **Memory and CPU (the one that took the site down):** per tick working set is ClipStatsSlim only (no raw payloads), so 120 clips is **well under 1MB** against a 4GB Node heap; OOM is not the risk (BL-116 already proved the historical kill was TIME, not memory). CPU: the concurrency 15 semaphore caps in flight fetches regardless of cap, so the per tick CPU **peak** does not rise with the cap; a bigger cap means fewer off-hour drain ticks are needed, so **total daily compute is roughly neutral to lower**. HOWEVER the **Railway plan CPU/RAM ceiling and the compute budget are NOT MEASURABLE FROM HERE**, and a compute cap took production down on 2026-07-18. This is the single unmeasured risk and the reason for the prerequisite in PART 4.
• **HikerAPI daily quota:** cap 120 => 120 x 6 x 24 = 17,280 polls/day, roughly half of them Instagram => ~8,640 HikerAPI calls/day versus current ~1,115/day and a ~400,000 prepaid quota. That is ~46 days per refill (versus ~358 today). Fine, just refilled more often. Not a wall at 120.
• **pgbouncer / DB pool:** the write loop is **per campaign sequential Serializable** (one transaction at a time), so concurrent DB load does not scale with the cap; connection_limit=1 through the transaction pooler holds exactly as it does at 30. Not a constraint at 120.

## PART 3 — what breaks, and how badly, if it is set too high
• **Failure mode is graceful and self healing.** If a tick runs long it hits the 210s loop budget check (between campaign group batches), stops starting new work, finalizes what it processed, and **defers the rest WITHOUT advancing their nextCheckAt**, so the next tick re collects them oldest first. It does not fall permanently behind; it drains.
• **No overlap, no double poll, no double write.** Ticks are single flighted by the tracking lock plus the isBatchTick claim, so two ticks cannot run the same clips at once. A deferred clip was not polled this tick, so it is not double written.
• **Can a 0 ever be stored under overload? NO.** A deferred or unprocessed clip writes **no ClipStat at all**, so it keeps its last known views. A tick killed before finalize writes nothing. The NULL never 0 contract (BL-543) therefore holds under overload by construction: overload produces *absence of a write*, never a zero. Confirmed in code (the loop defers un advanced jobs; there is no path that writes 0 on a miss).
• **Money impact: NONE, only freshness.** Raising CLIPS_PER_TICK is collection and scheduling only. Earnings math (views x CPM, writeClipEarnings, budget caps, per campaign Serializable writes, videoUnavailable) is byte equivalent; a deferred clip processes next tick with identical inputs. The only thing that degrades if the cap is set far too high is tracking **latency** (some clips update a tick later), never a wrong number and never a payout.
• **The only genuinely bad outcome** is the historical one: a tick whose wall clock exceeds the actual Railway kill window BEFORE the 210s graceful yield can fire, causing a SIGKILL that holds the tracking lock for the 15 minute stale window (TRACKING_LOCK_STALE_MS) and starves ticks for up to 15 minutes. The cap, the 210s budget, and the 5 fallback cron limit all exist to keep the tick short enough that this never happens. Setting the cap absurdly high (for example 500, the clamp ceiling) re introduces this risk on a slow provider evening. 120 does not, provided the write loop stays under ~180s, which the stepwise method verifies.

## PART 4 — the safe number and the safe method
**Safe number: 120 is safe** on every hard limit (rate 14 of 15/s bounded by the semaphore, quota ~46 days per refill, pool sequential so unchanged, memory under 1MB, money unaffected, NULL never 0 holds). The only thing to prove per environment is the **tick wall clock**, and the measured baseline (~74s at 30, mostly clip linear) projects a healthy 120 tick at **~150 to 200s**, under the 210s graceful yield. If a step shows runs pushing past ~180s, **stop at 90** rather than 120. Do not exceed 150; keep well below the 500 clamp ceiling.

**Safe method: stepwise, not one jump.**
1. Confirm the Railway compute budget has headroom (see prerequisite below). 
2. Set `CLIPS_PER_TICK=60`. Watch for a full day.
3. If healthy, `=90`. Watch a day. 
4. If healthy, `=120`. Watch a day.

**What to watch after each step** (all queryable or in Railway logs):
• Tick duration: the `[TRACKING]` / `[TRACKING-CONFIG]` run lines in Railway logs, and this DB proxy for wall clock per tick: the checkedAt span per 10 min bucket (it should stay well under 180s).
• Throughput and backlog draining: clip_stats written per day should rise and the overdue backlog should fall. Query: count of tracking_jobs where isActive and nextCheckAt < now(); it should trend down.
• Rate stays legal: peak HikerAPI calls per second should stay at or under 15. Query the max per second count on apify_usage_entries provider hikerapi-v2.
• Health: no repeated 15 minute gaps in cron_runs (isBatchTick rows), no stale lock warnings, error rate flat.
• Compute: Railway service CPU and memory graphs stay within the plan; no throttling.

**Exact rollback:** set `CLIPS_PER_TICK=30` and let Railway redeploy (the value is read at call time, so no code change is needed). Instant and total. Or `git revert` if a code default was touched (it need not be; this is an env change).

**Prerequisite, stated because it is the real risk:** the 2026-07-18 outage was a Railway **compute cap** against a small budget. Raising CLIPS_PER_TICK does not raise the per tick CPU peak (semaphore bounded) and slightly lowers total ticks, so it should not worsen compute materially, but the plan ceiling is NOT MEASURABLE FROM HERE. **Confirm the Railway compute/plan has headroom, or raise the budget, BEFORE stepping past 60.** If the box is already near its cap at 30, fix that first; more clips per tick is not what saves you there.

## What could not be measured from here
The Railway plan CPU/RAM ceiling and current compute headroom; the exact current Railway route/proxy kill threshold (historically it was once below 210s on self hosted Railway; the code now asserts maxDuration 300s); the cause of the 07-21 evening 200 to 486s tick spans; and the live per tick memory figure. No load test was run against production, by instruction.

## Verdict (repeated, PART 5)
**SAFE to raise CLIPS_PER_TICK to 120, stepwise (30 to 60 to 90 to 120) watching the tick duration, and ONLY after confirming Railway compute headroom.** It affects tracking freshness only, never money, and a 0 can never be stored under overload. The one way to repeat the old outage is to jump straight to a large value on a box already near its compute cap; the stepwise method plus the compute check removes that risk.
