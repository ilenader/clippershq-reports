# BL-568: the growth engine has reached 3 people, and its dashboard shows 225,080 sends that never happened

**AUDIT ONLY. READ ONLY on code, config, data and money. No email sent, no trigger enabled, no env var changed, nothing built.**
**Base: origin/main @ `ae148f3d`. Branch: checkpoint/BL-568. Measured live 2026-07-17 from prod. No probe cost (DB reads only; no external API called).**

---

## THE ANSWER IN THREE NUMBERS

* **3.** Real emails the growth engine has ever sent (all one trigger, FIRST_CLIP_CELEBRATE).
* **225,080.** Dry-run "practice" sends it has logged, which the admin feed surfaces as activity.
* **811.** Never-activated clippers a single WELCOME trigger would email at once if switched on, into a cold list, which is the real risk.

**R-5's leading suspect is wrong.** The engine is NOT dark because the cron does not run. The lifecycle cron runs every 15 minutes and has for days. The engine is dark because **only one trigger is on the allowlist, and it is an event trigger that can only reach brand-new signups.** The fix is an env var, not a build, but the safe version of that fix is not the obvious one.

## PART 1: why has it sent only 3 messages?

### Every trigger, its source, and whether it can send

16 triggers exist (`src/lib/growth/triggers.ts`). A trigger sends a real email only when THREE gates all pass: `GROWTH_ENGINE_ENABLED` not off (default on), `GROWTH_ENGINE_LIVE === "true"`, and the trigger id is in the `GROWTH_ENABLED_TRIGGERS` allowlist (**default empty means nothing sends**, `config.ts:47-59`).

|---|---|---|---|---|
| WELCOME | activation | poll | 0 | **811** |
| NO_CLIP_24H / 72H / 7D | activation | poll | 0 | 0 (no recent signups match) |
| FIRST_CLIP_CELEBRATE | event | event | **3** | n/a (event) |
| CLIP_APPROVED_POST_AGAIN | event | event | 0 | n/a (event) |
| CLIP_REJECTED_RECOVER | event | event | 0 | n/a (event) |
| STALLED_3D | stalled | poll | 0 | 21 |
| STALLED_7D | stalled | poll | 0 | 25 |
| STALLED_14D | stalled | poll | 0 | 52 |
| STALLED_MONTHLY | stalled | poll | 0 | 102 |

| COMMUNITY_SOCIAL_PROOF | broadcast | poll | 0 | 38 |
| STREAK_SAVER_TONIGHT | timing | poll | 0 | 73 |
| INACTIVITY_USUAL_TIME | timing | poll | 0 | 0 |
| ALMOST_THERE | event | event | 0 | 4 |

12 poll triggers, 4 event triggers. (R-5 said 11 poll; the registry has 12.)

### The cron is NOT the problem. It runs.

R-5's leading suspect was that `RAILWAY_NATIVE_CRON` omits `lifecycle`, so the poll triggers silently no-op. **Measured against live data, that is false:**

```
cron_runs where kind='lifecycle' : 219 heartbeats
first : 2026-07-15 11:37 UTC     last : 2026-07-17 18:45 UTC     cadence : every 15 min
```

The lifecycle sweep is firing on schedule. The 225,080 dry-run rows are the proof it runs: those are the poll triggers evaluating tens of thousands of candidates every sweep. `RAILWAY_NATIVE_CRON` in production clearly includes `lifecycle` (the BL-529 watchdog comments say `lifecycle,watchdog`, and the heartbeats confirm it). **So there is no free cron fix, because the cron is not broken.**

### The real reason: only FIRST_CLIP_CELEBRATE is allowlisted, and it can only reach new signups

The 3 real sends are all one trigger, sent to 3 different people:

```
FIRST_CLIP_CELEBRATE  piyush70508@...    2026-07-16 12:36 UTC   variant A
FIRST_CLIP_CELEBRATE  soussiahmed741@... 2026-07-17 01:22 UTC   variant A
FIRST_CLIP_CELEBRATE  tayyabharis672@... 2026-07-17 01:24 UTC   variant A
```

This is airtight evidence of the configuration, without needing to read Railway:

* **`GROWTH_ENGINE_LIVE` is "true".** Real (dryRun=false) sends exist, which is impossible unless the master live gate is on.
* **The allowlist contains only FIRST_CLIP_CELEBRATE.** WELCOME had **811** eligible users in dry-run and sent **zero** real emails. If WELCOME were allowlisted, some of those 811 would have sent. The same holds for every other poll trigger. So only FIRST_CLIP_CELEBRATE is enabled.

**And FIRST_CLIP_CELEBRATE is structurally unable to reach existing clippers.** It is an event trigger that fires on a clipper's first-clip event, and that event field was only added around 2026-07-13. Anyone who posted their first clip before then has no such event and will never get it. So the one live trigger reaches only people who activate after 07-13, which is exactly the owner's complaint: it reaches brand-new signups and nobody else. It fired 3 times because only 3 people had a qualifying first-clip event turn into a send since it was enabled.

**Answer to PART 1, plainly:** the engine has sent 3 messages because 15 of its 16 triggers are switched off at the allowlist, and the 1 that is on can only celebrate a first clip that, for every pre-07-13 clipper, was never recorded. The engine is not broken and the cron is not dark. It is simply configured to reach almost no one.

## PART 2: the fake activity feed

### What is real and what is theatre

```
email_events, all time:
  sent · dryRun=TRUE  · email    115,553      <- theatre
  sent · dryRun=TRUE  · in-app   109,527      <- theatre
  sent · dryRun=FALSE · email          3      <- real
```

**225,080 practice records. 3 real emails. Zero real in-app notifications, ever.** The true number of growth emails actually delivered in the engine's entire life is **3**, and I named all three above. I could not independently query Resend's API from here (no access), so this is the internal send ledger; the `dryRun=false` flag is the authoritative record of a real dispatch, and only 3 rows carry it.

To be clear about scope: this is the GROWTH engine only. The platform's transactional emails (CLIP_APPROVED, which R-5 measured at 1,931 sent and 1,719 read, plus payout and verification emails) are a separate system and are healthy. They are not part of this 3, and nothing here suggests they are broken.

### Why the feed looks alive when nothing is being sent

The admin growth events feed (`api/admin/growth/events/route.ts:51-56`) returns every `email_event` with no `dryRun` filter, and the render (`admin/growth/page.tsx:540-565`) tags each row "Practice" (grey) or "Live" (green). So per row the label is technically honest. The deception is structural, in two ways:

1. **The feed is 75,000-to-1 practice.** A "recent activity" list dominated by 225,080 practice records reads as a busy, working engine. "Welcome email to <someone>, Practice" scrolling endlessly looks like the engine is doing its job, when in fact nothing leaves the building. The label is a whisper under a firehose.

2. **Every row reads the same relative time, and increments together on reload.** This is not a page-load-derived timestamp bug. The timestamps are real `createdAt`. The cause is that each 15-minute sweep bulk-writes thousands of dry-run rows in the same instant. The 15 newest events, right now, are all from one sweep:

```
WELCOME  dryRun=true  2026-07-17 18:46:03.120
WELCOME  dryRun=true  2026-07-17 18:46:03.098
WELCOME  dryRun=true  2026-07-17 18:46:03.097
... all 15 within 18:46:02.7 to 18:46:03.1, one second, one sweep ...
```

The feed orders by `createdAt desc`, so its entire visible page is one sweep's worth of same-instant writes. They all render the same relative time (`relTime(e.createdAt)`), and as the page ages they all tick up together. On the next sweep they are all replaced by a new wall at a new single instant. That is precisely "every row reads the same relative time on reload."

**BL-524's rule inverted.** BL-524 said a dashboard must never let the owner confuse quiet with broken. This is worse: it lets the owner believe the engine is actively sending when it has sent 3 emails in its life. The feed manufactures the appearance of work.

### A second, quieter problem: table bloat

The engine writes roughly **90,000 email_event rows per day** (88,130 on 07-17; 97,626 on 07-16), almost entirely dry-run. That table will grow without bound for zero real output. It is not urgent, but it is the same root cause: the engine logs a practice "send" for every candidate on every sweep.

### The fix for the feed

Default the activity feed to LIVE sends only (or split it into "Sent" and "Practice" tabs, with Practice off by default), and stop writing one dry-run EmailEvent per candidate per sweep (a per-trigger dry-run counter would give the same insight without 90k rows a day). An owner who cannot trust the feed cannot manage any of the rest.

## PART 3: reaching the existing clippers

### The real audience size

```
users by role:  CLIPPER 1,109   REVIEWER 27   CLIENT 19   OWNER 3   (total 1,158)
```

**The owner is right: there are 1,109 clippers, not 50.** R-5's "48 to 50 with a weekly approval" and "127 distinct in 30 days" measured ACTIVE clippers; the full clipper base is 1,109, and **811 of them (73%) have never posted a clip** (that is the WELCOME audience, "never-activated"). The gap between 50 active and 1,109 total is the dormant base the owner wants to wake up.

### What is backfillable, and what is permanently lost

The distinction is exactly whether the trigger reads CURRENT STATE or a HISTORICAL EVENT.

**Backfillable now, because they compute from data we already hold** (join date, last-post date, streak, campaign activity). The engine is already computing these every sweep; the dry-run rows ARE the backfill:

| Trigger | Reads | Reaches (measured, distinct users) |
|---|---|---|
| WELCOME | never posted a clip | **811** |
| STALLED_MONTHLY | dormant, last post ~monthly ago | 102 |
| STREAK_SAVER_TONIGHT | streak about to break tonight | 73 |
| STALLED_14D / 7D / 3D | was active, paused N days | 52 / 25 / 21 |
| COMMUNITY_SOCIAL_PROOF | engaged | 38 |

**Permanently lost for older clippers, because they need an event that was never recorded:**

* **FIRST_CLIP_CELEBRATE** needs a first-clip event, which did not exist before ~07-13. No pre-07-13 clipper can ever receive it. This is genuinely unrecoverable; there is no backfill for an event that was never captured.

**Future-only, not lost:** CLIP_APPROVED_POST_AGAIN and CLIP_REJECTED_RECOVER fire on future approve/reject events, so they will work for everyone going forward once enabled; they simply cannot fire retroactively. ALMOST_THERE fires on future close-signals (4 matched in dry-run).

**Answer to PART 3:** almost everything the owner wants is backfillable right now. Every poll trigger reads current state and the engine already knows who qualifies. Only the first-clip celebration is lost for the existing base, and it is the one trigger currently switched on.

## PART 4: the risk, stated honestly

### The spam and domain-reputation risk is real and is NOT mitigated by the existing caps

The safety net that exists is good, per user:

* `canSendMarketing` (`can-send.ts`) is a single send chokepoint: it enforces suppression list, one-click unsubscribe (`emailMarketingOptOut`), and frequency caps (`FREQ_CAP_DAILY = 1`, `FREQ_CAP_WEEKLY = 3`). In-app is exempt (no domain risk). Dry-run rows are never counted against caps.
* The engine dispatches live email through a throttled background batch (concurrency <= 2) that respects Resend's rate wall.
* Kill switch: emptying `GROWTH_ENABLED_TRIGGERS` or setting `GROWTH_ENGINE_LIVE` to anything but "true" stops all sends instantly, no deploy. That is the one-action stop for a 2am misfire.

**But none of that caps the TOTAL volume of a cold burst.** The per-user cap of 1/day does nothing when the send is to 811 DIFFERENT users. `SCAN_BATCH_SIZE` is 5000, so a single sweep can queue every WELCOME match at once. Enabling WELCOME would therefore push **811 emails to people who have never received a single email from this platform**. A sudden burst to a cold, unengaged list is the classic way to wreck a sending domain's reputation. If that happens, the emails that stop landing are not just the growth nudges; the transactional payout and verification emails start going to spam too. **That would be far worse than the problem being solved.** This is the single most important sentence in this report.

### Would a new trigger duplicate one that already works?

Yes, one would. **CLIP_APPROVED_POST_AGAIN** is an in-app trigger that fires on clip approval, and the platform already sends a CLIP_APPROVED notification on approval (R-5: 1,931 sent, 1,719 read). Enabling it would put a second in-app notification on every approval. It is in-app only, so no deliverability risk, but it is redundant noise on top of a notification that already works and is already read. Do not enable it.

### The safest first step is not the biggest

Not WELCOME to 811 cold contacts. The safe sequence:

1. **The feed fix first**, so the owner can see what is actually happening.
2. **Enable ONE warm, small, relevant trigger** and watch deliverability before widening. **STALLED_7D (25 users)** or **STALLED_3D (21 users)** is the right first choice: these people WERE active and recently paused, so they are a warm audience, the message ("you paused, come back") is relevant, and 20-odd emails cannot dent a domain. Watch open and bounce rates for a few days.
3. **Only then** consider the larger dormant audiences, and when WELCOME's 811 are approached, do it in throttled daily batches (a few dozen a day), never in one sweep, so the domain warms rather than burns.

## PART 5: the verdict

**Why 3 messages:** not a broken cron and not a code bug. The lifecycle sweep runs every 15 minutes and evaluates everything. But 15 of 16 triggers are off at the `GROWTH_ENABLED_TRIGGERS` allowlist, and the one that is on, FIRST_CLIP_CELEBRATE, can only fire for clippers whose first-clip event exists, which means only people who activated after ~07-13. It fired 3 times because only 3 qualifying events became sends since it was switched on.

**Is the fix an env var or a build?** An env var. `GROWTH_ENGINE_LIVE` is already "true"; the lever is `GROWTH_ENABLED_TRIGGERS`, which takes a comma-separated list of trigger ids and needs no deploy. But the value to put there is a small warm cohort, not everything.

**Who is reachable, by which trigger:** the full base is 1,109 clippers, 811 of them never-activated. Every poll trigger is backfillable from current state and the engine already knows the audience: WELCOME 811, STALLED_MONTHLY 102, STREAK_SAVER_TONIGHT 73, STALLED_14D 52, COMMUNITY_SOCIAL_PROOF 38, STALLED_7D 25, STALLED_3D 21.

**What is genuinely lost:** only the first-clip celebration for pre-07-13 clippers, because the event was never recorded. Everything else is recoverable.

**The single next action:** fix the activity feed to show live sends by default (so the owner can trust his dashboard), then add **STALLED_7D** alone to `GROWTH_ENABLED_TRIGGERS` and watch deliverability on its 25 warm users for several days before widening. Do not open WELCOME to 811 cold contacts, and do not enable CLIP_APPROVED_POST_AGAIN, which duplicates a notification that already works.

**The fake activity feed, separately:** it shows 225,080 practice records as recent activity, labeled "Practice" but drowning the 3 real sends 75,000 to 1, and every visible row shares one sweep-instant timestamp so they all read the same relative time and tick together on reload. The engine has actually delivered 3 emails, ever. Default the feed to live sends and stop writing a dry-run row per candidate per sweep.
