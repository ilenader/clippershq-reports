**Conclusion: the cutover has NOT taken effect. The most likely cause: the key was disabled on one Railway service (e.g. the web app) but the tracking cron runs on a separate service/deploy (railway-cron-scheduler) whose variable was not changed, or the change was saved without a redeploy of that service. I cannot read Railway's env or deployed commit from here; the run-log is the authoritative live signal and it says Apify is alive.

Prefer an EMPTY value over `DISABLED`, because the code skips Apify only on a falsy key; `DISABLED` is truthy and merely forces a 401 per call (still fail-safe, but it needlessly calls Apify and logs 401s).
