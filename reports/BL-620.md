## PART 1 — batch submission now lives inside the modal
Every clipper taps "Submit a clip" and can submit ONE clip or MANY, in one place:
* **The single-clip default is byte-identical to before.** Step 2 still opens on the same account radiogroup + link field + "Submit clip" button, still posting through the existing `/api/clips` path with all its request-scoped hooks (idempotency, streak, first-clip celebration, analytics). Nothing about the one-clip flow got slower or more confusing.
* **One obvious control, in place.** A dashed "＋ Add another clip" button sits directly under the link field. Tapping it switches step 2 into the embedded batch experience WITHOUT leaving the modal, reloading, or navigating, and **seeds the first row from whatever the clipper already filled** (account + link), so nothing is re-typed. A "Back to a single clip" control returns to the simple view.
* **Each row is self-contained:** platform (constrained to the campaign's accepted platforms), the connected account it posts from (constrained to that platform), and the link, with a plain reminder "This link will attach to @handle on {Platform}." so mis-mapping is impossible to miss.
* **The old accordion is gone.** `BatchSubmitSection` was moved to `src/components/clips/` and given an `embedded` mode (it drops the disclosure heading + campaign picker, forces open, and accepts `seedRows`). The old `src/app/(app)/clips/BatchSubmitSection.tsx` and its render at the bottom of `/clips` were **deleted**, so there is exactly one place to submit one clip or many.

## PART 2 — the stale-session bug, audited
BL-618 already removed the `isTestUser` gate whose visibility read from the session JWT (the reason a clipper had to log out to see the feature). Re-audited the clips page and the submit modal on merged main: **no clipper-facing capability on this path hides behind a stale JWT claim.** `isTestUser` is no longer read anywhere in the submit flow (the const and the gate are gone); the batch endpoints gate on `canActAsClipperFromSession`, a per-request server session check, not a cached token. So no clipper needs to log out to see batch submission. (The one JWT-derived value still read on the page, `role`, is a coarse route guard that does not hide a feature and refreshes on the server session; noted as safe.)
