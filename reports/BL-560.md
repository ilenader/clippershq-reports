
**But it is NOT structurally the same as the BL-554 Instagram defect, and the difference decides how urgent it is.** Line 1109 sits **outside** the `if (isLamatokConfigured())` block (lines 1031 to ~1101). It is reached only when:

1. `isLamatokConfigured()` is **false** (LamaTok not configured), or

The `goneSet` is declared at `apify.ts:1039`, **inside** that block. On the line-1109 path it is never built. So line 1109 is not discarding an already-computed exclusion set (which is what BL-554 fixed on Instagram, where the Hiker fan-out computed `urlsForApify` and the fall-through wrongly re-derived from `urls`). At 1109 there is no exclusion set to discard, because no gone-detection ran.

**So can gone clips still reach clockworks through 1109? Plainly: yes, but only on batches where LamaTok did not run (unconfigured, or its overlay threw).** On those batches there is no gone classification at all, so it is a LamaTok-availability gap, not a discard-the-set bug. **In steady-state production, where LamaTok is up and answering (BL-557 and BL-559 both confirm it is the live primary), the batch returns at line 1094 using goneSet-filtered `missing`, and BL-559 makes that goneSet LARGER, so gone clips are kept out of clockworks more aggressively than before, not less.**
